## 🟢 Словарик №1: базовые слова Java (без них никуда)

Если ты не Java-разработчик, эти слова будут встречаться постоянно. Прочитай один раз — дальше будет легче.

- **Класс (class)** — чертёж, описание «из чего состоит вещь и что она умеет». Например, класс `User` описывает, что у пользователя есть имя и email.
- **Объект / экземпляр (object / instance)** — конкретная вещь, сделанная по чертежу. Класс `User` — чертёж, а `Алиса с почтой a@x.com` — объект.
- **Поле (field)** — переменная внутри класса, «свойство вещи» (имя, возраст, email).
- **Метод (method)** — действие, которое умеет объект. Это «функция, живущая внутри класса». Например, метод `pay()` — «оплатить».
- **Конструктор (constructor)** — особый метод, который вызывается в момент создания объекта, чтобы его «собрать» и заполнить поля.
- **`new`** — ключевое слово «создай новый объект»: `new User()` = «сделай нового пользователя по чертежу `User`».
- **Интерфейс (interface)** — список «что вещь должна уметь», но без того, *как именно*. Это как контракт или вакансия: «нужен тот, кто умеет `log()`». А уже конкретные классы (`ConsoleLogger`, `FileLogger`) этот контракт выполняют по-своему. Очень важная идея в Spring.
- **Реализация (implementation)** — конкретный класс, который выполняет контракт интерфейса.
- **Наследование (`extends`)** — один класс берёт за основу другой и добавляет своё.
- **Аннотация (`@Что-то`)** — «наклейка-пометка» на классе или методе, которая что-то говорит фреймворку. Например `@Service` — наклейка «это сервис, Spring, возьми его себе». Сами по себе аннотации ничего не делают — их *читает* фреймворк и ведёт себя по-особому.
- **Пакет (package)** — папка, в которой лежат классы (например `com.example.service`).
- **Исключение (exception)** — ошибка времени выполнения. Код «бросает» (`throw`) исключение, когда что-то пошло не так, а другой код может его «поймать» (`try/catch`).
  - **checked** — исключение, которое компилятор *заставляет* обработать (например, ошибка работы с файлом).
  - **unchecked (RuntimeException)** — необязательное к обработке (например, деление на ноль). Важно для темы транзакций.
- **Поток / нить (thread)** — «работник», выполняющий код. Программа может иметь много потоков, работающих одновременно. Запомни образ: поток = один официант. Это пригодится в темах про серверы, реактивность и Kafka.
- **Блокирующий код (blocking)** — код, который заставляет поток *стоять и ждать* (например, ждать ответа от базы данных). Поток в это время бесполезен.
- **Неблокирующий (non-blocking)** — код, который не заставляет поток ждать: «отправил запрос и пошёл заниматься другим, а когда ответ придёт — вернёмся».
- **Generics (`<T>`)** — «коробка с указанием, что внутри». `List<User>` = «список, в котором лежат `User`». `<T>` — заглушка под любой тип.
- **JSON** — текстовый формат для передачи данных по сети, выглядит как `{"name": "Alice", "age": 20}`. Браузеры и сервисы общаются друг с другом в основном им.
- **Сериализация / десериализация** — превращение объекта Java в JSON (сериализация) и обратно из JSON в объект (десериализация). Этим занимается библиотека **Jackson**.
- **POJO** (Plain Old Java Object) — «обычный простой Java-объект» без всякой магии и обязательных наследований. Spring любит работать с POJO.

## 🟢 Словарик №2: ключевые слова Spring

- **Spring** — большой набор инструментов (фреймворк) для написания серверных приложений на Java. Ядро — «контейнер», который сам создаёт и связывает объекты.
- **Spring Boot** — надстройка над Spring, которая берёт на себя всю скучную настройку, чтобы можно было запустить приложение почти без конфигурации. «Spring, но всё уже настроено за тебя».
- **Бин (bean)** — это просто **объект, которым управляет Spring**. Ты не создаёшь его сам через `new` — Spring создаёт его, хранит и выдаёт тому, кому он нужен. Слово «бин» будет встречаться сотни раз — это «объект под управлением Spring», и всё.
- **IoC-контейнер / ApplicationContext** — «коробка», в которой Spring хранит все бины. Можно представить как умный склад объектов: попроси нужный — он выдаст, да ещё и со всеми зависимостями.
- **IoC (Inversion of Control, инверсия управления)** — идея «не ты создаёшь объекты, а фреймворк создаёт их за тебя и сам отдаёт тебе».
- **DI (Dependency Injection, внедрение зависимостей)** — конкретный способ, которым Spring «подкладывает» объекту нужные ему другие объекты. Если сервису заказов нужен сервис писем — Spring сам его туда вставит, тебе не надо писать `new`.
- **Зависимость (dependency)** — другой объект, который нужен данному объекту для работы.
- **Прокси (proxy)** — «объект-обёртка вокруг настоящего объекта». Снаружи выглядит как настоящий, но перехватывает вызовы и добавляет поведение (логирование, транзакции). Очень важная идея — встретится в AOP, транзакциях, кешировании.
- **Аннотация-стереотип** — наклейки `@Component`, `@Service`, `@Repository`, `@Controller`, которые говорят Spring «возьми этот класс под управление как бин».
- **Эндпоинт (endpoint)** — конкретный «адрес» в твоём приложении, на который можно прислать запрос (например `GET /api/users/1`).
- **Entity (сущность)** — объект, который соответствует строке в таблице базы данных.
- **DTO** (Data Transfer Object) — объект «только для передачи данных» наружу (в ответе API). Намеренно отделён от Entity, чтобы не светить наружу внутренности базы.
- **Контроллер (controller)** — класс, который принимает HTTP-запросы и решает, что на них ответить.
- **Сервис (service)** — класс с бизнес-логикой («что приложение делает»).
- **Репозиторий (repository)** — класс, который ходит в базу данных (читает/пишет).

> 💡 Главная мысль про весь Spring: **ты пишешь классы и вешаешь на них наклейки-аннотации, а Spring сам создаёт из них объекты (бины), связывает их между собой и вызывает в нужный момент.** Всё остальное — детали.

## 1. Фреймворк или библиотека? Что такое Spring и Spring Boot

**Простыми словами.** Представь готовку на кухне.
- **Библиотека** — это набор приборов (блендер, миксер). Ты сам решаешь, когда какой взять и включить. Главный — ты, приборы просто ждут команды.
- **Фреймворк** — это шеф-повар, который руководит всем процессом и сам говорит тебе: «сейчас режь лук», «теперь соли». Главный — он, а ты лишь заполняешь места, которые он тебе оставил.

Вся разница в одном вопросе: **кто кого вызывает?** В случае библиотеки *твой* код вызывает чужой. В случае фреймворка *чужой* код (фреймворк) вызывает *твой* в нужные моменты. Этот «разворот» главного называют **инверсией управления (IoC, Inversion of Control)**, а в шутку — «принципом Голливуда»: «не звоните нам — мы сами вам позвоним».

**Разбираемся.**
- **Преимущества фреймворков:** быстрее разрабатывать (много готового — внедрение зависимостей, транзакции, безопасность), единый стиль кода в команде, большое сообщество, проверенные временем архитектурные решения.
- **Недостатки:** нужно время на изучение; много «магии» под капотом (трудно понять, что происходит); привязка к экосистеме (vendor lock-in — потом сложно «слезть»); для крошечного проекта это перебор.

**Spring Framework** — модульный фреймворк для Java. Его сердце — **IoC-контейнер** (умный склад объектов) и **DI** (внедрение зависимостей). Вокруг — модули: Spring MVC (веб), Spring Data (базы), Spring Security (безопасность) и др. Spring позволяет писать обычные простые классы (**POJO**) — без обязательных наследований, только с аннотациями-наклейками.

**Spring Boot** — надстройка над Spring, убирающая три главные боли:
1. **«Ад конфигурации»** — раньше настройку писали десятками строк XML. Теперь хватает одной наклейки `@SpringBootApplication` и метода `main()`.
2. **Автоконфигурация** — Boot смотрит, какие библиотеки подключены, и настраивает всё сам: увидел сервер Tomcat — поднял веб-сервер; увидел драйвер базы — настроил подключение.
3. **Стартеры** — одна зависимость `spring-boot-starter-web` тянет за собой весь согласованный набор для веба (сервер, MVC, Jackson для JSON, валидацию).

**Разбор кода.**
```java
@SpringBootApplication // = @Configuration + @EnableAutoConfiguration + @ComponentScan
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args); // поднимает контекст + встроенный сервер
    }
}
```
**Что здесь происходит:**
- `@SpringBootApplication` — главная наклейка Spring Boot. Она объединяет три наклейки сразу: `@Configuration` (класс с настройками), `@EnableAutoConfiguration` (включи автонастройку) и `@ComponentScan` (найди все мои классы-бины в этой папке и ниже).
- `main(String[] args)` — метод, с которого Java начинает выполнение программы.
- `SpringApplication.run(...)` — одна строчка, которая поднимает весь склад бинов (контекст) и запускает встроенный веб-сервер. После неё приложение уже работает.

**⚠️ Подводные камни.**
- Автоконфигурация удобна, но «прячет» происходящее. Если настроилось не так, как ждёшь, запусти с флагом `--debug` — получишь отчёт (Conditions Evaluation Report), что и почему Boot включил; или посмотри список бинов на `/actuator/beans`.
- Spring нужен не всегда: для крошечной утилиты он избыточен. Но порог входа у Boot настолько низкий, что его берут даже для маленьких проектов.

**📌 Запомнить.** Библиотека — *ты* зовёшь её; фреймворк — *он* зовёт тебя (это и есть IoC). Spring Boot = Spring + автонастройка + стартеры, запуск в одну строку.

## 2. IoC и Dependency Injection (внедрение зависимостей)

**Простыми словами.** Представь, что тебе на работе нужны инструменты. Можно каждый раз самому бегать в магазин и покупать (это `new` — создавать всё руками внутри класса). А можно, чтобы заботливый завхоз заранее всё закупил и просто выдавал тебе нужное под задачу. Spring — это и есть тот завхоз: он сам создаёт объекты и сам «выдаёт» их тем классам, которым они нужны. Тебе не нужно знать, откуда они взялись.

**Разбираемся.**
- **IoC (Inversion of Control, инверсия управления)** — это когда **контейнер** (тот самый «умный склад») сам создаёт объекты, сам управляет их жизнью и сам связывает их между собой. Объекты, которыми управляет Spring, называются **бинами (beans)**.
- **DI (Dependency Injection, внедрение зависимостей)** — это конкретный механизм, как именно работает IoC: зависимости (нужные классу объекты) **внедряются извне**, то есть класс не создаёт их сам через `new`, а получает готовыми.

**Зачем это нужно.** Главная цель — убрать **жёсткую связность (tight coupling, когда классы намертво привязаны друг к другу)**. Разберём на примере. Пусть класс `OrderService` (обработка заказов) сам внутри пишет `new EmailService()` (создаёт отправщик писем):
- Его **нельзя нормально протестировать** — каждый тест будет реально слать письма.
- Его **нельзя заменить** отправщик на `SmsService` (через SMS), не переписав код самого `OrderService`.

С DI зависимость задаётся снаружи и через **интерфейс (общий «контракт», описывающий, что объект умеет, без деталей как)**. Тогда в тестах легко подсунуть **мок (mock — поддельную заглушку вместо настоящего объекта)** и легко подменить реализацию.

**Реализации IoC-контейнера** (разные «версии склада»):
- `BeanFactory` — базовый, самый простой контейнер.
- `ApplicationContext` — расширенный, именно его обычно и используют.
- `AnnotationConfigApplicationContext` — контейнер, настраиваемый через Java-аннотации.
- `ClassPathXmlApplicationContext` — контейнер, настраиваемый через XML-файлы.

**Три типа внедрения** (три способа, как зависимость попадает в класс):

|Тип|Плюсы|Минусы|
|---|---|---|
|**Конструктор** (рекомендуется)|поля `final`, гарантированная инициализация, легко тестировать через `new`, циклические зависимости видны на старте|многословнее при многих зависимостях|
|**Поле** (`@Autowired` над полем)|минимум кода|нарушает инкапсуляцию (reflection), нельзя `final`, нужен Spring/reflection для тестов, прячет зависимости|
|**Сеттер**|для опциональных (`required=false`), можно менять в рантайме|объект мутабелен, может быть в невалидном состоянии между created и set|

Здесь `final` — это пометка «значение задаётся один раз и больше не меняется» (надёжно), а **reflection (рефлексия)** — приём, когда Spring «лезет» внутрь объекта в обход обычных правил доступа.

**Ключевые аннотации:**
- **`@Autowired`** — помечает точку, куда Spring должен внедрить зависимость. На единственном конструкторе её можно не писать — Spring догадается сам.
- **`@Qualifier("beanName")`** — когда подходящих бинов несколько, эта аннотация говорит, какой именно взять (по имени). Важно: **`@Qualifier` имеет приоритет над `@Primary`** (то есть точное указание сильнее, чем «бин по умолчанию»).

**Способы конфигурирования** (как объяснить Spring, какие бины существуют):
- **XML-based** — описание `<bean>` в XML-файле (устаревает).
- **Annotation-based** — наклейки `@Component`/`@Service`/… плюс `@ComponentScan` (Spring сам всё находит и создаёт).
- **Java-based** — `@Configuration` + `@Bean` (методы вручную возвращают бины). Нужно для классов из **сторонних библиотек**, где нельзя поставить `@Component` (чужой код не отредактируешь).

**Разбор кода.**
```java
public interface Logger { void log(String m); }

@Component @Primary
class ConsoleLogger implements Logger { public void log(String m){ System.out.println(m); } }

@Component("fileLogger")
class FileLogger implements Logger { public void log(String m){ /* в файл */ } }

@Service
public class AuditService {
    private final Logger logger;
    // @Qualifier перебивает @Primary → сюда придёт FileLogger, а не ConsoleLogger
    public AuditService(@Qualifier("fileLogger") Logger logger) { this.logger = logger; }
}

// Java-based для стороннего класса без аннотаций
@Configuration
class ApiConfig {
    @Bean
    ExternalApiClient apiClient(@Value("${app.api.url}") String url) {
        return new ExternalApiClient(url);
    }
}
```
**Что здесь происходит:**
- `interface Logger` — это «контракт»: любой логгер должен уметь метод `log`. Дальше есть две реализации.
- `@Component @Primary` над `ConsoleLogger` — это бин, и он помечен как «выбор по умолчанию», если кандидатов несколько.
- `@Component("fileLogger")` — второй бин, ему явно задано имя `fileLogger`.
- `@Service` над `AuditService` — это тоже бин (с оттенком «бизнес-логика»).
- В конструкторе `@Qualifier("fileLogger")` говорит: несмотря на то что у `ConsoleLogger` стоит `@Primary`, сюда внедри именно `FileLogger` (точное указание сильнее).
- `@Configuration` + `@Bean` в `ApiConfig` — это «ручная сборка» бина `ExternalApiClient` для класса, который мы не можем пометить `@Component` (он из чужой библиотеки). `@Value("${app.api.url}")` подставляет в аргумент значение из настроек.

**⚠️ Подводные камни.**
- 🎯 **Циклические зависимости.** Если `A` нужен `B`, а `B` нужен `A` (`A→B→A`) — это замкнутый круг. При **конструкторном** внедрении Spring обнаружит этот цикл прямо на старте и упадёт с ошибкой `BeanCurrentlyInCreationException`. И это плюс: проблему видно сразу, а не в случайный момент. Правильное лечение — **рефакторинг** (вынести общую часть в третий, отдельный сервис), а не затыкать дыру костылём `@Lazy` (ленивая инициализация).

**📌 Запомнить.** IoC — контейнер сам создаёт и связывает объекты; DI — он внедряет зависимости извне, а не класс делает `new`. Лучший способ внедрения — через конструктор. `@Qualifier` сильнее `@Primary`.

## 3. Виды архитектур: слоистая, модульная, чистая

**Простыми словами.** Архитектура — это то, как ты раскладываешь код по полкам, чтобы в нём не утонуть. Можно делить по «этажам» (на одном — общение с пользователем, на другом — логика, на третьем — база) — это слоистая. Можно делить по «комнатам-фичам» (заказы отдельно, платежи отдельно) — это модульная. А можно строить так, чтобы самое ценное — суть бизнеса — лежало в центре и ни от чего не зависело — это чистая.

**Разбираемся.**

**Слоистая (Layered)** — деление по горизонтали, то есть по **технической ответственности**:
`Controller` (представление, общение с внешним миром) → `Service` (бизнес-логика) → `Repository` (доступ к данным) → БД.
Spring прямо отражает эту архитектуру своими стереотипами `@Controller`/`@Service`/`@Repository`.
- **Плюсы:** простая, понятная, фактически стандарт для enterprise (крупных корпоративных систем).
- **Минусы:** при росте проекта слои «толстеют», бизнес-логика тянет за собой инфраструктуру, а изменения «протекают» сквозь все слои сразу.

**Модульная (Modular)** — деление по вертикали, то есть по **доменам/фичам**. Пакеты `order/`, `payment/`, `user/`, и в каждом — свои controller/service/repository/model, плюс общий `shared/`.
- **Плюсы:** модули самодостаточны, удобно для разных команд и для будущего распила на **микросервисы** (отдельные мелкие приложения), границы явные.
- **Минусы:** дублирование общих частей, нужно дисциплинированно следить за связями между модулями.

**Чистая (Clean) / гексагональная** — здесь все зависимости направлены **внутрь, к домену** (это называют **Dependency Rule, правило зависимостей**). Внутреннее ядро (**Entities** — сущности, объекты бизнеса, и **Use Cases** — сценарии использования) не знает ничего ни о Spring, ни о базе, ни о вебе. Внешние слои (контроллеры, БД, внешние API) подключаются через **порты (интерфейсы, которые объявляет сам домен — он диктует, что ему нужно)** и **адаптеры (реализации этих портов, живущие в инфраструктуре)**.
- **Плюсы:** максимальная тестируемость и независимость от фреймворка, легко поменять базу или способ передачи данных.
- **Минусы:** больше абстракций и кода, для простого CRUD (создать-прочитать-обновить-удалить) это избыточно.

**Разбор кода.**
```text
Clean Architecture — правило зависимостей (всё указывает ВНУТРЬ):
[ Controllers / Adapters (Spring, JPA, REST) ]
            ↓ зависят от
[ Use Cases (сценарии приложения) ]
            ↓ зависят от
[ Domain Entities (чистый Java, без Spring) ]   ← ядро ни от кого не зависит
```
**Что здесь происходит:** это схема направления зависимостей. Самый внешний слой (контроллеры, JPA, REST — то есть всё «техническое») зависит от слоя сценариев, а тот — от ядра с сущностями. Ядро же не зависит ни от кого — это и есть смысл «чистой» архитектуры: техника крутится вокруг бизнеса, а не наоборот.

```java
// Порт объявлен в домене — домен диктует, что ему нужно
public interface UserPort {
    Optional<User> findById(Long id);
}
// Адаптер живёт в инфраструктуре и реализует порт
@Repository
class JpaUserAdapter implements UserPort {
    private final JpaUserRepo jpa;
    JpaUserAdapter(JpaUserRepo jpa){ this.jpa = jpa; }
    public Optional<User> findById(Long id){ return jpa.findById(id).map(this::toDomain); }
}
// Use case зависит только от порта (интерфейса), не от JPA
class GetUserUseCase {
    private final UserPort port;
    GetUserUseCase(UserPort port){ this.port = port; }
}
```
**Что здесь происходит:**
- `interface UserPort` — это **порт**: домен говорит «мне нужно уметь найти пользователя по id», но не уточняет как. `Optional<User>` — обёртка «может быть пользователь, а может и нет».
- `JpaUserAdapter` с `@Repository` — это **адаптер**: он реально умеет лезть в базу через JPA и выполняет обещание порта. Он живёт в инфраструктуре, отдельно от ядра.
- `GetUserUseCase` — сценарий: он держит у себя только `UserPort` (интерфейс), и ему совершенно всё равно, JPA там внутри или что-то другое. Именно поэтому ядро не зависит от техники.

**📌 Запомнить.** Слоистая — по техническим этажам (стандарт Spring). Модульная — по фичам (удобно дробить на микросервисы). Чистая — зависимости смотрят внутрь, ядро ни от чего не зависит (порты + адаптеры).

## 4. AOP: Join Point, Pointcut, Advice, Aspect

**Простыми словами.** Представь, что в каждом кабинете офиса нужно при входе и выходе отмечаться в журнале. Можно заставить каждого сотрудника писать это вручную в своём коде (дублирование и бардак). А можно поставить на двери автоматический турникет, который сам отмечает всех — и сотрудники про него даже не думают. **AOP (Aspect-Oriented Programming, аспектно-ориентированное программирование)** — это и есть такой «турникет»: оно выносит повторяющиеся задачи (логирование, безопасность) в одно место и само «навешивает» их на нужные методы.

**Разбираемся.** AOP выносит **сквозную функциональность (cross-cutting concerns)** — логирование, транзакции, безопасность, метрики, аудит — в отдельные модули. Это то, что нужно «везде понемногу» и что иначе пришлось бы копировать в каждый метод, засоряя бизнес-логику. Четыре главных понятия:

- **Join Point (точка соединения)** — место в выполнении программы, куда можно вклиниться. **В Spring AOP это всегда выполнение метода** (method execution). Это динамическая штука — существует только в момент работы программы (в рантайме). Объект `JoinPoint` в коде даёт доступ к сигнатуре метода, его аргументам (`getArgs()`) и к целевому объекту (`getTarget()`).
- **Pointcut (срез)** — это выражение-фильтр (на языке AspectJ), которое отбирает нужные Join Points. Может фильтровать по сигнатуре, пакету, типу возвращаемого значения, модификатору, аннотации. Фильтры можно комбинировать (`&&` — и, `||` — или, `!` — не) и давать им имена через `@Pointcut`.
- **Advice (совет)** — собственно код, который выполняется в точке Join Point. Виды:
    - `@Before` — **до** метода;
    - `@AfterReturning` — после **успешного** возврата (есть доступ к результату);
    - `@AfterThrowing` — при **исключении** (есть доступ к самому исключению);
    - `@After` — **всегда** (как блок `finally`);
    - `@Around` — **оборачивает** метод: через `ProceedingJoinPoint.proceed()` он сам решает, вызывать ли оригинальный метод вообще, может менять аргументы и результат, ловить исключения.
- **Aspect (аспект)** — класс с аннотацией `@Aspect`, который объединяет в себе Pointcut'ы и Advice (то есть «что отбираем» + «что делаем»).

**Разбор кода.**
```java
@Aspect
@Component
public class LoggingAspect {

    // именованный pointcut: все методы любого класса в пакете service
    @Pointcut("execution(* com.example.service.*.*(..))")
    public void serviceMethods() {}

    // по аннотации метода
    @Pointcut("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void transactional() {}

    @Before("serviceMethods()")
    public void before(JoinPoint jp) {
        System.out.println("Вызов " + jp.getSignature().getName()
            + " args=" + Arrays.toString(jp.getArgs()));
    }

    @AfterThrowing(pointcut = "serviceMethods()", throwing = "ex")
    public void onError(JoinPoint jp, Exception ex) {
        System.out.println("Ошибка в " + jp.getSignature().getName() + ": " + ex.getMessage());
    }

    @Around("serviceMethods()")
    public Object measure(ProceedingJoinPoint pjp) throws Throwable {
        long t = System.currentTimeMillis();
        try {
            return pjp.proceed();        // вызов оригинального метода
        } finally {
            System.out.println(pjp.getSignature().getName()
                + " занял " + (System.currentTimeMillis() - t) + "ms");
        }
    }
}
```
**Что здесь происходит:**
- `@Aspect` + `@Component` — класс является аспектом и одновременно бином (чтобы Spring его подхватил).
- Первый `@Pointcut` с выражением `execution(* com.example.service.*.*(..))` — это фильтр «все методы любого класса в пакете `service`». Звёздочки означают «любой тип возврата / любой класс / любой метод», а `(..)` — «любые аргументы». Метод `serviceMethods()` пустой — он нужен только как имя для этого фильтра.
- Второй `@Pointcut` отбирает методы по наличию аннотации `@Transactional`.
- `@Before("serviceMethods()")` — этот advice сработает **до** каждого метода-сервиса и напечатает его имя и аргументы.
- `@AfterThrowing(..., throwing = "ex")` — сработает, только если метод **выбросил исключение**; параметр `throwing = "ex"` связывает пойманное исключение с аргументом `ex`.
- `@Around` — оборачивает метод: засекает время, вызывает оригинал через `pjp.proceed()`, а в `finally` печатает, сколько метод выполнялся. Это самый мощный вид advice, потому что он управляет самим вызовом.

**📌 Запомнить.** AOP убирает дублирование «общих» задач. Join Point — где вклиниться (в Spring — выполнение метода), Pointcut — фильтр «куда именно», Advice — что сделать, Aspect — класс, который их связывает. `@Around` — самый мощный advice.

## 5. Spring AOP vs AspectJ. CGLIB/JDK-прокси, Weaving

**Простыми словами.** Чтобы «навесить» дополнительное поведение на твой объект, Spring подставляет вместо него двойника-посредника — **прокси (proxy)**. Все обращаются к двойнику, а он по пути добавляет нужное (например, открывает транзакцию) и передаёт вызов настоящему объекту. AspectJ работает иначе и мощнее: он не подставляет двойника, а вшивает дополнительный код прямо «в кости» твоих классов.

**Разбираемся.**

**Spring AOP — proxy-based (на основе прокси).** Вокруг бина контейнер создаёт **прокси-объект**, который перехватывает вызовы и добавляет advice. Работает **только с бинами Spring** и **только на уровне выполнения методов**.

Есть два механизма создания прокси:
- **JDK Dynamic Proxy** — используется, если бин реализует интерфейс; тогда прокси реализует тот же интерфейс (через `java.lang.reflect.Proxy`).
- **CGLIB** — используется, если интерфейса нет; тогда в рантайме генерируется **подкласс** (наследник твоего класса). Из-за этого ни сам класс, ни проксируемые методы **не должны быть `final`** (от `final` нельзя наследоваться), и нужен доступный конструктор. В Spring Boot CGLIB используется по умолчанию даже когда интерфейс есть.

**AspectJ** — это полноценное AOP-расширение Java с гораздо более богатым набором Join Points: оно умеет вклиниваться в доступ к полям, обработку исключений, инициализацию и даже **конструкторы и внутренние вызовы**. Вместо прокси оно использует **Weaving (вплетение)** — внедрение аспектного кода прямо в **байткод** (скомпилированный код Java). Виды weaving:
- **compile-time weaving** — при компиляции (специальным компилятором `ajc`);
- **post-compile (binary) weaving** — в уже готовые `.class`-файлы / jar-архивы;
- **load-time weaving (LTW)** — в момент загрузки класса загрузчиком (ClassLoader'ом), через специальный javaagent.

Spring умеет работать с AspectJ-аннотациями, но для полной мощи (перехват конструкторов и полей) нужен именно AspectJ-weaving, а не Spring-прокси.

**Разбор кода.**
```java
@Configuration
@EnableAspectJAutoProxy // включает proxy-based AOP Spring
class AppConfig {}

// AspectJ LTW: аспект НЕ помечается @Component (им управляет AspectJ-агент),
// и он умеет то, чего не может Spring AOP — перехват КОНСТРУКТОРА
@Aspect
public class ConstructorAuditAspect {
    @Pointcut("execution(com.example.AuditRecord.new(..))") // .new(..) = конструктор
    public void ctor() {}

    @Around("ctor()")
    public Object onCreate(ProceedingJoinPoint pjp) throws Throwable {
        System.out.println("[LTW] создаём AuditRecord " + Arrays.toString(pjp.getArgs()));
        return pjp.proceed();
    }
}
```
**Что здесь происходит:**
- `@EnableAspectJAutoProxy` — включает в Spring обычное прокси-based AOP (то самое, что работает только с методами бинов).
- Второй аспект `ConstructorAuditAspect` — пример возможностей AspectJ. Обрати внимание: он **не помечен** `@Component`, потому что им управляет не Spring, а AspectJ-агент (LTW).
- Выражение `execution(com.example.AuditRecord.new(..))` с записью `.new(..)` отбирает **конструктор** класса — то, чего обычное Spring AOP вообще не умеет.
- `@Around` оборачивает создание объекта: печатает аргументы перед созданием и вызывает оригинальный конструктор через `pjp.proceed()`.

**⚠️ Подводные камни.**
- 🎯 **Главное ограничение прокси.** Прокси перехватывает только вызовы, которые идут **снаружи, через него**. **Внутренний вызов `this.method()` (метод вызывает соседний метод того же класса) и вызовы конструкторов НЕ перехватываются** — они идут мимо двойника напрямую. Это классическая причина, почему `@Transactional` или `@Cacheable` «не срабатывает», когда метод вызван из другого метода того же класса: вызов не прошёл через прокси, а значит, обёртки не было.

**📌 Запомнить.** Spring AOP = прокси-двойник, только методы бинов, только вызовы снаружи. JDK-прокси — когда есть интерфейс; CGLIB (наследник, не любит `final`) — когда нет. AspectJ = вплетение в байткод, умеет конструкторы/поля/внутренние вызовы. Внутренний `this.method()` мимо прокси не перехватывается.

## 6. Bean (бин). Жизненный цикл

**Простыми словами.** **Бин (bean)** — это объект, которым управляет Spring. И как у живого существа, у него есть жизненный цикл: он «рождается» (создаётся), его «снаряжают» (внедряют зависимости), он «настраивается» к работе, живёт-трудится, а потом «умирает» (уничтожается при закрытии приложения). Spring проводит бин через все эти этапы строго по порядку, и в нужные моменты даёт тебе вклиниться (например, открыть ресурсы на старте и закрыть их перед смертью).

**Разбираемся.** Контейнер полностью управляет жизненным циклом синглтон-бина (того, что существует в одном экземпляре). Этапы по порядку:

1. **Инстанцирование** — вызов конструктора (объект создан).
2. **Внедрение зависимостей** (populate properties) — заполняются поля/сеттеры/конструктор.
3. **`*Aware`-интерфейсы** — `BeanNameAware.setBeanName()`, `BeanFactoryAware`, `ApplicationContextAware`. Через них бин получает «знание» об инфраструктуре (как его зовут, в каком контейнере он живёт).
4. **`BeanPostProcessor.postProcessBeforeInitialization()`** — хук (точка вмешательства) **до** инициализации.
5. **Инициализация (строго в этом порядке):** `@PostConstruct` → `InitializingBean.afterPropertiesSet()` → `init-method` (заданный в `@Bean(initMethod=...)`).
6. **`BeanPostProcessor.postProcessAfterInitialization()`** — **здесь создаются AOP-прокси** (бин оборачивается двойником).
7. **Бин готов** — лежит в контейнере и используется.
8. **Уничтожение (при закрытии контекста, только для singleton):** `@PreDestroy` → `DisposableBean.destroy()` → `destroy-method`.

> Важное исключение: **`prototype`-бины** (новый экземпляр на каждый запрос) контейнер создаёт и отдаёт, но **не уничтожает** — `@PreDestroy` у них не вызывается. За их закрытие отвечает тот, кто их попросил.

**Разбор кода.**
```java
@Component
public class ConnectionPoolBean implements InitializingBean, DisposableBean {

    public ConnectionPoolBean() { System.out.println("1. конструктор"); }

    @PostConstruct
    public void postConstruct() { System.out.println("2. @PostConstruct — открываем ресурсы"); }

    @Override public void afterPropertiesSet() { System.out.println("3. afterPropertiesSet"); }

    @PreDestroy
    public void preDestroy() { System.out.println("4. @PreDestroy — закрываем ресурсы"); }

    @Override public void destroy() { System.out.println("5. destroy"); }
}
```
**Что здесь происходит:**
- `@Component` — класс является бином, значит, Spring проведёт его по всему жизненному циклу.
- `implements InitializingBean, DisposableBean` — бин подписывается на два «спринговых» интерфейса: первый даёт метод `afterPropertiesSet()` (после внедрения), второй — `destroy()` (перед уничтожением).
- Конструктор — этап 1 (рождение).
- `@PostConstruct` — метод инициализации, удобное место **открыть ресурсы** (например, пул соединений). Вызывается раньше `afterPropertiesSet()`.
- `@PreDestroy` — метод перед смертью, удобное место **закрыть ресурсы**.
- Числа 1–5 в выводе показывают реальный порядок вызова этих методов.

**⚠️ Подводные камни.**
- 🎯 **Почему AOP-прокси создаётся на шаге 6, а не в конструкторе?** Потому что прокси оборачивает уже **полностью инициализированный** бин (это делает `BeanPostProcessor` после инициализации). Практическое следствие: **внутри конструктора нельзя рассчитывать на проксированное поведение** — там объект ещё «голый», обёртки (транзакций, кэша) ещё нет.

**📌 Запомнить.** Этапы: конструктор → внедрение → Aware → before-init → инициализация (`@PostConstruct` → `afterPropertiesSet` → init-method) → after-init (тут рождаются прокси) → работа → уничтожение (`@PreDestroy` → `destroy`). Прототипы не уничтожаются контейнером. Прокси появляется после инициализации, не в конструкторе.

## 7. Стереотипные аннотации, скоупы, ObjectFactory/ObjectProvider/Provider

**Простыми словами.** **Стереотипные аннотации** — это наклейки-ярлычки на классах, которые говорят Spring «это бин» и заодно подсказывают его роль (логика, база, веб). А **скоуп (scope, область видимости)** — это правило, сколько экземпляров бина делать: один на всех или новый каждому. Иногда «один на всех» (синглтон) хочет получать «каждому новый» (прототип) — и тут возникает хитрая проблема, для которой есть специальные «выдавальщики» (провайдеры).

**Разбираемся.**

**Стереотипные аннотации** — все они частные случаи `@Component`, но различаются по смыслу и иногда по дополнительному поведению:
- `@Component` — базовая наклейка «это бин».
- `@Service` — бизнес-логика (отличие чисто смысловое, для читаемости).
- `@Repository` — доступ к данным; **дополнительно переводит `SQLException` и специфичные исключения базы в `DataAccessException`** — это unchecked-исключение (которое не обязательно ловить) из единой иерархии Spring. То есть прячет различия конкретных баз за общим типом ошибок.
- `@Controller` / `@RestController` — web-слой (приём запросов).

Чтобы бин вообще нашёлся, класс должен: лежать в пакете `@SpringBootApplication` или **ниже** (или быть указан в `@ComponentScan`), не быть абстрактным и иметь стереотип.

**Скоупы (области видимости):**
- `singleton` (по умолчанию) — **один** экземпляр на весь контейнер.
- `prototype` — **новый** экземпляр при каждом запросе бина.
- веб-скоупы: `request` (на один HTTP-запрос), `session` (на сессию пользователя), `application`, `websocket`.

**Разбор кода.**
```java
@Component @Scope("prototype")
class ReportTask { /* состояние под одну задачу */ }

@Service
public class ReportService {
    private final ObjectProvider<ReportTask> provider; // prototype

    public ReportService(ObjectProvider<ReportTask> provider) { this.provider = provider; }

    public void run() {
        ReportTask task = provider.getObject(); // КАЖДЫЙ раз новый экземпляр
        // task.execute();
    }
}
```
**Что здесь происходит:**
- `@Component @Scope("prototype")` — `ReportTask` объявлен прототипом: каждый запрос такого бина должен давать новый объект (потому что у него «состояние под одну задачу»).
- `ReportService` — это синглтон (`@Service`), и он создаётся один раз. Если бы он внедрил `ReportTask` напрямую, то получил бы один-единственный экземпляр навсегда (внедрение происходит лишь однажды). Поэтому вместо самого `ReportTask` он внедряет **провайдер** `ObjectProvider<ReportTask>`.
- В методе `run()` строка `provider.getObject()` каждый раз обращается к контейнеру и получает **свежий** прототип-экземпляр. Это и есть решение проблемы.

**⚠️ Подводные камни.**
- 🎯 **Проблема singleton → prototype.** Если синглтон-бин внедряет prototype-бин обычным способом, он получит **один и тот же** экземпляр — потому что внедрение случается всего один раз, в момент создания синглтона. Чтобы получать свежий prototype-бин на **каждый** вызов, нужен «провайдер» — объект, который по требованию идёт в контейнер за новым экземпляром:

|Способ|Откуда|Метод|Особенности|
|---|---|---|---|
|`ObjectFactory<T>`|Spring|`getObject()`|минимальный|
|`ObjectProvider<T>`|Spring|`getObject()`, `getIfAvailable()`, `getIfUnique()`, `stream()`|самый гибкий, безопасен при 0/много кандидатов|
|`Provider<T>`|JSR-330 (`jakarta.inject`)|`get()`|стандартный, переносимый|

Здесь `ObjectProvider` удобнее всего: его методы `getIfAvailable()` / `getIfUnique()` не падают, если бина нет вовсе или, наоборот, кандидатов несколько. `Provider<T>` — стандарт (JSR-330), поэтому переносим между фреймворками.

**📌 Запомнить.** Стереотипы — частные случаи `@Component`; `@Repository` ещё и переводит ошибки БД в `DataAccessException`. Скоуп по умолчанию — singleton (один на всех), prototype — новый каждый раз. Синглтон, которому нужен свежий прототип на каждый вызов, должен внедрять провайдер (`ObjectProvider`/`ObjectFactory`/`Provider`), а не сам бин.

## 8. Аннотации конфигураций (@Configuration, @ComponentScan, @Import, @Profile, @Primary, @DependsOn, @Order)

**Простыми словами.** Это набор наклеек, которыми ты управляешь сборкой приложения: где искать бины, какие конфиги подключить, какой бин включать только в нужной среде (например, заглушку на разработке и настоящий сервис на проде), какой считать «выбором по умолчанию» и в каком порядке всё создавать. Считай их пультом управления конвейером бинов.

**Разбираемся.**
- **`@Configuration`** — класс с `@Bean`-методами (методами, вручную создающими бины). Spring оборачивает такой класс CGLIB-прокси, поэтому повторный вызов `@Bean`-метода внутри конфига вернёт **тот же** singleton-бин, а не создаст новый объект. Если поставить `@Configuration(proxyBeanMethods=false)` (так называемый «lite»-режим), этой гарантии нет, зато работает быстрее.
- **`@ComponentScan(basePackages=...)`** — указывает, где искать классы с `@Component`. По умолчанию — пакет `@SpringBootApplication` и всё, что ниже.
- **`@Import(OtherConfig.class)`** — подключить другой конфиг-класс или компонент.
- **`@Profile("dev")`** — бин или конфиг включается, только если активен указанный **профиль** (среда запуска; задаётся через `spring.profiles.active`). Так разделяют настройки dev/test/prod.
- **`@Primary`** — пометить кандидата «по умолчанию», когда подходящих бинов несколько (перебивается более точным `@Qualifier`).
- **`@DependsOn("b")`** — гарантировать порядок создания: бин `b` будет создан **раньше** текущего.
- **`@Order(n)`** — задаёт порядок в коллекциях бинов, в цепочках фильтров, в аспектах. Правило: **меньше число = выше приоритет**.

**Разбор кода.**
```java
@Configuration
@Import(SecurityConfig.class)
public class AppConfig {

    @Bean @Primary @Profile("prod")
    PaymentGateway realGateway() { return new StripeGateway(); }

    @Bean @Profile({"dev", "test"})
    PaymentGateway mockGateway() { return new MockGateway(); }

    @Bean @DependsOn("flywayMigrator") // создать только после миграций
    DataInitializer dataInit() { return new DataInitializer(); }
}
```
**Что здесь происходит:**
- `@Configuration` — это класс-конфиг со «сборочными» `@Bean`-методами.
- `@Import(SecurityConfig.class)` — подтягивает в сборку ещё один конфиг (настройки безопасности).
- `realGateway()` — настоящий платёжный шлюз; помечен `@Primary` (выбор по умолчанию) и `@Profile("prod")` (включается только на проде).
- `mockGateway()` — заглушка; включается только в профилях `dev` и `test`. То есть в разработке используется фейк, в проде — реальный шлюз, и переключение — без правки кода.
- `dataInit()` с `@DependsOn("flywayMigrator")` — инициализатор данных будет создан **только после** бина `flywayMigrator` (то есть после применения миграций базы), иначе он работал бы по ещё не готовой схеме.

**📌 Запомнить.** `@Configuration` гарантирует «тот же бин» при повторном вызове `@Bean`-метода (CGLIB-прокси), кроме lite-режима. `@Profile` включает бины по среде, `@Primary` — выбор по умолчанию, `@DependsOn` — порядок создания, `@Order` — порядок в коллекциях (меньше = приоритетнее).

## 9. Properties: @Value, @PropertySource, @ConfigurationProperties

**Простыми словами.** Настройки приложения (адреса, пароли, лимиты) не вшивают в код — их выносят в отдельные текстовые файлы. Тогда одно и то же приложение можно по-разному настроить для разработки и для боевого сервера, ничего не перекомпилируя. А чтобы дотянуться из кода до этих настроек, есть три инструмента: один тянет одно значение, другой подключает доп. файл, третий аккуратно раскладывает целую группу настроек по полям класса.

**Разбираемся.** Конфигурацию выносят во внешние файлы:
- `application.properties` — плоский формат `ключ=значение`.
- `application.yaml` — иерархический (с вложенностью), удобнее, когда настроек много и они сгруппированы.
- Профильные файлы — `application-{profile}.yml` (свои настройки под каждую среду).

**Приоритет источников** (если значение задано в нескольких местах, побеждает то, что выше): аргументы командной строки → переменные окружения → профильные файлы → базовый файл. Boot склеивает все источники в одну картину.

Три инструмента доступа:
- **`@Value("${app.name}")`** — внедрить **одно** свойство. Значение по умолчанию задаётся через двоеточие: `${app.name:default}`. Поддерживает **SpEL (Spring Expression Language, язык выражений Spring)** через `#{...}`.
- **`@PropertySource("classpath:custom.properties")`** — подключить дополнительный `.properties`-файл. Важно: для YAML штатно **не работает** (только для `.properties`).
- **`@ConfigurationProperties(prefix="app")`** — **type-safe** (типобезопасный) маппинг целой группы свойств на поля класса. Рекомендуется для наборов настроек, потому что даёт: автодополнение, типизацию (`Duration` — длительность, `DataSize` — размер данных), валидацию (`@Validated` + Bean Validation), а также **релаксированный биндинг** (`max-attempts` спокойно ложится на поле `maxAttempts` — разный стиль написания допустим).

**Разбор кода.**
```yaml
app:
  name: MyService
  retry:
    max-attempts: 3
    delay: 500ms      # Boot сам сконвертит в Duration
```
**Что здесь происходит:** это YAML-файл с настройками. Отступы создают вложенность: `app.name`, `app.retry.max-attempts`, `app.retry.delay`. Значение `500ms` записано по-человечески, а Boot сам превратит его в тип `Duration` (длительность).

```java
@Component
@ConfigurationProperties(prefix = "app.retry")
@Validated
public class RetryProps {
    @Min(1) private int maxAttempts;   // <- app.retry.max-attempts
    private Duration delay;            // <- app.retry.delay (500ms)
    // обязательны геттеры/сеттеры (или record + @ConstructorBinding)
}

@Service
class MyService {
    @Value("${app.name}") private String name;          // одиночное
    private final RetryProps retry;
    MyService(RetryProps retry) { this.retry = retry; }  // type-safe группа
}
```
**Что здесь происходит:**
- `@ConfigurationProperties(prefix = "app.retry")` — говорит Spring: «возьми все настройки из ветки `app.retry` и разложи их по полям этого класса». Поле `maxAttempts` получит `app.retry.max-attempts` (заметь — разный стиль написания, это и есть релаксированный биндинг), а `delay` — `app.retry.delay`.
- `@Validated` + `@Min(1)` — включают проверку: если `maxAttempts` окажется меньше 1, приложение не запустится с такой настройкой. Это защита от неверной конфигурации.
- Для `@ConfigurationProperties` обязательны геттеры/сеттеры (или `record` с `@ConstructorBinding`).
- В `MyService` показаны оба подхода рядом: `@Value("${app.name}")` тянет **одно** свойство в поле `name`, а целую группу настроек повторов класс получает через внедрённый `RetryProps` (типобезопасно).

**⚠️ Подводные камни.**
- 🎯 **Когда `@Value`, а когда `@ConfigurationProperties`?** `@Value` — для 1–2 разрозненных свойств (быстро и просто). `@ConfigurationProperties` — для связного набора настроек, когда нужны валидация, типизация и удобство тестирования. Правило простое: одно-два значения — `@Value`; группа — `@ConfigurationProperties`.

**📌 Запомнить.** Настройки выносят в `application.properties`/`yaml` и профильные файлы; приоритет: командная строка → переменные окружения → профиль → базовый файл. `@Value` — одно свойство (с SpEL и дефолтом), `@PropertySource` — доп. `.properties`-файл (не YAML), `@ConfigurationProperties` — типобезопасная группа с валидацией.

## 10. Servlet, DispatcherServlet и полный путь HTTP-запроса

**Простыми словами.** Представь большой офис с одним ресепшеном. Любой посетитель (HTTP-запрос) сначала попадает к одному администратору на входе, и уже он решает, в какой кабинет тебя направить. В Spring роль такого администратора играет **DispatcherServlet** — единое окно, через которое проходят все запросы и которое раздаёт работу остальным сотрудникам.

**Разбираемся.**
- **Servlet (сервлет)** — это Java-компонент, который умеет обрабатывать HTTP-запросы внутри **контейнера сервлетов** (например, Tomcat). Контейнер принимает соединение по сети, а сервлет — это код, который отвечает на запрос.
- **DispatcherServlet** — особый сервлет от Spring. Он реализует паттерн **Front Controller (фронт-контроллер)** — это когда есть одна-единственная точка входа для всех запросов, а она уже делегирует (передаёт) обработку другим компонентам. То есть сам DispatcherServlet ничего не считает по бизнес-логике — он только дирижирует.

**Полный путь запроса** (внутри Spring это происходит в методе `doDispatch`):

1. Запрос приходит в контейнер сервлетов и передаётся в **DispatcherServlet**.
2. **HandlerMapping** — компонент, который по URL и HTTP-методу находит нужный обработчик (handler): конкретный контроллер и его метод. Заодно он собирает цепочку **интерсепторов** — `HandlerExecutionChain`. **Интерсептор (interceptor)** — это перехватчик, который может вклиниться до и после контроллера.
3. **HandlerInterceptor.preHandle()** — выполняется ДО контроллера. Тут удобно делать логирование или проверку прав (auth).
4. **HandlerAdapter** — компонент, который умеет правильно вызвать метод контроллера. Но перед вызовом срабатывают **ArgumentResolver**'ы (резолверы аргументов) — они «достают» параметры метода из запроса:
   - `@PathVariable` — из самого пути URL;
   - `@RequestParam` — из query-параметров (то, что после `?`);
   - `@RequestBody` — из тела запроса, при этом **HttpMessageConverter** (конвертер сообщений) десериализует JSON в Java-объект;
   - `@RequestHeader` — из заголовков.
5. Контроллер выполняет логику и возвращает результат: обычный объект, `ResponseEntity` или имя view (представления).
6. Дальше развилка:
   - **Если стоит `@ResponseBody` или `@RestController`** — `HttpMessageConverter` (обычно библиотека **Jackson**) сериализует объект в JSON прямо в тело ответа.
   - **Иначе** — **ViewResolver** находит представление (HTML-шаблон) и рендерит `ModelAndView`.
7. **HandlerInterceptor.postHandle()** и `afterCompletion()` — перехватчики уже ПОСЛЕ контроллера.
8. Ответ уходит клиенту. Если по дороге выскочило исключение, его подхватывает **HandlerExceptionResolver** (в том числе `@ControllerAdvice` — про него тема 15).

```text
HTTP Request → DispatcherServlet → HandlerMapping → HandlerAdapter → Controller
                                                                         │
   HTTP Response ← (HttpMessageConverter | ViewResolver) ← ModelAndView ─┘
```

**Что здесь происходит:** это та самая схема пути запроса. Запрос заходит слева в DispatcherServlet, проходит через поиск обработчика (HandlerMapping), адаптер (HandlerAdapter) и попадает в контроллер. На обратном пути результат превращается либо в JSON через конвертер, либо в HTML через ViewResolver — и уходит клиентом ответом.

**Разбор кода.**
```java
@RestController
@RequestMapping("/api/users")
public class UserController {
    @GetMapping("/{id}") // HandlerMapping свяжет GET /api/users/1 с этим методом
    public User get(@PathVariable Long id) {
        return new User(id, "Alice"); // конвертер сериализует в JSON (200 OK)
    }
}
```
**Что здесь происходит:**
- `@RestController` — наклейка, говорящая, что это контроллер и что все результаты его методов надо отдавать как JSON (а не как имя HTML-страницы).
- `@RequestMapping("/api/users")` — общий префикс пути для всех методов класса.
- `@GetMapping("/{id}")` — метод отвечает на `GET /api/users/{id}`. Часть `{id}` в фигурных скобках — это «дырка» в пути, куда подставится число.
- `@PathVariable Long id` — берёт значение из этой «дырки». Для запроса `GET /api/users/1` сюда попадёт `1`.
- `return new User(...)` — возвращаем обычный объект. Благодаря `@RestController` конвертер сам превратит его в JSON и отдаст со статусом `200 OK`.

**⚠️ Подводные камни.** Частые проблемы маппинга (сопоставления URL с методами):
- **Неоднозначные пути.** Если объявить `/{id}` и `/{name}` рядом, Spring не сможет различить их по типу — оба «дырки» в одном месте. Решение — задать ограничение через регулярное выражение, например `/{id:\\d+}` (только цифры). Почему так: Spring смотрит на форму URL, а не на тип переменной, и два одинаковых по форме шаблона для него неразличимы.
- **Разница `/users` и `/users/`** — слеш в конце может восприниматься как другой путь. Это легко приводит к неожиданным 404.
- **Кодировка кириллицы.** Если русские буквы «ломаются», нужно настроить `CharacterEncodingFilter` с UTF-8 — иначе байты раскодируются не в той кодировке.

**📌 Запомнить.** DispatcherServlet — единое окно (Front Controller): сам не считает, а раздаёт работу. Весь путь запроса проходит через HandlerMapping → HandlerAdapter → контроллер → конвертер/ViewResolver.

## 11. Контейнеры сервлетов: Tomcat, Jetty, Undertow, Netty

**Простыми словами.** Контейнер сервлетов — это «движок» веб-сервера, который принимает запросы по сети и отдаёт их твоему коду. Это как двигатель в машине: их бывает несколько марок, машина едет с любым, но у каждого свой характер. Spring Boot по умолчанию ставит один (Tomcat), но его можно заменить.

**Разбираемся.**
- **Контейнер сервлетов** — программа, которая держит сетевой порт, принимает HTTP-соединения и передаёт их сервлетам (а значит, и Spring MVC). Без него твоё приложение не услышало бы сеть.
- Ключевое различие между движками — **модель ввода-вывода (I/O)**:
  - **blocking, thread-per-request (блокирующая, поток на запрос)** — на каждый запрос выделяется отдельный поток, который «занят», пока запрос не обработан. Просто и понятно, но потоки — дорогой ресурс.
  - **non-blocking, event loop (неблокирующая, событийный цикл)** — один поток обслуживает много соединений, реагируя на события. Экономнее по памяти, но сложнее в коде.

Сравнение четырёх движков:

|Контейнер|Модель I/O|Особенности|Когда брать|
|---|---|---|---|
|**Tomcat**|blocking, поток на запрос|стоит по умолчанию в `spring-boot-starter-web`, зрелый, стабильный|большинство обычных (синхронных) приложений|
|**Jetty**|blocking + async|лёгкий, хорош для встраивания и асинхронных сценариев|embedded, асинхронные сценарии|
|**Undertow**|blocking + non-blocking|высокая производительность, мало памяти (от JBoss)|когда нужен максимум производительности синхронного стека|
|**Netty**|**non-blocking**, event loop|это НЕ классический сервлет-контейнер; работает **только с WebFlux** (реактивный стек)|реактивные приложения|

- Tomcat, Jetty и Undertow реализуют **Servlet API** — то есть классическую синхронную модель «поток на запрос».
- **Netty** — особняком: это событийная неблокирующая модель (event loop), её использует **реактивный стек Spring WebFlux**. Это не сервлет-контейнер в классическом смысле.
- Сменить движок можно: исключить (`exclude`) стартер Tomcat и подключить нужный.

**Разбор кода.**
```yaml
# Тюнинг пула потоков Tomcat (важно под нагрузкой)
server:
  tomcat:
    threads:
      max: 200
      min-spare: 10
    accept-count: 100
```
**Что здесь происходит:** это настройка пула потоков Tomcat в `application.yml`.
- `threads.max: 200` — максимум потоков (одновременно обрабатываемых запросов). Раз модель «поток на запрос», это и есть потолок параллелизма.
- `threads.min-spare: 10` — сколько потоков держать «про запас» наготове, даже когда нагрузки нет.
- `accept-count: 100` — длина очереди ожидания: сколько запросов поставить в очередь, когда все потоки заняты.

```xml
<!-- Заменить Tomcat на Undertow -->
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-web</artifactId>
  <exclusions>
    <exclusion>
      <groupId>org.springframework.boot</groupId>
      <artifactId>spring-boot-starter-tomcat</artifactId>
    </exclusion>
  </exclusions>
</dependency>
<dependency>
  <groupId>org.springframework.boot</groupId>
  <artifactId>spring-boot-starter-undertow</artifactId>
</dependency>
```
**Что здесь происходит:** это файл зависимостей Maven (`pom.xml`).
- Берём обычный веб-стартер `spring-boot-starter-web`, но через блок `<exclusions>` выкидываем из него `spring-boot-starter-tomcat` — то есть отключаем движок Tomcat.
- Отдельной зависимостью подключаем `spring-boot-starter-undertow` — и теперь приложение работает на Undertow вместо Tomcat.

**📌 Запомнить.** Tomcat/Jetty/Undertow — синхронные сервлет-контейнеры (поток на запрос); Tomcat стоит по умолчанию. Netty — неблокирующий event loop и только с WebFlux. Движок меняется через exclude стартера.

## 12. Формирование ответов: ResponseEntity, кастомные хедеры, CORS

**Простыми словами.** Когда метод просто возвращает объект, Spring сам заворачивает его в ответ со статусом 200. Но иногда тебе нужен полный контроль над «посылкой» — другой статус, особые ярлыки на коробке (заголовки), адрес созданного ресурса. Для этого есть `ResponseEntity` — это как самому собрать посылку: положить содержимое, наклеить нужные стикеры и выбрать «класс доставки».

**Разбираемся.**
- **`ResponseEntity<T>`** — объект, дающий полный контроль над HTTP-ответом: его статус-код, заголовки и тело. Его берут, когда нужен нестандартный статус (201, 204, 404), кастомные заголовки или `Location` (адрес созданного ресурса). Если же хватает простого `200 OK` с телом — можно вернуть объект напрямую (через `@ResponseBody`).
- **Статические фабрики** делают код читаемым (это короткие методы-«заготовки»):
  - `ResponseEntity.ok(body)` — 200 с телом;
  - `.created(uri)` — 201 с адресом созданного;
  - `.noContent().build()` — 204 (успех без тела);
  - `.notFound().build()` — 404;
  - `.status(HttpStatus.X)` — любой нужный статус.

**CORS (Cross-Origin Resource Sharing, совместное использование ресурсов между источниками)** — механизм, который разрешает браузеру делать запросы на **другой origin (источник)**. Origin — это связка «схема + домен + порт» (например, `https://site.com:443`).
- CORS срабатывает **только в браузере** — из-за защиты **Same-Origin Policy (политика одного источника)**, которая запрещает странице ходить на чужие источники без разрешения. У `curl`/Postman таких ограничений нет — поэтому классика «в браузере CORS-ошибка, а в Postman всё работает».
- Для «сложных» запросов браузер сначала шлёт **preflight (предварительный запрос)** методом `OPTIONS` — спрашивает сервер, можно ли вообще.
- Настройка — через `@CrossOrigin` на контроллере/методе или глобально через `WebMvcConfigurer`.

**Разбор кода.**
```java
@RestController
@RequestMapping("/api/users")
public class UserController {

    @PostMapping
    public ResponseEntity<User> create(@RequestBody User u) {
        User created = service.create(u);
        URI location = ServletUriComponentsBuilder.fromCurrentRequest()
            .path("/{id}").buildAndExpand(created.getId()).toUri();
        return ResponseEntity.created(location)        // 201 + Location
            .header("X-Custom-Header", "value")        // кастомный хедер
            .body(created);
    }

    @DeleteMapping("/{id}")
    public ResponseEntity<Void> delete(@PathVariable Long id) {
        service.delete(id);
        return ResponseEntity.noContent().build();     // 204 No Content
    }
}

@Configuration
class CorsConfig implements WebMvcConfigurer {
    @Override public void addCorsMappings(CorsRegistry r) {
        r.addMapping("/api/**")
         .allowedOrigins("https://my-frontend.com")    // белый список, НЕ "*"
         .allowedMethods("GET", "POST", "PUT", "DELETE")
         .allowedHeaders("*")
         .allowCredentials(true);
    }
}
```
**Что здесь происходит:**
- `create(...)` — создаёт пользователя. `ServletUriComponentsBuilder` строит URI нового ресурса (`/api/users/{id}`). `ResponseEntity.created(location)` отдаёт статус `201 Created` и кладёт адрес в заголовок `Location`. `.header("X-Custom-Header", "value")` добавляет собственный заголовок. `.body(created)` — тело ответа.
- `delete(...)` — после удаления возвращает `ResponseEntity.noContent().build()`, то есть `204 No Content`: успех, тела нет.
- `CorsConfig implements WebMvcConfigurer` — глобальная настройка CORS. `addMapping("/api/**")` — на какие пути распространять. `allowedOrigins("https://my-frontend.com")` — белый список разрешённых источников. `allowedMethods(...)` — разрешённые методы. `allowedHeaders("*")` — любые заголовки. `allowCredentials(true)` — разрешить отправку куки/учётных данных.

**⚠️ Подводные камни.**
- **В production нельзя `allowedOrigins("*")`** — то есть нельзя разрешать «всем подряд». Нужен белый список конкретных доменов. Почему: звёздочка открывает твоё API любому сайту в интернете, что особенно опасно вместе с `allowCredentials(true)` (тогда чужой сайт сможет дёргать твоё API от имени залогиненного пользователя).

**📌 Запомнить.** `ResponseEntity` — полный контроль над статусом, заголовками и телом; для простого 200 хватает возврата объекта. CORS — браузерная история (Same-Origin Policy), поэтому в Postman ошибки нет; в проде только белый список, не `*`.

## 13. Контроллеры, параметры, DTO, маппинг, валидация

**Простыми словами.** Контроллер — это «приёмное окно» API: он принимает запрос, разбирает параметры и отдаёт ответ. Чтобы не выставлять наружу внутреннюю кухню (таблицы БД, пароли), наружу отдают не «сырую» сущность, а специально подготовленную «выписку» — DTO. А чтобы в окно не пролезали кривые данные, на входе стоит контролёр-валидатор.

**Разбираемся.**

Основные аннотации:
- `@Controller` возвращает имя view (это для MVC с шаблонизатором — отдаёт HTML-страницы). А **`@RestController` = `@Controller` + `@ResponseBody`** — тело результата сериализуется в JSON.
- `@RequestMapping` (и его «короткие» варианты `@GetMapping`, `@PostMapping` и т. д.) — задаёт URL и HTTP-метод.
- `@RequestBody` — десериализовать JSON из тела запроса в объект; `@ResponseBody` — сериализовать результат обратно в тело ответа.
- `@PathVariable` — переменная из пути (`/users/{id}`); `@RequestParam` — query-параметр (`?page=2&size=20`), поддерживает `defaultValue` (значение по умолчанию) и `required` (обязателен ли).

**DTO (Data Transfer Object, объект передачи данных)** — отдельный объект, через который данные отдают наружу или гоняют между слоями. Зачем не отдавать **Entity (сущность БД)** напрямую:
- **защита от over-posting** — клиент не сможет задать лишние поля, которые ему задавать не положено;
- **сокрытие внутренних полей** — пароли, технические id наружу не уходят;
- **независимость API от схемы БД** — поменяли таблицу, а контракт API остался прежним;
- **версионирование** — можно держать `v1` и `v2` API одновременно;
- **разрыв lazy-связей Hibernate** — иначе при сериализации Hibernate может полезть в базу за «ленивыми» данными и кинуть `LazyInitializationException`.
- **Маппинг DTO↔Entity** (перекладывание данных между ними) делают вручную или через библиотеку MapStruct.

**Валидация на уровне контроллера** — это `@Valid` плюс **Bean Validation** (стандартный набор проверок: `@NotNull`, `@NotBlank`, `@Email`, `@Size`, `@Min`). Если проверка не прошла, бросается `MethodArgumentNotValidException`, его ловит `@ControllerAdvice` и возвращает клиенту `400`.

**Разбор кода.**
```java
public record UserCreateDto(
    @NotBlank String username,
    @Email     String email,
    @Min(18)   int    age) {}

@RestController
@RequestMapping("/api/v1/users")
public class UserController {

    @PostMapping
    public ResponseEntity<UserDto> create(@Valid @RequestBody UserCreateDto in) {
        User entity = new User(in.username(), in.email(), in.age()); // DTO -> Entity
        User saved  = service.save(entity);
        return ResponseEntity.status(HttpStatus.CREATED).body(toDto(saved)); // Entity -> DTO
    }

    // /api/v1/users/42 + ?includeDeleted=true
    @GetMapping("/{id}")
    public UserDto get(@PathVariable Long id,
                       @RequestParam(defaultValue = "false") boolean includeDeleted) {
        return toDto(service.findById(id, includeDeleted));
    }
}
```
**Что здесь происходит:**
- `record UserCreateDto(...)` — это DTO для входных данных. **record (запись)** — компактный неизменяемый класс только под хранение полей. На полях висят проверки: `@NotBlank` (строка не пустая), `@Email` (валидный email), `@Min(18)` (возраст не меньше 18).
- `create(@Valid @RequestBody UserCreateDto in)` — `@RequestBody` берёт JSON из тела и превращает в `UserCreateDto`, а `@Valid` запускает проверки. Если что-то не так — 400 ещё до входа в метод. Внутри DTO перекладывается в Entity, сохраняется, и обратно отдаётся DTO со статусом `201 CREATED`.
- `get(@PathVariable Long id, @RequestParam(defaultValue = "false") boolean includeDeleted)` — `id` берётся из пути, а `includeDeleted` — из query-параметра. Если параметр не передан, `defaultValue = "false"` подставит `false`.

**⚠️ Подводные камни.**
- Не отдавай Entity напрямую — иначе словишь over-posting, утечку внутренних полей, жёсткую привязку API к схеме БД и риск `LazyInitializationException` при сериализации «ленивых» связей Hibernate. Именно поэтому существует слой DTO.

**📌 Запомнить.** `@RestController` = контроллер + JSON. DTO отделяет «внешний» контракт API от «внутренней» Entity (безопасность, версионирование, разрыв lazy-связей). `@Valid` + Bean Validation проверяет вход и даёт 400 при нарушении.

## 14. Загрузка файлов и работа с состоянием

**Простыми словами.** HTTP по своей природе «беспамятный» (stateless): каждый запрос сервер видит как будто впервые, без воспоминаний о предыдущем. Чтобы всё-таки «узнавать» клиента и хранить контекст, используют заголовки, куки и сессии. А ещё через те же запросы пользователи присылают файлы — и тут важно не утонуть в памяти и аккуратно их хранить.

**Разбираемся.**

Загрузка файлов — два подхода:
- **`MultipartFile`** — стандартный способ для `multipart/form-data` (так шлют файлы HTML-формы). Spring сам разбирает «части» запроса. Лимиты на размер задаются в `application.yml`.
- **raw (сырое тело)** — чтение тела как `InputStream`/`byte[]` (бинарная загрузка, тип `application/octet-stream`). Для больших файлов лучше **стримить** (потоково перекачивать: `request.getInputStream().transferTo(out)`), а не грузить целиком в память.

Состояние (HTTP stateless):
- **Заголовки** — читаются через `@RequestHeader`.
- **Куки (cookies)** — читаются через `@CookieValue`; устанавливаются через `ResponseCookie` с защитными флагами:
  - `httpOnly` — куку нельзя прочитать из JavaScript (защита от кражи через XSS);
  - `secure` — куку отправляют только по HTTPS;
  - `sameSite` — ограничивает отправку куки с чужих сайтов (защита от CSRF);
  - `maxAge` — срок жизни; `path` — для каких путей действует.
  - Куки хранятся на стороне клиента, весят до 4KB, уязвимы к **XSS** (внедрение чужого скрипта) и **CSRF** (подделка запроса) — поэтому защитные флаги обязательны.
- **Сессии** — `HttpSession` — это серверное состояние, привязанное к клиенту куки `JSESSIONID`. В stateless API (где используют JWT-токены) сессии не применяют.

**Разбор кода.**
```java
@RestController
@RequestMapping("/api/files")
public class FileController {

    @PostMapping("/upload")
    public ResponseEntity<?> upload(@RequestParam("file") MultipartFile file) throws IOException {
        if (file.isEmpty()) return ResponseEntity.badRequest().body("empty");
        String id = storage.store(file); // сервис хранит файл + метаданные в БД
        return ResponseEntity.ok(Map.of("id", id, "name", file.getOriginalFilename(),
                                        "size", file.getSize()));
    }

    @PostMapping("/login")
    public ResponseEntity<?> login(@RequestBody LoginRequest req) {
        String token = auth(req);
        ResponseCookie cookie = ResponseCookie.from("authToken", token)
            .httpOnly(true).secure(true).path("/")
            .maxAge(Duration.ofDays(7)).sameSite("Strict").build(); // защита XSS/CSRF
        return ResponseEntity.ok().header(HttpHeaders.SET_COOKIE, cookie.toString()).build();
    }

    @GetMapping("/me")
    public String me(@RequestHeader("User-Agent") String ua,
                     @CookieValue(value = "theme", defaultValue = "light") String theme,
                     HttpSession session) {
        session.setAttribute("lastVisit", Instant.now());
        return ua + " / " + theme;
    }
}
```
**Что здесь происходит:**
- `upload(@RequestParam("file") MultipartFile file)` — принимает загруженный файл. Если он пустой — сразу `400`. Иначе сервис хранения сохраняет файл и его метаданные, а в ответ уходит JSON с id, оригинальным именем и размером.
- `login(...)` — после проверки логина создаёт куку `authToken` с токеном и защитными флагами `httpOnly`, `secure`, `sameSite("Strict")`, сроком 7 дней. Кука кладётся в заголовок `Set-Cookie`.
- `me(...)` — показывает чтение состояния: `@RequestHeader("User-Agent")` берёт заголовок, `@CookieValue(... defaultValue = "light")` берёт куку `theme` (или `light`, если её нет), а в `HttpSession` пишется время последнего визита.

```yaml
spring:
  servlet:
    multipart:
      max-file-size: 10MB
      max-request-size: 25MB
      file-size-threshold: 2KB  # меньше — в памяти, больше — во временный файл
```
**Что здесь происходит:** это лимиты загрузки в `application.yml`.
- `max-file-size: 10MB` — максимальный размер одного файла.
- `max-request-size: 25MB` — максимальный размер всего запроса (нескольких файлов вместе).
- `file-size-threshold: 2KB` — порог: всё, что меньше, держится в памяти, что больше — сбрасывается во временный файл (чтобы не забивать память).

**⚠️ Подводные камни.**
- **Правило хранения файлов:** сам файл кладут во внешнее хранилище (диск или S3), а **метаданные — в БД** (id, оригинальное имя, размер, MIME-тип, ссылка, владелец). Почему так: имея метаданные в базе, можно делать запросы (найти файлы пользователя, удалить устаревшие) без копания в файловой системе. Дополнительно проверяй MIME-тип и «магические числа» (первые байты файла, по которым опознают формат) и генерируй безопасное имя файла, чтобы загруженное имя не сломало ничего на сервере.

**📌 Запомнить.** Файлы: `MultipartFile` для форм, raw-стриминг для больших; файл — во внешнее хранилище, метаданные — в БД. Состояние при stateless HTTP держат в заголовках/куки/сессии; куки защищают флагами `httpOnly`/`secure`/`sameSite`.

📖 _`Longread_4.md` (MultipartFile, raw-стриминг, хранение метаданных, куки с флагами, сессии, заголовки)._

## 15. Обработка ошибок: @ControllerAdvice, @ExceptionHandler

**Простыми словами.** Без централизованной обработки в каждом контроллере пришлось бы писать свои try/catch — и формат ошибок везде был бы разный. `@ControllerAdvice` — это как единая «служба рекламаций» на весь дом: что бы ни сломалось в любом контроллере, она ловит проблему, оформляет аккуратный единый ответ клиенту и прячет внутренние детали в лог.

**Разбираемся.**
- **`@ExceptionHandler`** — метод, который обрабатывает конкретный тип исключения. Может работать локально (внутри одного контроллера) или глобально.
- **`@ControllerAdvice` / `@RestControllerAdvice`** — глобальный перехватчик для всех контроллеров сразу. Он даёт единый формат ошибок, отдельные обработчики под разные типы исключений и позволяет разделять бизнес-ошибки (400 — виноват клиент) и системные (500 — виноват сервер).

Зачем централизовать обработку:
- единый JSON-формат ошибки (timestamp, status, message);
- внутренние детали не утекают клиенту (**stack trace** — цепочка вызовов до сбоя — уходит только в лог);
- правильные HTTP-статусы;
- нет дублирования try/catch в каждом контроллере.

Дополнительно: несколько `@ControllerAdvice` упорядочиваются аннотацией `@Order`. Современный стандарт для тела ошибки — **Problem Details (RFC 7807)** / класс `ProblemDetail`.

**Разбор кода.**
```java
@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(EntityNotFoundException.class)
    public ResponseEntity<Map<String,Object>> notFound(EntityNotFoundException ex) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND).body(Map.of(
            "timestamp", Instant.now(), "status", 404,
            "error", "Not Found", "message", ex.getMessage()));
    }

    @ExceptionHandler(MethodArgumentNotValidException.class) // ошибки @Valid
    public ResponseEntity<Map<String,Object>> validation(MethodArgumentNotValidException ex) {
        Map<String,String> fields = ex.getBindingResult().getFieldErrors().stream()
            .collect(Collectors.toMap(FieldError::getField, FieldError::getDefaultMessage));
        return ResponseEntity.badRequest().body(Map.of(
            "error", "Validation failed", "fields", fields));   // 400
    }

    @ExceptionHandler(DataAccessException.class) // ошибки БД
    public ResponseEntity<Map<String,Object>> db(DataAccessException ex) {
        // полный stack trace — в лог, клиенту — общее сообщение
        return ResponseEntity.status(500).body(Map.of("error", "Database Error"));
    }
}
```
**Что здесь происходит:**
- `@RestControllerAdvice` — глобальный обработчик ошибок, отдающий JSON, для всех контроллеров.
- `notFound(...)` — ловит `EntityNotFoundException` (не нашли объект) и возвращает `404` с аккуратным телом: timestamp, status, error, message.
- `validation(...)` — ловит `MethodArgumentNotValidException` (это ошибки `@Valid`), собирает карту «поле → сообщение об ошибке» и отдаёт `400 Bad Request`. Так клиент видит, какие именно поля кривые.
- `db(...)` — ловит `DataAccessException` (ошибки БД) и отдаёт `500` с общим сообщением. Обрати внимание: полный stack trace идёт только в лог, а клиенту — обтекаемое «Database Error», чтобы не раскрывать внутренности.

**⚠️ Подводные камни.**
- Никогда не отдавай клиенту stack trace и внутренние детали ошибки — это утечка информации, которой могут воспользоваться. Полный stack trace — только в лог, клиенту — общее сообщение и корректный статус. Именно поэтому в обработчике БД-ошибок клиент видит лишь «Database Error».

**📌 Запомнить.** `@ExceptionHandler` ловит конкретный тип исключения; `@(Rest)ControllerAdvice` делает это глобально и единообразно. Бизнес-ошибки → 4xx, системные → 5xx; детали — в лог, не клиенту.

## 16. HTTP, REST, OpenAPI + Spring

**Простыми словами.** REST — это набор «правил хорошего тона» для API: к данным обращаются по адресам (URI), а что с ними делать — говорят HTTP-методом (получить, создать, заменить, удалить). А OpenAPI — это «техпаспорт» твоего API в виде файла: по нему люди и программы понимают, какие есть эндпоинты, что они принимают и возвращают, и даже могут сгенерировать готовый код.

**Разбираемся.**

**REST** — архитектурный стиль:
- ресурсы адресуются через **URI** (адреса);
- действия выражаются HTTP-методами: **GET** — чтение, **POST** — создание, **PUT** — полная замена, **PATCH** — частичное обновление, **DELETE** — удаление;
- сервер **stateless** (без памяти между запросами);
- корректные статус-коды (2xx — успех, 4xx — ошибка клиента, 5xx — ошибка сервера);
- **HATEOAS** (когда ответ содержит ссылки на связанные действия) — опционально.
- **Идемпотентность** (повторный запрос даёт тот же результат, что и один): GET/PUT/DELETE идемпотентны, POST — нет (повторный POST создаст дубль).

**OpenAPI** — стандарт описания REST API (спецификация в формате YAML/JSON). Два подхода:
- **Code-first (сначала код)** — библиотека `springdoc-openapi` сама генерирует спецификацию и **Swagger UI** (интерактивную веб-страницу с документацией) из аннотированного кода (`@Operation`, `@ApiResponse`, `@Schema`, `@Tag`, `@Parameter`). Быстро, но контракт «отстаёт» от реализации (сначала пишут код, потом из него получается описание).
- **Contract-first (сначала контракт)** — сначала пишут спецификацию (контракт = единый источник правды), а затем **`openapi-generator`** по ней генерирует:
  - **серверные интерфейсы контроллеров** (их реализует разработчик),
  - **клиентов** (готовые классы для вызова чужого API),
  - а также DTO.
  Удобно, когда нужно согласовать API между разными командами или сервисами.

**Разбор кода.**
```java
@RestController
@Tag(name = "Пользователи", description = "Управление пользователями")
public class UserController {

    @Operation(summary = "Получить пользователя по ID")
    @ApiResponses({
        @ApiResponse(responseCode = "200", description = "Найден",
            content = @Content(schema = @Schema(implementation = UserDto.class))),
        @ApiResponse(responseCode = "404", description = "Не найден")
    })
    @GetMapping("/api/users/{id}")
    public ResponseEntity<UserDto> get(
        @Parameter(description = "ID", example = "1") @PathVariable Long id) { /* ... */ }
}
// springdoc: спецификация на /v3/api-docs, UI на /swagger-ui.html
```
**Что здесь происходит:** это пример подхода code-first.
- `@Tag(...)` — группирует методы под общим разделом «Пользователи» в документации.
- `@Operation(summary = ...)` — краткое описание операции.
- `@ApiResponses` / `@ApiResponse` — описывают возможные ответы: 200 (найден, с указанием схемы `UserDto`) и 404 (не найден).
- `@Schema(implementation = UserDto.class)` — говорит, какой класс описывает тело ответа.
- `@Parameter(description = "ID", example = "1")` — описание и пример значения параметра пути.
- Комментарий внизу напоминает: spec доступна на `/v3/api-docs`, а Swagger UI — на `/swagger-ui.html`.

```gradle
// contract-first: генерация по спецификации
plugins { id "org.openapi.generator" version "7.x" }
openApiGenerate {
    generatorName = "spring"
    inputSpec = "$rootDir/specs/users-api.yaml"
    apiPackage = "com.example.api"; modelPackage = "com.example.api.dto"
}
```
**Что здесь происходит:** это пример подхода contract-first в файле сборки Gradle.
- Подключается плагин `org.openapi.generator`.
- `generatorName = "spring"` — генерировать код под Spring.
- `inputSpec = ".../users-api.yaml"` — путь к заранее написанной спецификации (тот самый «контракт = источник правды»).
- `apiPackage` / `modelPackage` — куда положить сгенерированные интерфейсы API и классы DTO.

**📌 Запомнить.** REST: ресурсы — это URI, действия — HTTP-методы, сервер stateless, верные статус-коды; GET/PUT/DELETE идемпотентны, POST — нет. OpenAPI: code-first (springdoc генерирует spec из кода) против contract-first (openapi-generator генерирует код из spec).

📖 _`Longread_5.md` (REST, OpenAPI), `Seminar_5.md` (Swagger-аннотации, документирование DTO, Swagger UI)._

## 17. Логирование: SLF4J, уровни, конфигурация, MDC

**Простыми словами.** Логи — это бортовой журнал приложения: записи о том, что и когда происходило. SLF4J тут — как универсальная розетка-переходник: твой код «втыкается» в неё, а какой именно «прибор» (реализацию логирования) подключить за ней — Logback или Log4j2 — решаешь ты, не меняя сам код.

**Разбираемся.**
- **SLF4J (Simple Logging Facade for Java)** — это **фасад (API)**, а не реализация. То есть это единый интерфейс, за которым стоит настоящий движок логирования. Реализации: **Logback** (по умолчанию в Spring Boot) и Log4j2.
- Зачем нужен фасад: библиотеки пишут логи через SLF4J, а ты выбираешь реализацию (Logback или Log4j2) **без переписывания кода**. Всегда импортируй `org.slf4j.Logger`/`LoggerFactory` (или используй Lombok-аннотацию `@Slf4j`). `System.out.println` в серверном коде запрещён.

**Уровни логирования** (по возрастанию важности):
- **TRACE** — детальнейшая диагностика, почти никогда не в проде;
- **DEBUG** — для разработчика: ветвления, значения параметров;
- **INFO** — бизнес-события («заказ создан»);
- **WARN** — необычно, но работаем дальше («внешний сервис медленный»);
- **ERROR** — сбой, исключение.

Критерий выбора: INFO — то, что хочешь видеть каждый день; DEBUG — когда «что-то странное» и нужно копнуть; ERROR — то, что требует внимания.

**Правила логирования:**
- **параметризация через `{}`** — она ленивая: строка не собирается, если уровень выключен, и это быстрее обычной конкатенации (склейки строк через `+`);
- не логировать секреты и **PII (персональные данные)** — пароли, токены, номера карт нужно маскировать;
- одно событие — одна осмысленная запись;
- логировать «что делали» и «что получилось» (особенно вокруг интеграций с внешними сервисами).

**MDC (Mapped Diagnostic Context, отображаемый диагностический контекст)** — хранилище, привязанное к потоку (на базе `ThreadLocal`), куда кладут контекст запроса (`traceId`, `userId`). Эти значения автоматически подставляются в каждую строку лога — и можно **скоррелировать** (связать) все записи одного запроса. Обязательно вызывать `MDC.clear()` — особенно при работе с пулом потоков, иначе значение «протечёт» в чужой запрос (поток переиспользуется, и старый `traceId` прилипнет к новому запросу).

**Разбор кода.**
```java
@Slf4j               // Lombok: private static final Logger log
@Service
public class PaymentService {
    public void pay(Long orderId) {
        MDC.put("traceId", UUID.randomUUID().toString()); // попадёт во все логи потока
        try {
            log.info("Payment started: orderId={}", orderId);   // параметризация
            // ...
            log.info("Payment ok: orderId={}", orderId);
        } catch (Exception e) {
            log.error("Payment failed: orderId={}", orderId, e); // исключение последним аргументом
            throw e;
        } finally {
            MDC.clear(); // ОБЯЗАТЕЛЬНО
        }
    }
}
```
**Что здесь происходит:**
- `@Slf4j` — Lombok-аннотация, которая сама создаёт поле `private static final Logger log`. Не надо писать логгер вручную.
- `MDC.put("traceId", ...)` — кладёт уникальный идентификатор запроса в контекст; он попадёт во все строки лога этого потока.
- `log.info("Payment started: orderId={}", orderId)` — параметризованный лог: `{}` заменится на `orderId`. Строка собирается лениво.
- `log.error("Payment failed: orderId={}", orderId, e)` — исключение `e` передаётся последним аргументом: тогда логгер запишет полный stack trace.
- `MDC.clear()` в `finally` — обязательная очистка контекста, чтобы `traceId` не «протёк» в следующий запрос на том же потоке.

```yaml
logging:
  level:
    root: INFO
    com.example.demo: DEBUG
    org.hibernate.SQL: WARN
  pattern:
    console: "%d{HH:mm:ss.SSS} %-5level [%thread] %logger{36} trace=%X{traceId} - %msg%n"
```
**Что здесь происходит:** это настройка логирования в `application.yml`.
- `level.root: INFO` — общий уровень по умолчанию.
- `com.example.demo: DEBUG` — для своего пакета включаем подробный DEBUG.
- `org.hibernate.SQL: WARN` — для Hibernate приглушаем до WARN, чтобы не засорять логи.
- `pattern.console` — шаблон строки лога: время, уровень, поток, логгер, и `trace=%X{traceId}` — это как раз подстановка значения из MDC.

**⚠️ Подводные камни.**
- **Обязательно `MDC.clear()`** (особенно с пулом потоков). Почему: потоки переиспользуются, и если не очистить контекст, `traceId` (или `userId`) от предыдущего запроса «протечёт» в чужой следующий запрос — и логи разных запросов перепутаются.
- Никогда не логируй секреты и персональные данные (пароли, токены, карты) — их нужно маскировать, иначе они утекут в журналы.

**📌 Запомнить.** SLF4J — фасад, реализация (Logback по умолчанию) меняется без правки кода. Уровни: TRACE→DEBUG→INFO→WARN→ERROR. Параметризация `{}` лениво и безопасно; MDC связывает логи одного запроса, но требует `MDC.clear()`.

📖 _`Longread_9.md` (SLF4J/Logback/Log4j2, уровни, правила, MDC), `Seminar_9.md` (параметризация, уровни, pattern, access log через фильтр)._

## 18. Тестирование: пирамида, виды, инструменты

**Простыми словами.** Тесты бывают разные: одни проверяют один маленький класс, другие — как всё работает вместе, третьи — всю систему глазами пользователя. Хорошее правило — представить их пирамидой: внизу много быстрых и дешёвых мелких тестов, наверху мало медленных и дорогих больших. Так и багов ловишь много, и тесты не тормозят.

**Разбираемся.**

**Пирамида тестирования** (снизу вверх — чем ниже, тем больше, быстрее и дешевле):
1. **Статический анализ** — без запуска кода (компилятор, Checkstyle) — мгновенно.
2. **Unit (модульные)** — изолированный класс, зависимости заменяются заглушками (**мокаются** через Mockito), без инфраструктуры — миллисекунды, самый большой слой.
3. **Интеграционные** — проверяют взаимодействие компонентов (сервис + БД, контроллер + сервис), ловят проблемы маппинга/SQL/транзакций — медленнее.
4. **E2E (end-to-end, сквозные)** — вся система целиком глазами пользователя — самые медленные и хрупкие, их минимум.

> Если код трудно тестировать — это сигнал плохой архитектуры (высокая связность, создание зависимостей через `new` вместо DI, глобальное состояние).

**Две школы тестирования:**
- **лондонская (mockist)** — мокать все зависимости и проверять взаимодействия через `verify()`;
- **детройтская/классическая** — проверять состояние (результат), а мокать по минимуму.

**Инструменты Spring:**
- **`@SpringBootTest`** — поднимает полный контекст приложения (для интеграции/E2E). Варианты: `webEnvironment=NONE` (без веба), `RANDOM_PORT` (реальный сервер на случайном порту).
- **`@WebMvcTest(Controller.class)`** — поднимает только web-слой (контроллер + конвертеры + advice), без сервиса и БД; зависимости мокаются.
- **`@DataJpaTest`** — только JPA-слой, in-memory или Testcontainers-БД, **откатывает транзакции** после каждого теста, даёт `TestEntityManager`.
- **`@MockBean`/`@MockitoBean`** — заменить бин в контексте моком; **`@SpyBean`** — частичный мок (реальный объект, но можно переопределить отдельные методы и проверять вызовы).
- **`MockMvc`** — отправляет фейковые HTTP-запросы без реальной сети; проверяет статус и JSON (через `jsonPath`).
- **Testcontainers** — поднимает реальную БД (например, PostgreSQL) в Docker — это точнее, чем H2 (тот же диалект SQL). `@DynamicPropertySource` подменяет настройки подключения. Контейнер делают `static` (один на класс) или через JUnit Extension (один на все тесты).

**Разбор кода.**
```java
// Unit-тест сервиса (Mockito)
@ExtendWith(MockitoExtension.class)
class UserServiceTest {
    @Mock UserRepository repo;
    @InjectMocks UserService service;

    @Test void createsUser() {
        when(repo.existsByEmail("a@x.com")).thenReturn(false);
        when(repo.save(any())).thenAnswer(i -> i.getArgument(0));
        service.create("a", "a@x.com", 20);
        verify(repo).save(any());                 // проверка взаимодействия
    }
}

// Web-слой
@WebMvcTest(UserController.class)
class UserControllerTest {
    @Autowired MockMvc mvc;
    @MockitoBean UserService service;
    @Test void returnsUser() throws Exception {
        when(service.getById(1L)).thenReturn(new User(1L, "Alice"));
        mvc.perform(get("/api/users/1"))
           .andExpect(status().isOk())
           .andExpect(jsonPath("$.username").value("Alice"));
    }
}

// Интеграция репозитория с реальной PostgreSQL
@Testcontainers @DataJpaTest
class UserRepositoryIT {
    @Container static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:15-alpine");
    @DynamicPropertySource static void props(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", pg::getJdbcUrl);
        r.add("spring.datasource.username", pg::getUsername);
        r.add("spring.datasource.password", pg::getPassword);
    }
    @Autowired UserRepository repo;
    @Test void savesAndFinds() { /* ... */ }
}
```
**Что здесь происходит:**
- **Unit-тест.** `@ExtendWith(MockitoExtension.class)` включает Mockito. `@Mock UserRepository repo` — заглушка репозитория; `@InjectMocks UserService service` — реальный сервис, в который вставляется этот мок. `when(...).thenReturn(...)` задаёт поведение заглушки. `verify(repo).save(any())` проверяет, что метод `save` был вызван — это и есть проверка взаимодействия (лондонский стиль).
- **Web-слой.** `@WebMvcTest(UserController.class)` поднимает только контроллер. `@MockitoBean UserService service` подменяет сервис моком. `MockMvc` шлёт фейковый `GET /api/users/1`, а `andExpect(...)` проверяет статус `200 OK` и что в JSON поле `$.username` равно `Alice`.
- **Интеграционный тест.** `@Testcontainers @DataJpaTest` поднимает JPA-слой и реальную PostgreSQL в Docker. `@Container static PostgreSQLContainer<?> pg` — сам контейнер (static — один на класс). `@DynamicPropertySource` подставляет URL, логин и пароль запущенного контейнера в настройки подключения.

**⚠️ Подводные камни.**
- **Почему `@DataJpaTest` откатывает изменения?** Каждый тест выполняется в транзакции, которая в конце откатывается (rollback). Так тесты не влияют друг на друга — база остаётся чистой, и порядок тестов не имеет значения.
- **Почему Testcontainers лучше H2?** Testcontainers поднимает ту же СУБД, что и в проде (тот же диалект, типы, функции). H2 — это другая in-memory база, и на специфичном SQL она «врёт»: тест проходит, а в проде ломается.

**📌 Запомнить.** Пирамида: много быстрых unit внизу, мало медленных E2E наверху. `@WebMvcTest` — web-слой, `@DataJpaTest` — JPA-слой (с откатом транзакций), `@SpringBootTest` — весь контекст. Testcontainers точнее H2, потому что это та же СУБД, что в проде.

📖 _`Longread_11.md` (пирамида, виды, школы, тестируемость и архитектура), `Seminar_11.md` (@WebMvcTest, @DataJpaTest, @SpringBootTest+Mockito, Testcontainers, кастомный Extension)._

## 1. JDBC: как Java разговаривает с базой данных напрямую

**Простыми словами.** Представь, что база данных — это магазин в другом городе, а тебе нужно туда позвонить и сделать заказ. **JDBC** — это телефонная линия и язык, на котором ты с магазином говоришь. Линия общая (стандартная для всех магазинов), но чтобы дозвониться до конкретного магазина, нужен его «номер» и переходник под него — это **драйвер**. Сам разговор стоит дорого: каждый дозвон — это набрать номер, представиться, подождать соединения. Поэтому звонить по сто раз подряд — расточительно.

**Разбираемся.**

**JDBC (Java Database Connectivity)** — низкоуровневый стандарт доступа к реляционным базам данных из Java. «Низкоуровневый» значит, что ты пишешь всё руками: открыть соединение, написать SQL, разобрать ответ, закрыть соединение.

- **Driver (драйвер)** — это конкретная реализация JDBC под конкретную СУБД. Например, для PostgreSQL это «PostgreSQL JDBC Driver». Драйвер — переходник между общим стандартом и особенностями именно этой базы.
- Соединение создаётся вызовом `DriverManager.getConnection(url, user, pass)`, который возвращает объект **`Connection`** (соединение). Это **дорогая операция**: под капотом происходит TCP-рукопожатие (установка сетевого канала), аутентификация (проверка логина и пароля), инициализация сессии. К тому же `Connection` **не потокобезопасен** — его нельзя одновременно использовать из нескольких потоков.

Три главных рабочих инструмента JDBC:

- **`Statement`** — отправка простого SQL без параметров. Уязвим к **SQL-инъекциям** (это атака, когда злоумышленник подсовывает в данные кусок SQL-кода, и он выполняется как часть запроса) — потому что данные просто склеиваются в строку запроса. Плюс база заново разбирает (парсит) такой запрос при каждом выполнении.
- **`PreparedStatement`** — параметризованный запрос. Вместо значений ставятся плейсхолдеры `?`, а сами значения подставляются методами `setString`, `setLong` и т. д. (важно: нумерация параметров начинается **с 1**, а не с 0). Преимущества: предкомпиляция и кеширование плана запроса (база один раз разбирает запрос и потом переиспользует план), и **защита от инъекций** — данные передаются отдельно от текста SQL на уровне сетевого протокола (так называемый extended query), и драйвер не воспринимает их как код. **Всегда предпочтительнее `Statement`.**
- **`ResultSet`** — курсор (бегунок) по результату `SELECT`-запроса. Двигаешься по строкам методом `next()`, читаешь колонки методами `getXxx("имя_колонки")`. Требует, чтобы соединение оставалось открытым, пока ты читаешь.

**Транзакции в JDBC.** **Транзакция** — это набор операций, которые должны выполниться либо все вместе, либо никак (всё или ничего). В JDBC это делается так: `setAutoCommit(false)` отключает автоматическую фиксацию каждого запроса и начинает транзакцию, затем при успехе вызывается `commit()` (зафиксировать изменения), а при ошибке — `rollback()` (откатить, как будто ничего не было). Ресурсы (`Connection`, `Statement`, `ResultSet`) нужно закрывать в обратном порядке их открытия — проще всего через **try-with-resources** (конструкция Java, которая закрывает ресурсы автоматически), иначе будут утечки ресурсов.

**Разбор кода.**

```java
String url = "jdbc:postgresql://localhost:5432/db";
try (Connection conn = DriverManager.getConnection(url, "user", "pass")) {
    conn.setAutoCommit(false);                              // начало транзакции
    try (PreparedStatement ps = conn.prepareStatement(
             "INSERT INTO users(username, email) VALUES (?, ?)")) {
        ps.setString(1, "alice");                          // защита от инъекций
        ps.setString(2, "a@x.com");
        ps.executeUpdate();
        conn.commit();                                     // фиксация
    } catch (SQLException e) {
        conn.rollback();                                   // откат
        throw e;
    }
}
```

**Что здесь происходит:**
- `url` — адрес базы: протокол `jdbc:postgresql`, хост, порт и имя базы `db`.
- `try (Connection conn = DriverManager.getConnection(...))` — открываем соединение внутри try-with-resources, чтобы оно само закрылось в конце.
- `conn.setAutoCommit(false)` — отключаем автокоммит, то есть начинаем транзакцию вручную.
- `prepareStatement(...)` с плейсхолдерами `?` — создаём безопасный параметризованный запрос.
- `ps.setString(1, "alice")` и `ps.setString(2, "a@x.com")` — подставляем значения в первый и второй `?` (нумерация с 1). Данные идут отдельно от SQL — отсюда защита от инъекций.
- `ps.executeUpdate()` — выполняем запрос на изменение данных (INSERT/UPDATE/DELETE).
- `conn.commit()` — если всё прошло хорошо, фиксируем изменения.
- `catch (SQLException e)` → `conn.rollback()` — если случилась ошибка, откатываем всё и пробрасываем исключение дальше.

Сравнение двух способов отправки SQL:

||`Statement`|`PreparedStatement`|
|---|---|---|
|Параметры|нет|`?` + `setXxx`|
|Производительность|парсинг каждый раз|предкомпиляция + кеш плана|
|Безопасность|уязвим к инъекциям|защищён|

**⚠️ Подводные камни.**
- Создание `Connection` — дорогая операция (TCP-рукопожатие + аутентификация). Поэтому открывать новое соединение под каждый запрос неэффективно — это решается пулом соединений (см. тему 2).
- `Connection` не потокобезопасен — нельзя гонять одно соединение из нескольких потоков одновременно.
- Никогда не используй `Statement` с пользовательскими данными — это прямая дорога к SQL-инъекции. Всегда `PreparedStatement`.
- Если не закрывать ресурсы (или закрывать не в обратном порядке), будут утечки. try-with-resources спасает.

**📌 Запомнить.** `PreparedStatement` всегда лучше `Statement`: безопасно (нет инъекций) и быстрее (кеш плана). Транзакция в JDBC = `setAutoCommit(false)` → `commit()` или `rollback()`. Ресурсы закрывать через try-with-resources.

## 2. Spring JDBC: пул соединений, DataSource и JdbcTemplate

**Простыми словами.** «Голый» JDBC из прошлой темы — это как каждый раз заново звонить в магазин: набрать номер, дождаться соединения, поговорить, повесить трубку. Утомительно и медленно. **Пул соединений** — это как держать несколько уже установленных линий открытыми и переиспользовать их (не вешать трубку, а передавать следующему). А **`JdbcTemplate`** — это помощник-секретарь, который сам открывает и закрывает линии, обрабатывает ошибки и переводит ответ магазина в удобные объекты. Тебе остаётся только продиктовать сам запрос.

**Разбираемся.**

«Голый» JDBC требует много **boilerplate** (повторяющегося шаблонного кода): открыть и закрыть ресурсы, обработать исключения, вручную замапить `ResultSet` в объекты. Это рискует утечками, привязано к диалекту конкретной СУБД и плохо тестируется. **Spring JDBC** это решает тремя кирпичами:

- **`DataSource`** — это фабрика соединений, стандартная абстракция (приходит на смену `DriverManager`). Обычно работает поверх пула соединений.
- **Connection Pool (пул соединений)** — переиспользование уже открытых соединений. Самое дорогое в JDBC — это создание соединения (TCP + аутентификация), поэтому их держат наготове. Обязанности пула:
    - инициализация соединений при старте приложения;
    - выдача соединения по запросу и возврат обратно. Важный нюанс: `connection.close()` **возвращает соединение в пул**, а не закрывает его физически;
    - ограничение максимального числа соединений;
    - валидация (периодическая проверка живости соединения запросом вроде `SELECT 1`);
    - thread-safety (безопасность при работе из нескольких потоков);
    - автоматический реконнект при обрыве.

    Пул реализует паттерн **Object Pool** (пул объектов). По умолчанию в Spring Boot используется **HikariCP** — он даёт минимальную задержку.
- **`JdbcTemplate`** — обёртка над JDBC, реализующая паттерн **Template Method** (шаблонный метод: каркас работы фиксирован, а ты подставляешь только меняющиеся куски). Он сам открывает и закрывает ресурсы, переводит проверяемое `SQLException` в `DataAccessException` (unchecked-исключение, которое не обязательно ловить), и маппит результат через **`RowMapper`** (callback, описывающий, как из строки `ResultSet` собрать объект). Есть вариант `NamedParameterJdbcTemplate` с именованными параметрами вида `:name` вместо `?`.

**Разбор кода.**

```java
@Repository
@RequiredArgsConstructor
public class UserRepository {
    private final JdbcTemplate jdbc;

    private final RowMapper<User> mapper = (rs, n) -> new User(
        rs.getLong("id"), rs.getString("username"), rs.getString("email"));

    public User findById(Long id) {                         // queryForObject — 1 строка
        return jdbc.queryForObject("SELECT * FROM users WHERE id = ?", mapper, id);
    }
    public List<User> findAll() {                           // query — список
        return jdbc.query("SELECT * FROM users", mapper);
    }
    public int save(String username, String email) {       // update — DML
        return jdbc.update("INSERT INTO users(username,email) VALUES (?,?)", username, email);
    }
}
```

**Что здесь происходит:**
- `@Repository` — наклейка, помечающая класс как компонент доступа к данным (Spring им управляет и переводит исключения БД).
- `@RequiredArgsConstructor` — Lombok-аннотация, которая сама создаёт конструктор для всех `final`-полей; через него Spring внедряет `JdbcTemplate`.
- `RowMapper<User> mapper` — описывает, как из строки результата (`rs`) собрать объект `User`: берём колонки `id`, `username`, `email`.
- `findById` использует `queryForObject` — для запроса, который возвращает **ровно одну строку**.
- `findAll` использует `query` — возвращает **список** объектов.
- `save` использует `update` — для операций изменения данных (DML: INSERT/UPDATE/DELETE), возвращает число затронутых строк.

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 10
      minimum-idle: 5
      connection-timeout: 20000
      max-lifetime: 600000
```

**Что здесь происходит:**
- `maximum-pool-size: 10` — максимум 10 соединений в пуле одновременно.
- `minimum-idle: 5` — минимум 5 «дежурных» соединений держим открытыми всегда.
- `connection-timeout: 20000` — сколько миллисекунд ждать свободного соединения, прежде чем сдаться (20 секунд).
- `max-lifetime: 600000` — максимальное время жизни соединения (10 минут), после чего оно пересоздаётся.

**⚠️ Подводные камни.**
- `connection.close()` в коде с пулом **не закрывает** соединение физически, а возвращает его в пул для переиспользования. Это не баг, а суть пула — но об этом надо помнить.
- Размер пула надо настраивать осознанно: слишком маленький — запросы ждут в очереди, слишком большой — лишняя нагрузка на базу.

**📌 Запомнить.** `DataSource` поверх пула (по умолчанию HikariCP) экономит на дорогом создании соединений. `JdbcTemplate` убирает весь boilerplate: сам открывает/закрывает ресурсы и маппит результат через `RowMapper`.

## 3. Транзакции в Spring: декларативные и императивные, изоляция, propagation

**Простыми словами.** Транзакция — это «всё или ничего»: перевод денег с одного счёта на другой должен либо пройти целиком, либо не пройти вовсе (иначе деньги «испарятся»). В Spring есть ленивый способ — повесить наклейку `@Transactional` на метод, и фреймворк сам откроет транзакцию, зафиксирует или откатит. А есть способ «руками», когда нужен тонкий контроль. Плюс две тонкие настройки: **уровень изоляции** (насколько транзакции «видят» чужие незавершённые правки) и **propagation** (что делать, если одна транзакция вызвана внутри другой).

**Разбираемся.**

**Декларативные транзакции** — через наклейку `@Transactional`. Работают через **AOP-прокси** (Spring оборачивает твой объект в обёртку-посредника, которая до метода открывает транзакцию, а после — фиксирует). Поведение по умолчанию:
- открыть транзакцию перед методом;
- сделать `commit` при успешном завершении;
- сделать **`rollback` по умолчанию только при `RuntimeException` или `Error`** (непроверяемых исключениях). Для проверяемых (checked) исключений откат нужно включать явно: `rollbackFor = Exception.class`.

Это самый частый способ.

**Императивные транзакции** — полный контроль вручную:
- `TransactionTemplate` (работа через callback);
- или совсем вручную через `PlatformTransactionManager` (`getTransaction` / `commit` / `rollback`).

Нужны, когда требуется тонко управлять границами транзакции.

**Уровни изоляции** — это степень защиты от «аномалий» (когда параллельные транзакции мешают друг другу). Чем выше уровень, тем безопаснее, но медленнее:

|Уровень|Dirty read|Non-repeatable read|Phantom read|
|---|:-:|:-:|:-:|
|READ_UNCOMMITTED|❌ возможен|❌|❌|
|READ_COMMITTED (default в PostgreSQL)|✅ нет|❌|❌|
|REPEATABLE_READ|✅|✅ нет|❌|
|SERIALIZABLE|✅|✅|✅ нет (медленно)|

Расшифровка аномалий, с которыми борется изоляция:
- **Dirty read (грязное чтение)** — ты увидел чужие изменения, которые ещё не зафиксированы (и могут откатиться).
- **Non-repeatable read (неповторяемое чтение)** — прочитал строку дважды и получил разные значения, потому что между чтениями кто-то её поменял и закоммитил.
- **Phantom read (фантомное чтение)** — повторил один и тот же выборочный запрос и получил новые строки, которые кто-то добавил.

**Propagation (распространение)** — поведение при вложенных транзакциях (когда транзакционный метод вызывает другой транзакционный метод):
- `REQUIRED` (по умолчанию) — присоединиться к уже существующей транзакции или создать новую, если её нет.
- `REQUIRES_NEW` — всегда создавать новую транзакцию (текущая при этом приостанавливается). Полезно для аудита/логов, которые должны сохраниться, даже если внешняя транзакция откатится.
- `NESTED` — вложенная транзакция через savepoint (точку сохранения): откат вложенной не рушит внешнюю.
- Остальные: `SUPPORTS`, `MANDATORY`, `NEVER`, `NOT_SUPPORTED`.

**Разбор кода.**

```java
@Service
@RequiredArgsConstructor
public class OrderService {
    private final OrderRepository repo;
    private final AuditService audit;

    @Transactional(isolation = Isolation.READ_COMMITTED,
                   propagation = Propagation.REQUIRED,
                   rollbackFor = Exception.class)         // откатывать и на checked
    public void placeOrder(Order o) throws BusinessException {
        repo.save(o);
        audit.log("order " + o.getId());  // вызов ДРУГОГО бина — прокси сработает
        if (o.getAmount() < 0) throw new BusinessException("bad amount"); // откат всего
    }
}

@Service
class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW) // отдельная транзакция
    public void log(String msg) { /* сохранится даже при откате внешней */ }
}
```

**Что здесь происходит:**
- `@Transactional(...)` на `placeOrder` задаёт: уровень изоляции `READ_COMMITTED`, propagation `REQUIRED`, и `rollbackFor = Exception.class` — чтобы откатывалось даже на проверяемых исключениях (по умолчанию бы не откатилось).
- `audit.log(...)` — это вызов метода **другого бина** (`AuditService`), а не `this.log(...)`. Поэтому вызов проходит через прокси и транзакционность сработает.
- `throw new BusinessException(...)` — благодаря `rollbackFor` это исключение откатит всю транзакцию (включая `repo.save`).
- В `AuditService` метод `log` помечен `REQUIRES_NEW` — он выполняется в **отдельной** транзакции, поэтому запись аудита сохранится, даже если внешняя транзакция заказа откатится.

**⚠️ Подводные камни.**
- 🎯 **Главные грабли `@Transactional`** (помни, что он работает через прокси-обёртку):
    - **не действует на `private`-методах** — прокси не может их перехватить;
    - **не работает при внутреннем вызове `this.method()`** — такой вызов идёт напрямую, минуя прокси, и транзакция не открывается. Именно поэтому в примере вызов `audit.log` (другой бин) сработает, а вот если бы метод вызывал свой же транзакционный метод через `this`, транзакция бы не применилась;
    - по умолчанию **не откатывается на checked-исключениях** — нужен `rollbackFor`.

**📌 Запомнить.** `@Transactional` — самый частый способ, но работает через прокси: не на `private` и не при `this.method()`, и по умолчанию откатывается только на `RuntimeException`. Уровень изоляции в PostgreSQL по умолчанию — `READ_COMMITTED`. `REQUIRES_NEW` — для записей, которые должны выжить при откате внешней транзакции.

## 4. Миграции базы данных: Flyway (и Liquibase)

**Простыми словами.** Структура базы (таблицы, колонки) со временем меняется. **Миграции** — это как система контроля версий, но для самой базы: набор пронумерованных скриптов, которые описывают, как из «пустой базы» шаг за шагом получить нужную структуру. Благодаря этому база на твоём ноутбуке, на тестовом сервере и в продакшене будет одинаковой, а историю изменений всегда можно воспроизвести.

**Разбираемся.**

**Миграции** — версионирование схемы БД прямо в коде (в репозитории), чтобы dev/test/prod-окружения имели идентичную структуру и история изменений была воспроизводимой.

**Flyway** — популярный инструмент миграций:
- версионные SQL-скрипты с именами вида `V{версия}__{описание}.sql` (например, `V1__create_users.sql`) лежат в папке `db/migration`;
- применяются по порядку при старте приложения;
- пройденные миграции записываются в служебную таблицу `flyway_schema_history`. Flyway хранит контрольную сумму каждого применённого скрипта — поэтому **нельзя менять уже применённый скрипт** (контрольная сумма не сойдётся, и Flyway ругнётся);
- есть **repeatable-миграции** с именами `R__` — они выполняются заново при каждом изменении (например, для пересоздания вьюх).

**Liquibase** — альтернатива Flyway: изменения описываются **changeset'ами** в форматах XML/YAML/JSON/SQL, абстрагированы от диалекта конкретной СУБД, и есть **rollback (откат) из коробки**.

**Разбор кода.**

```sql
-- src/main/resources/db/migration/V1__create_users.sql
CREATE TABLE users (
    id         BIGSERIAL PRIMARY KEY,
    username   VARCHAR(100) NOT NULL,
    email      VARCHAR(100) NOT NULL UNIQUE,
    created_at TIMESTAMP NOT NULL DEFAULT now()
);
-- V2__add_age_column.sql
ALTER TABLE users ADD COLUMN age INT;
```

**Что здесь происходит:**
- Первый скрипт `V1__create_users.sql` создаёт таблицу `users` с первичным ключом-автоинкрементом (`BIGSERIAL PRIMARY KEY`), обязательными полями и уникальным email.
- Второй скрипт `V2__add_age_column.sql` — это следующая версия, добавляющая колонку `age`. Flyway применит их строго по порядку: сначала V1, потом V2.

```yaml
spring:
  flyway:
    enabled: true
    locations: classpath:db/migration
  jpa:
    hibernate:
      ddl-auto: validate     # Hibernate только проверяет, миграции делает Flyway
```

**Что здесь происходит:**
- `flyway.enabled: true` — включаем Flyway.
- `locations: classpath:db/migration` — где искать скрипты миграций.
- `ddl-auto: validate` — Hibernate только **проверяет**, что схема в базе соответствует сущностям, но сам структуру **не меняет**. За изменение структуры отвечает только Flyway.

**⚠️ Подводные камни.**
- 🎯 Когда используешь миграции, ставь Hibernate `ddl-auto` в **`validate`** (только проверять соответствие схемы сущностям, **не менять** структуру). Значения `ddl-auto: create` или `update` в продакшене — это **антипаттерн**: Hibernate начнёт сам менять структуру базы, и ты потеряешь контроль и историю изменений (а `create` вообще может снести данные).
- Нельзя править уже применённую миграцию — Flyway сверяет контрольные суммы. Новое изменение = новый скрипт с новой версией.

**📌 Запомнить.** Миграции = версионирование схемы базы в репозитории. Flyway: скрипты `V1__...sql`, применяются по порядку, фиксируются в `flyway_schema_history`, нельзя менять задним числом. С миграциями обязательно `ddl-auto: validate`.

## 5. JPA и Hibernate: как объекты превращаются в строки таблиц

**Простыми словами.** Писать SQL руками (как в JDBC) — это как переводить каждое предложение со своего языка на язык базы вручную. **ORM** — это автоматический переводчик: ты работаешь с обычными Java-объектами, а он сам превращает их в SQL и обратно. **JPA** — это правила такого перевода (стандарт), а **Hibernate** — самый популярный «переводчик», который этим правилам следует.

**Разбираемся.**

**JPA (Java Persistence API)** — это спецификация ORM, то есть набор правил, как маппить (сопоставлять) объекты на таблицы. **ORM (Object-Relational Mapping)** — отображение объектов на реляционные таблицы. **Hibernate** — самая популярная реализация JPA.

**Принцип работы Hibernate:**
- держит **Persistence Context (контекст персистентности)** — это кеш первого уровня, карта управляемых сущностей внутри текущей сессии;
- отслеживает изменения через **dirty checking (проверку на «грязь»)**: запоминает, как объект выглядел при загрузке, и при `flush`/`commit` сам сравнивает и генерирует нужный SQL для изменившихся полей — тебе не нужно явно звать `save`;
- поддерживает **lazy-загрузку** связей через прокси (связанные данные подгружаются только при первом обращении);
- абстрагирует диалект конкретной СУБД (один и тот же код работает с разными базами).

**Аннотации маппинга** (наклейки, которые объясняют Hibernate, как класс связан с таблицей):
- `@Entity` — помечает класс как **сущность** (то, что хранится в базе); `@Table(name)` — задаёт имя таблицы.
- `@Id` — первичный ключ; `@GeneratedValue(strategy)` — как генерировать значение ключа:
    - `IDENTITY` — автоинкремент на стороне БД (как PostgreSQL serial);
    - `SEQUENCE` — через последовательность БД (позволяет батчить, то есть группировать, INSERT-ы);
    - `TABLE` — через отдельную таблицу-счётчик (медленно);
    - `AUTO` — Hibernate выбирает стратегию сам.
- `@Column(name, nullable, length, unique, updatable)` — настройки колонки (имя, можно ли NULL, длина, уникальность, можно ли менять).
- Нужен **конструктор без аргументов** (`protected` или `public`) — Hibernate создаёт объект через **рефлексию** (механизм Java для создания объектов и доступа к полям «на лету»). `private`-конструктор не подойдёт для проксирования и наследования.

**Spring Data JPA-репозитории** — ты объявляешь интерфейс, который наследует `JpaRepository<Entity, ID>`, а реализацию Spring генерирует автоматически: базовые CRUD-операции плюс методы по имени.

**Разбор кода.**

```java
@Entity
@Table(name = "users")
public class User {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "username", nullable = false, length = 100)
    private String username;

    @Column(name = "email", nullable = false, length = 100, unique = true)
    private String email;

    @Column(name = "created_at", updatable = false)
    private Instant createdAt;

    @PrePersist void onCreate() { createdAt = Instant.now(); } // callback перед INSERT

    protected User() {}   // обязателен для Hibernate
    // конструктор, геттеры/сеттеры
}

public interface UserRepository extends JpaRepository<User, Long> {
    Optional<User> findByEmail(String email);   // реализация генерируется
    boolean existsByEmail(String email);
}
```

**Что здесь происходит:**
- `@Entity` + `@Table(name = "users")` — класс `User` — это сущность, хранится в таблице `users`.
- `@Id @GeneratedValue(strategy = GenerationType.IDENTITY)` — поле `id` — первичный ключ, генерируется автоинкрементом базы.
- `@Column(...)` на полях — задаёт ограничения колонок: `username` не может быть NULL и не длиннее 100; `email` ещё и уникален; `created_at` нельзя обновлять (`updatable = false`).
- `@PrePersist void onCreate()` — **callback** (метод, который Hibernate вызывает сам) перед первым сохранением (INSERT). Здесь он проставляет дату создания.
- `protected User() {}` — пустой конструктор, обязательный для Hibernate.
- `UserRepository extends JpaRepository<User, Long>` — репозиторий для сущности `User` с ключом типа `Long`. Методы `findByEmail` и `existsByEmail` Spring реализует сам, разобрав их имена.

**⚠️ Подводные камни.**
- 🎯 **Почему конструктор `protected`, а не `private`?** Hibernate создаёт прокси-подклассы (для lazy-загрузки) — а подклассу нужен доступный (не приватный) конструктор родителя. С `private` проксирование и наследование сломаются.
- 🎯 **Что будет без `@Entity`?** Класс не будет управляться Hibernate — он просто не сущность, и сохранить его в базу через JPA не выйдет.

**📌 Запомнить.** JPA — спецификация (правила), Hibernate — реализация. Ключевые наклейки: `@Entity`, `@Id`, `@GeneratedValue`, `@Column`. Hibernate сам отслеживает изменения (dirty checking) и генерирует SQL. Нужен непустой доступный конструктор без аргументов.

## 6. Hibernate: жизненный цикл сущности

**Простыми словами.** Объект-сущность в Hibernate проходит как бы через несколько «гражданских состояний» по отношению к базе. Сначала он просто новый объект, о котором база ничего не знает. Потом он «прописывается» под надзор Hibernate, и тот следит за каждым его изменением. Потом надзор снимается (сессия закрылась). А может и вовсе быть помечен на удаление. От состояния зависит, попадут ли твои изменения в базу автоматически.

**Разбираемся.**

Состояния сущности определяются относительно **Persistence Context** (контекста персистентности):

- **Transient (новый)** — только что созданный объект (`new User()`). Hibernate о нём ничего не знает, у объекта ещё нет id, его нет в кеше.
- **Managed / Persistent (управляемый)** — привязан к открытой сессии. Hibernate его **отслеживает**: любые изменения автоматически синхронизируются в базу при flush/commit благодаря **dirty checking** — явный `save` не нужен. Объект становится managed после `persist()`/`save()` или после загрузки через `find()`/`findById()`.
- **Detached (отсоединённый)** — был managed, но сессия закрылась (транзакция завершилась). Изменения такого объекта **уже не отслеживаются**. Чтобы вернуть его в managed-состояние, используют `merge()` — он копирует состояние detached-объекта в управляемую сущность. ⚠️ Обращение к lazy-связи у detached-объекта вызовет `LazyInitializationException` (сессии-то уже нет, подгрузить нечем).
- **Removed (удаляемый)** — помечен на удаление (`remove()`), физически удалится из базы при flush.

**Разбор кода.**

```java
@Transactional
public void demo() {
    User u = new User("a@x.com");      // TRANSIENT
    em.persist(u);                     // -> MANAGED (отслеживается)
    u.setEmail("new@x.com");           // dirty checking: UPDATE полетит сам при commit
    User found = em.find(User.class, u.getId()); // MANAGED (из L1-кеша, без запроса)
    em.remove(found);                  // -> REMOVED
}                                      // commit: flush + close -> остальные становятся DETACHED

// detached + merge в новой транзакции
@Transactional
public void update(User detached) {
    User managed = em.merge(detached); // detached -> managed (по содержимому)
    managed.setEmail("x@x.com");       // теперь отслеживается
}
```

**Что здесь происходит:**
- `new User("a@x.com")` — объект в состоянии **TRANSIENT**, Hibernate о нём не знает.
- `em.persist(u)` — переводит его в **MANAGED**, теперь он под надзором.
- `u.setEmail("new@x.com")` — мы просто меняем поле; благодаря dirty checking при commit Hibernate сам отправит UPDATE, без явного вызова `save`.
- `em.find(User.class, u.getId())` — возвращает тот же объект из кеша первого уровня (L1) **без запроса в базу**, потому что он уже managed.
- `em.remove(found)` — переводит в **REMOVED**, при flush будет DELETE.
- В конце метода commit делает flush (отправляет накопленные изменения) и закрывает сессию — после чего оставшиеся объекты становятся **DETACHED**.
- Во второй транзакции `em.merge(detached)` берёт отсоединённый объект и возвращает новый **managed**-объект с тем же содержимым; дальнейшие изменения `managed` снова отслеживаются.

**⚠️ Подводные камни.**
- Обращение к lazy-связи у **detached**-объекта (вне открытой сессии) выбросит `LazyInitializationException` — частая ошибка, когда пытаются дочитать связь уже после закрытия транзакции.
- `merge()` не делает detached-объект managed — он **возвращает новый** managed-объект. Менять дальше нужно именно возвращённый объект, а не исходный detached.

**📌 Запомнить.** Четыре состояния: Transient (новый, неизвестен) → Managed (под надзором, dirty checking, save не нужен) → Detached (сессия закрыта, не отслеживается, вернуть через merge) → Removed (на удаление). У managed-сущности изменения летят в базу сами.

## 7. Hibernate: связи между сущностями и каскадные операции

**Простыми словами.** В реальной жизни данные связаны: у поста есть комментарии, у заказа — позиции. Hibernate умеет описывать эти связи наклейками. А **каскад** — это правило «делай со мной — делай и с моими детьми»: удаляешь пост — удаляются и его комментарии. Главное — аккуратно держать связь синхронной с обеих сторон, иначе база не поймёт, кто к кому относится.

**Разбираемся.**

**Типы связей:**
- `@OneToOne` — один-к-одному;
- `@ManyToOne` / `@OneToMany` — многие-к-одному и один-ко-многим (часто описывают одну и ту же связь с двух сторон — двунаправленно);
- `@ManyToMany` — многие-ко-многим (через промежуточную join-таблицу).

**Владелец связи** — это сторона, на которой лежит внешний ключ (помечается `@JoinColumn`). Обратная (не владеющая) сторона указывает `mappedBy` — ссылку на поле-владелец.

**Fetch (как грузить связи):**
- Коллекции (`@OneToMany`, `@ManyToMany`) по умолчанию **LAZY** (ленивые) — грузятся только при обращении.
- `@ManyToOne` и `@OneToOne` по умолчанию **EAGER** (жадные) — грузятся сразу. Часто стоит явно переопределить на `LAZY`, чтобы не тянуть лишние данные.

**Каскады (`cascade`)** — операции с родителем распространяются на детей. Типы: `PERSIST`, `MERGE`, `REMOVE`, `REFRESH`, `DETACH`, `ALL` (всё сразу). Отдельно есть **`orphanRemoval=true`** — удалять «осиротевших» детей, отвязанных от родителя (убрал ребёнка из коллекции → он удаляется из базы, DELETE). Отличие от `CascadeType.REMOVE`: `orphanRemoval` срабатывает и при простом отвязывании ребёнка, а не только при удалении самого родителя.

**Разбор кода.**

```java
@Entity
public class Post {
    @Id @GeneratedValue Long id;

    @OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<Comment> comments = new ArrayList<>();   // mappedBy: владелец — Comment

    public void addComment(Comment c) {                    // синхронизация обеих сторон
        comments.add(c); c.setPost(this);
    }
}

@Entity
public class Comment {
    @Id @GeneratedValue Long id;
    @ManyToOne(fetch = FetchType.LAZY)                     // не грузить пост заранее
    @JoinColumn(name = "post_id")                          // здесь FK -> владелец связи
    private Post post;
}
```

**Что здесь происходит:**
- В `Post` коллекция `comments` помечена `@OneToMany(mappedBy = "post", ...)` — то есть владелец связи находится на стороне `Comment` (в его поле `post`).
- `cascade = CascadeType.ALL` — все операции (сохранение, удаление и т. д.) с постом распространяются на комментарии.
- `orphanRemoval = true` — если убрать комментарий из коллекции, он удалится из базы.
- `addComment(...)` — **helper-метод**, который синхронизирует **обе стороны** связи: добавляет коммент в список (`comments.add(c)`) и проставляет ссылку обратно (`c.setPost(this)`). Без этого внешний ключ не проставится.
- В `Comment` поле `post` помечено `@ManyToOne(fetch = FetchType.LAZY)` — пост не подгружается заранее, только при обращении.
- `@JoinColumn(name = "post_id")` — здесь живёт внешний ключ (FK), значит `Comment` — владелец связи.

**⚠️ Подводные камни.**
- 🎯 **Гигиена двунаправленных связей:** всегда синхронизируй обе стороны через helper-метод (как `addComment`), иначе внешний ключ (FK) не проставится и связь окажется «однобокой».
- 🎯 `@ManyToMany` в продакшене часто заменяют двумя связями `@OneToMany`/`@ManyToOne` к промежуточной join-сущности — чтобы можно было хранить дополнительные поля самой связи (например, дату или роль).
- `@ManyToOne` и `@OneToOne` по умолчанию EAGER — легко случайно тянуть кучу лишних данных; обычно лучше явно ставить `LAZY`.

**📌 Запомнить.** Владелец связи — сторона с `@JoinColumn` (внешним ключом); обратная сторона ставит `mappedBy`. Коллекции по умолчанию LAZY, `@ManyToOne`/`@OneToOne` — EAGER. Каскады распространяют операции на детей; `orphanRemoval` удаляет отвязанных детей. Всегда синхронизируй обе стороны двунаправленной связи.

📖 _`Seminar_7.md` (@ManyToOne/@OneToMany, каскады), `Longread_8.md`._

## 8. Hibernate: производительность — кеши L1/L2 и проблема N+1

**Простыми словами.** Каждый поход в базу стоит времени. Чтобы не бегать туда лишний раз, Hibernate запоминает уже загруженное в **кешах**. Кеш первого уровня — личный блокнот на время одной транзакции (всегда включён). Кеш второго уровня — общая доска для всех (включается отдельно). А самая известная беда производительности — **проблема N+1**, когда вместо одного запроса Hibernate незаметно делает целую лавину.

**Разбираемся.**

**Кеш L1 (первого уровня)** — живёт на уровне Persistence Context / сессии, **включён всегда**. Повторный `findById` в рамках одной транзакции не пойдёт в базу — объект возьмётся из кеша. Очищается при завершении транзакции.

**Кеш L2 (второго уровня)** — общий на уровне `EntityManagerFactory`, разделяется между транзакциями и потоками. **Выключен по умолчанию**, требует подключения провайдера (Ehcache, Hazelcast, Infinispan — через стандарт JCache/JSR-107). Подходит для справочников (данные, которые часто читают и редко меняют). Хранит данные в десериализованном виде.

**Query Cache (кеш запросов)** — кеширует **список id** результата запроса (по принципу `{запрос + параметры} → [id...]`), а сами сущности берутся уже из L2 или из базы. Важная тонкость: он **инвалидируется (сбрасывается) при любом изменении таблицы** запроса — даже если изменённая строка не подходила под условие WHERE. Так происходит, потому что Hibernate не может гарантировать корректность результата иначе. Поэтому Query Cache годится только для статичных справочников и отчётов.

**Проблема N+1** — классический антипаттерн. При загрузке N сущностей с LAZY-связью Hibernate делает 1 запрос на сам список и потом по 1 запросу на связь каждого элемента — итого **N+1 запросов** вместо одного-двух. Решения:
- **`JOIN FETCH`** в JPQL — подтянуть связь одним запросом;
- **`@EntityGraph`** на методе репозитория — декларативно указать, что подгрузить;
- **batch fetching** (`@BatchSize(size=N)` или `hibernate.default_batch_fetch_size`) — группирует догрузку связей в один запрос с `IN (...)`.

**Разбор кода.**

```java
// ПЛОХО — N+1:
List<Post> posts = repo.findAll();              // 1 запрос (посты)
posts.forEach(p -> p.getComments().size());     // + N запросов (комменты на каждый пост)

// РЕШЕНИЕ 1 — JOIN FETCH (один запрос)
public interface PostRepository extends JpaRepository<Post, Long> {
    @Query("SELECT DISTINCT p FROM Post p JOIN FETCH p.comments")
    List<Post> findAllWithComments();

    // РЕШЕНИЕ 2 — @EntityGraph
    @EntityGraph(attributePaths = "comments")
    List<Post> findAll();
}
```

**Что здесь происходит:**
- Первый блок — иллюстрация проблемы: `findAll()` делает 1 запрос за постами, а затем обращение к `getComments()` у каждого поста порождает ещё по запросу — вот и N+1.
- `@Query("SELECT DISTINCT p FROM Post p JOIN FETCH p.comments")` — решение через JOIN FETCH: посты и комментарии тянутся одним запросом. `DISTINCT` убирает дубликаты постов, возникающие из-за соединения.
- `@EntityGraph(attributePaths = "comments")` — решение декларативно: говорим Spring Data сразу подгрузить связь `comments`, без ручного JPQL.

```yaml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 50   # batch-догрузка lazy через IN(...)
        cache: { use_query_cache: true }
```

**Что здесь происходит:**
- `default_batch_fetch_size: 50` — включает batch fetching: вместо N отдельных запросов Hibernate догрузит до 50 связей одним запросом через `IN (...)`.
- `cache: { use_query_cache: true }` — включает кеш запросов.

**⚠️ Подводные камни.**
- 🎯 **Как обнаружить N+1?** Включи `spring.jpa.show-sql=true` или `org.hibernate.SQL=DEBUG` — и в логах будет видна лавина SELECT-ов. Это первый признак проблемы.
- Query Cache инвалидируется при любом изменении таблицы (даже не подходящей под WHERE), поэтому на изменчивых данных он почти бесполезен — только статичные справочники/отчёты.

**📌 Запомнить.** L1-кеш — всегда включён, на одну транзакцию. L2-кеш — выключен по умолчанию, общий, для редко меняющихся справочников. N+1 = 1 запрос на список + N на связи; лечится `JOIN FETCH`, `@EntityGraph` или batch fetching. Ищи N+1 по логам SQL.

## 9. Hibernate: пагинация и сортировка

**Простыми словами.** Если в базе миллион пользователей, выгружать их всех разом — безумие. **Пагинация** — это выдача данных страницами (как книга: 20 записей на страницу). **Сортировка** — упорядочивание (по имени, по дате). Spring Data умеет это «из коробки»: передаёшь, какую страницу и какого размера хочешь, а он сам добавит в SQL нужные `LIMIT` и `OFFSET`.

**Разбираемся.**

Spring Data предоставляет объекты `Pageable` (запрос страницы) и `Sort` (сортировка) из коробки. На уровне SQL это превращается в `LIMIT` (сколько брать) и `OFFSET` (сколько пропустить).

- **`Page<T>`** — данные плюс метаданные: всего элементов, всего страниц, есть ли следующая. Чтобы посчитать общее число, делает **дополнительный `count`-запрос**.
- **`Slice<T>`** — то же, но **без `count`-запроса** (легче, когда общее число элементов не нужно — например, для бесконечной прокрутки).
- В контроллере Spring сам соберёт объект `Pageable` из query-параметров запроса вида `?page=0&size=20&sort=username,asc`.

**Разбор кода.**

```java
public interface UserRepository extends JpaRepository<User, Long> {
    Page<User> findByAgeGreaterThan(int age, Pageable pageable);
}

// сервис
Pageable pageable = PageRequest.of(0, 20, Sort.by("username").ascending()
                                            .and(Sort.by("id").descending()));
Page<User> page = repo.findByAgeGreaterThan(18, pageable);
page.getContent();        // элементы
page.getTotalElements();  // всего (стоит count-запрос)
page.hasNext();

// контроллер — Pageable собирается из query-параметров автоматически
@GetMapping("/users")
public Page<User> list(@PageableDefault(size = 20) Pageable pageable) {
    return repo.findAll(pageable);
}
```

**Что здесь происходит:**
- `findByAgeGreaterThan(int age, Pageable pageable)` — метод репозитория принимает `Pageable`, возвращает `Page<User>` (страницу с метаданными).
- `PageRequest.of(0, 20, ...)` — запрос нулевой (первой) страницы по 20 элементов с сортировкой: сначала по `username` по возрастанию, затем по `id` по убыванию.
- `page.getContent()` — сами элементы страницы; `page.getTotalElements()` — общее число (тут как раз выполняется дополнительный count-запрос); `page.hasNext()` — есть ли следующая страница.
- В контроллере `@PageableDefault(size = 20) Pageable pageable` — Spring сам соберёт `Pageable` из query-параметров запроса (`?page=&size=&sort=`), задав размер по умолчанию 20.

**⚠️ Подводные камни.**
- 🎯 **OFFSET-пагинация деградирует на больших страницах**: чтобы отдать, скажем, 100000-ю страницу, база вынуждена пропустить (проскипать) offset строк — а это медленно. Для глубокой пагинации используют **keyset/seek-пагинацию**: вместо OFFSET ставят условие `WHERE id > :lastId ORDER BY id LIMIT n` (берём «следующие после последнего виденного»).
- 🎯 Если совместить `JOIN FETCH` коллекции с пагинацией, Hibernate может начать **пагинировать в памяти** (выдаст предупреждение `HHH000104`) — то есть загрузит всё и порежет страницы уже у себя, что убивает смысл пагинации. Решение — использовать `@EntityGraph` или batch fetch вместо `JOIN FETCH`.

**📌 Запомнить.** `Page<T>` = данные + count-запрос (знаешь общее число); `Slice<T>` = без count (легче). OFFSET-пагинация тормозит на глубоких страницах — для них keyset-пагинация. `JOIN FETCH` + пагинация = пагинация в памяти, избегай.

## 10. Spring Data Repository: свои методы, JPQL, HQL

**Простыми словами.** Базовые операции (найти по id, сохранить, удалить) Spring Data даёт даром. Но рано или поздно нужен свой запрос. Способов несколько: можно просто хитро назвать метод — и Spring сам догадается, что ты хочешь, по его имени. Можно написать запрос явно на «объектном» языке JPQL (он оперирует не таблицами, а классами и полями). А можно спуститься до чистого SQL, если нужна специфика конкретной базы.

**Разбираемся.**

Способы добавить методы в репозиторий:

1. **Derived query (запрос по имени метода)** — например, `findByEmailAndAgeGreaterThanOrderByUsername`. Spring парсит имя метода и строит запрос сам.
2. **`@Query` с JPQL** — **JPQL (Java Persistence Query Language)** — объектно-ориентированный язык запросов JPA. Он оперирует **сущностями и их полями** (`User`, `u.email`), а не таблицами и колонками. **HQL (Hibernate Query Language)** — диалект Hibernate, по сути расширенный JPQL. Поддерживает именованные параметры через `@Param`.
3. **Native query (нативный запрос)** — `@Query(value="...", nativeQuery=true)` — чистый SQL. Нужен, когда требуется специфика конкретной СУБД.
4. **`@Modifying`** — для запросов UPDATE/DELETE (требует транзакции; обычно ставят `clearAutomatically`, чтобы очистить кеш после изменения).
5. **Custom impl (своя реализация)** — интерфейс `XxxRepositoryCustom` плюс класс `XxxRepositoryImpl`, где через `EntityManager` или Criteria API строят сложные динамические запросы.

**Разбор кода.**

```java
public interface UserRepository extends JpaRepository<User, Long> {

    List<User> findByAgeGreaterThanOrderByUsername(int age);   // 1. derived

    @Query("SELECT u FROM User u WHERE u.email = :email")       // 2. JPQL (по сущностям)
    Optional<User> byEmail(@Param("email") String email);

    @Query(value = "SELECT * FROM users WHERE age > ?1", nativeQuery = true) // 3. native
    List<User> olderThanNative(int age);

    @Modifying                                                 // 4. UPDATE
    @Query("UPDATE User u SET u.age = u.age + 1 WHERE u.id = :id")
    int incAge(@Param("id") Long id);
}
```

**Что здесь происходит:**
- `findByAgeGreaterThanOrderByUsername(int age)` — derived query: Spring разберёт имя («age больше чем», «сортировать по username») и сам построит запрос.
- `@Query("SELECT u FROM User u WHERE u.email = :email")` — JPQL: обрати внимание, что мы пишем `User` (класс) и `u.email` (поле), а не `users` и колонку. `:email` — именованный параметр, связанный с аргументом через `@Param("email")`.
- `@Query(value = "SELECT * FROM users WHERE age > ?1", nativeQuery = true)` — нативный SQL: здесь уже настоящая таблица `users`; `?1` — позиционный параметр (первый аргумент).
- `@Modifying` + `@Query("UPDATE User u SET ...")` — изменяющий запрос (UPDATE). `@Modifying` обязателен для UPDATE/DELETE, и такой метод должен выполняться в транзакции.

**⚠️ Подводные камни.**
- JPQL оперирует именами классов и полей, а native query — именами таблиц и колонок. Их легко перепутать: `User` vs `users`, `u.email` vs колонка `email`.
- `@Modifying`-запросы требуют транзакции и обходят кеш персистентности — после них стоит включать `clearAutomatically`, чтобы в кеше не остались устаревшие данные.

**📌 Запомнить.** Пять способов: derived (по имени), JPQL/HQL (по сущностям и полям), native (чистый SQL), `@Modifying` (UPDATE/DELETE в транзакции), custom impl (для сложной динамики). JPQL — про объекты, native — про таблицы.

## 11. Spring Security: архитектура, AuthenticationManager, PasswordEncoder, JWT

**Простыми словами.** Spring Security — это охрана на входе в твоё приложение. Каждый запрос проходит через строй охранников (цепочку фильтров), и если хоть один не пропустил — запрос дальше не идёт. Сначала охрана выясняет «кто ты?» (**аутентификация** — проверка логина/пароля), потом «что тебе можно?» (**авторизация** — проверка прав). Пароли в базе хранят не как есть, а в виде нечитаемого «отпечатка» (хеша). А **JWT** — это как браслет на входе в клуб: один раз показал документы, получил подписанный браслет и дальше просто машешь им, не доставая паспорт каждый раз.

**Разбираемся.**

**Архитектура — это цепочка сервлет-фильтров.** Запрос проходит путь: `DelegatingFilterProxy` → `FilterChainProxy` → **`SecurityFilterChain`** (цепочка из 15–20 фильтров). Если хоть один фильтр отклонил запрос — запрос «мёртв», дальше не идёт. Конфигурация задаётся бином `SecurityFilterChain` (старый способ через наследование `WebSecurityConfigurerAdapter` устарел с версии 5.7).

**Аутентификация** («кто ты?» — проверка учётных данных) и **авторизация** («что тебе можно?» — проверка прав уже после аутентификации) — разные вещи.

Цепочка проверки логина/пароля:
- **`AuthenticationManager`** делегирует работу списку **`AuthenticationProvider`** (провайдеров аутентификации);
- фильтр создаёт объект `UsernamePasswordAuthenticationToken` (логин + пароль);
- менеджер находит подходящий провайдер;
- провайдер обращается к **`UserDetailsService`** (загрузка пользователя из БД) и сверяет пароль через **`PasswordEncoder`** (кодировщик паролей).

**PasswordEncoder** — про хеширование паролей. Хеш — это «односторонний отпечаток»: из пароля легко получить хеш, но из хеша пароль не восстановить. Best practice — `DelegatingPasswordEncoder` (умеет понимать старые хеши) плюс rehash on login (при логине перехешировать пароль, если его алгоритм устарел).

**JWT (JSON Web Token)** — способ stateless-аутентификации (без хранения сессии на сервере). Сервер выдаёт подписанный токен из трёх частей `header.payload.signature`, а клиент присылает его в заголовке `Authorization: Bearer <token>`. При работе с JWT:
- `csrf().disable()` — отключаем защиту от CSRF (раз нет кук, то и CSRF-атаки неоткуда взяться);
- `SessionCreationPolicy.STATELESS` — не создавать серверную сессию;
- кастомный фильтр (на базе `OncePerRequestFilter` — выполняется один раз на запрос) на каждом запросе валидирует токен и кладёт объект `Authentication` в `SecurityContextHolder` (хранилище текущего пользователя).

**Разбор кода.**

```java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean PasswordEncoder passwordEncoder() { return new BCryptPasswordEncoder(12); }

    @Bean
    SecurityFilterChain chain(HttpSecurity http, JwtAuthFilter jwt) throws Exception {
        return http
            .csrf(c -> c.disable())                          // stateless JWT
            .sessionManagement(s -> s.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            .authorizeHttpRequests(a -> a
                .requestMatchers("/auth/**").permitAll()     // частное — раньше
                .requestMatchers("/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated())               // общее — позже
            .addFilterBefore(jwt, UsernamePasswordAuthenticationFilter.class)
            .build();
    }
}

@Component
public class JwtAuthFilter extends OncePerRequestFilter {
    private final JwtService jwtService;
    public JwtAuthFilter(JwtService s) { this.jwtService = s; }

    @Override protected boolean shouldNotFilter(HttpServletRequest r) {
        return r.getRequestURI().startsWith("/auth/");
    }
    @Override protected void doFilterInternal(HttpServletRequest req,
            HttpServletResponse res, FilterChain chain) throws ServletException, IOException {
        String h = req.getHeader(HttpHeaders.AUTHORIZATION);
        if (h != null && h.startsWith("Bearer ")) {
            var jws = jwtService.parseAndValidate(h.substring(7));
            String user = jws.getPayload().getSubject();        // sub
            String roles = jws.getPayload().get("roles", String.class);
            var auth = new UsernamePasswordAuthenticationToken(user, null,
                roles == null ? List.of() : List.of(new SimpleGrantedAuthority("ROLE_" + roles)));
            SecurityContextHolder.getContext().setAuthentication(auth);
        }
        chain.doFilter(req, res);
    }
}
```

**Что здесь происходит:**
- `@Configuration` + `@EnableWebSecurity` — класс настройки безопасности.
- `passwordEncoder()` возвращает `BCryptPasswordEncoder(12)` — кодировщик BCrypt с параметром стойкости (cost) 12.
- В `chain(...)` собирается `SecurityFilterChain`:
    - `csrf(c -> c.disable())` — отключаем CSRF (stateless JWT, кук нет);
    - `sessionCreationPolicy(STATELESS)` — не создаём сессию;
    - `authorizeHttpRequests(...)` — правила доступа: `/auth/**` открыт всем (`permitAll`), `/admin/**` только для роли ADMIN, а всё остальное (`anyRequest`) требует аутентификации. Порядок важен — от частного к общему;
    - `addFilterBefore(jwt, UsernamePasswordAuthenticationFilter.class)` — встраиваем наш JWT-фильтр в цепочку перед стандартным фильтром логина.
- `JwtAuthFilter extends OncePerRequestFilter` — фильтр выполняется один раз на каждый запрос:
    - `shouldNotFilter` — пропустить фильтрацию для `/auth/` (там токена ещё нет);
    - `doFilterInternal` — читает заголовок `Authorization`, проверяет префикс `Bearer `, отрезает его (`substring(7)`), валидирует токен;
    - достаёт из токена пользователя (`getSubject()` — claim `sub`) и роли, собирает `Authentication` и кладёт его в `SecurityContextHolder`;
    - `chain.doFilter(req, res)` — передаёт запрос дальше по цепочке.

**⚠️ Подводные камни.**
- 🎯 **Только `BCryptPasswordEncoder` или `Argon2PasswordEncoder`** (с cost ≥ 10–12), **никогда** MD5 или хранение в открытом виде. Соль (случайная добавка к паролю) — только через `SecureRandom`. Слабые алгоритмы давно взламываются перебором — это прямой путь к утечке всех паролей.
- 🎯 **Порядок `requestMatchers` — строго от частного к общему:** сначала `permitAll()` для `/auth/**`, и только потом `anyRequest().authenticated()`. Если поставить общее правило раньше частного, оно «перехватит» все запросы, и частное правило никогда не сработает.

**📌 Запомнить.** Spring Security — цепочка фильтров (`SecurityFilterChain`); один отклонил — запрос мёртв. Аутентификация («кто ты») ≠ авторизация («что можно»). Пароли — только BCrypt/Argon2, никогда plain/MD5. JWT — stateless: `csrf().disable()` + `STATELESS` + кастомный фильтр. Правила доступа — от частного к общему.

## 12. RestClient: запросы, заголовки, авторизация, обработка ответов

**Простыми словами.** Часто приложению нужно само сходить в чужой сервис по HTTP (получить погоду, дёрнуть платёжный шлюз). **RestClient** — это удобный «телефон» для таких звонков: набираешь адрес, прикладываешь нужные заголовки (в том числе пропуск-токен для авторизации), отправляешь запрос и разбираешь ответ. Главное — правильно оформить заголовок авторизации и аккуратно обработать ошибки, ведь чужой сервис может ответить отказом или вообще не ответить.

**Разбираемся.**

**RestClient** — современный синхронный HTTP-клиент Spring (появился в версии 6.1). У него удобный fluent-API (цепочка вызовов через точку), и он пришёл на замену устаревшему `RestTemplate`. Реактивный (асинхронный) аналог — `WebClient`. Заголовки можно задавать глобально (`defaultHeader` в конфигурации) или локально под конкретный запрос (`header(...)`).

**Авторизация** — это, по сути, правильный заголовок `Authorization`:
- **Basic:** `Authorization: Basic base64(user:pass)`. Важно: Base64 — это **не шифрование**, а просто кодировка. Без HTTPS пароль фактически идёт в открытом виде.
- **Bearer (JWT/OAuth2):** `Authorization: Bearer <token>` — отправляем токен.

**Обработка ответов и ошибок.** Категории сбоев при походе в чужой сервис:
- сеть/инфраструктура (DNS не разрезолвился, TLS-ошибка, timeout);
- HTTP-ошибки 4xx (клиент виноват) / 5xx (сервер виноват);
- неверный формат тела ответа;
- «бизнес-ошибка со статусом 200» (формально OK, но в теле — отказ).

По умолчанию метод `retrieve()` на ответы 4xx/5xx **бросает исключение** (`HttpClientErrorException` / `RestClientResponseException`). Через `onStatus(...)` можно перехватить конкретный статус, разобрать тело ошибки (часто в формате RFC 7807 Problem Details) и бросить своё осмысленное исключение. Дополнительно: для списка используют `ParameterizedTypeReference`; для ответа 204 (нет тела) — `toBodilessEntity()`; чтобы достать заголовок `Location` при 201 — `toEntity().getHeaders().getLocation()`.

**Разбор кода.**

```java
@Configuration
class RestClientConfig {
    @Bean RestClient restClient(RestClient.Builder b) {
        return b.baseUrl("https://api.example.com")
                .defaultHeader("User-Agent", "demo/1.0")
                .build();
    }
}

@Service
public class TasksClient {
    private final RestClient rc;
    TasksClient(RestClient rc) { this.rc = rc; }

    // GET с query-параметрами + Bearer
    public List<TaskDto> list(boolean completed, int limit, String token) {
        return rc.get()
            .uri(u -> u.path("/v1/tasks")
                       .queryParam("completed", completed)
                       .queryParam("limit", limit).build())
            .header(HttpHeaders.AUTHORIZATION, "Bearer " + token)  // пробел!
            .accept(MediaType.APPLICATION_JSON)
            .retrieve()
            .body(new ParameterizedTypeReference<List<TaskDto>>() {}); // список
    }

    // POST + обработка ошибок
    public TaskDto create(TaskCreateRequest req) {
        return rc.post().uri("/v1/tasks")
            .contentType(MediaType.APPLICATION_JSON)
            .body(req)
            .retrieve()
            .onStatus(s -> s.value() == 409, (rq, rs) -> {        // распарсить тело ошибки
                throw new DuplicateException(new String(rs.getBody().readAllBytes()));
            })
            .body(TaskDto.class);
    }

    // 404 -> Optional.empty
    public Optional<TaskDto> find(long id) {
        try {
            return Optional.ofNullable(rc.get().uri("/v1/tasks/{id}", id)
                .retrieve().body(TaskDto.class));
        } catch (HttpClientErrorException.NotFound e) {
            return Optional.empty();
        }
    }
}
```

**Что здесь происходит:**
- В `RestClientConfig` создаётся бин `RestClient` с базовым URL и глобальным заголовком `User-Agent` (он будет у всех запросов).
- `list(...)` — GET-запрос: через `uri(u -> ...)` строится путь с query-параметрами `completed` и `limit`; локально добавляется заголовок `Authorization: Bearer <token>` (обрати внимание на пробел после `Bearer`); `.accept(...)` указывает, что ждём JSON; `.body(new ParameterizedTypeReference<List<TaskDto>>() {})` — разбираем ответ в список (для дженериков нужен именно `ParameterizedTypeReference`).
- `create(...)` — POST-запрос: задаём `contentType` и тело `req`; через `onStatus(s -> s.value() == 409, ...)` перехватываем статус 409 (конфликт), читаем тело ошибки и бросаем своё `DuplicateException`.
- `find(...)` — показывает, как превратить 404 в `Optional.empty()`: ловим `HttpClientErrorException.NotFound` и возвращаем пустой Optional вместо исключения.

**⚠️ Подводные камни.**
- 🎯 Самая частая ошибка при Bearer-авторизации — забыть **пробел** после слова `Bearer` (`"Bearer " + token`). Без пробела сервер не распознает токен.
- 🎯 **Чеклист «401 при, казалось бы, правильном токене»:** заголовок именно `Authorization`? есть пробел после `Bearer`? токен не просрочен? верный audience? тот ли тип схемы (`Bearer` против `Token`)? то ли окружение (staging и prod часто имеют разные ключи)? И логируй только маску токена (первые/последние 6 символов), а не токен целиком — иначе он утечёт в логи.
- Basic auth по Base64 — это не шифрование; без HTTPS пароль виден всем по дороге.

**📌 Запомнить.** RestClient — современный синхронный клиент (замена RestTemplate); `WebClient` — реактивный. Авторизация = заголовок `Authorization` (`Basic` или `Bearer <token>` — не забудь пробел!). `retrieve()` сам бросает исключение на 4xx/5xx; `onStatus(...)` — чтобы разобрать ошибку по-своему.

## 13. Rate Limiter и Circuit Breaker: отказоустойчивость с Resilience4j

**Простыми словами.** Когда твоё приложение зависит от чужих сервисов, надо исходить из того, что они могут тормозить или падать. **Circuit Breaker** («предохранитель») — как пробка в электрощитке: если внешний сервис «искрит» ошибками, предохранитель размыкается и временно перестаёт его дёргать, чтобы не делать хуже. **Rate Limiter** — вахтёр, который пускает не больше N запросов за период, чтобы тебя не забанили и один клиент не съел все ресурсы. Девиз всего этого: «проектируй так, будто всё вокруг уже сломано».

**Разбираемся.**

Паттерны отказоустойчивости при работе с внешними сервисами:

- **Circuit Breaker (предохранитель)** — считает долю сбоев и медленных вызовов на **sliding window (скользящем окне)** — оно бывает Count-based (по числу последних вызовов) или Time-based (по времени). Состояния:
    - `CLOSED` (замкнут, норма) — всё работает, вызовы проходят;
    - при превышении порога `failureRateThreshold` (доля ошибок) или `slowCallRateThreshold` (доля медленных) → переход в `OPEN` (разомкнут) — теперь вызовы сразу отдают fallback, не нагружая упавший сервис;
    - через время `waitDurationInOpenState` → `HALF_OPEN` (полуоткрыт) — пропускаем несколько пробных вызовов;
    - по их результату → обратно в `CLOSED` (если ожил) или снова в `OPEN`.
- **Rate Limiter (ограничитель частоты)** — ограничивает **частоту** запросов (алгоритмы Token Bucket / Fixed Window), чтобы не получить бан от провайдера и чтобы один клиент не съел все ресурсы. При превышении лимита бросается `RequestNotPermitted` → срабатывает fallback (можно вернуть HTTP-статус 429).
- **Bulkhead (переборка)** — ограничивает **параллелизм**, то есть число одновременных вызовов/потоков.
- Ещё два полезных паттерна: **Retry** (повторы при ошибке, лучше с backoff + jitter — нарастающей паузой со случайным разбросом) и **TimeLimiter** (таймаут на вызов).

**Разбор кода.**

```java
@Service
public class ExternalApiService {
    private final RestClient rc;
    ExternalApiService(RestClient rc) { this.rc = rc; }

    @CircuitBreaker(name = "backendA", fallbackMethod = "fallback")
    @RateLimiter(name = "backendA")
    public String call() {
        return rc.get().uri("/data").retrieve().body(String.class);
    }
    // fallback ловит и RequestNotPermitted (лимит), и CallNotPermitted (CB open), и ошибки
    private String fallback(Throwable t) {
        return "DEGRADED: " + t.getClass().getSimpleName(); // деградация, не падение
    }
}
```

**Что здесь происходит:**
- `@CircuitBreaker(name = "backendA", fallbackMethod = "fallback")` — оборачиваем метод предохранителем с именем настройки `backendA`; если он сработает (или будет ошибка), вызовется метод `fallback`.
- `@RateLimiter(name = "backendA")` — тот же метод ограничен по частоте.
- `call()` — собственно поход во внешний сервис через RestClient.
- `fallback(Throwable t)` — запасной вариант: ловит и `RequestNotPermitted` (превышен лимит), и `CallNotPermitted` (предохранитель разомкнут), и обычные ошибки. Возвращает «деградированный» ответ вместо падения — приложение продолжает работать.

```yaml
resilience4j:
  circuitbreaker:
    instances:
      backendA:
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        minimumNumberOfCalls: 5
        failureRateThreshold: 50          # >50% ошибок -> OPEN
        slowCallRateThreshold: 50
        slowCallDurationThreshold: 200ms
        waitDurationInOpenState: 5s
  ratelimiter:
    instances:
      backendA:
        limitForPeriod: 5
        limitRefreshPeriod: 10s
        timeoutDuration: 0
```

**Что здесь происходит:**
- Блок `circuitbreaker.instances.backendA`:
    - `slidingWindowType: COUNT_BASED` и `slidingWindowSize: 10` — считаем по последним 10 вызовам;
    - `minimumNumberOfCalls: 5` — статистику применяем только после минимум 5 вызовов (иначе рано судить);
    - `failureRateThreshold: 50` — если ошибок больше 50%, размыкаемся (OPEN);
    - `slowCallRateThreshold: 50` + `slowCallDurationThreshold: 200ms` — вызов дольше 200мс считается медленным; если таких больше 50% — тоже OPEN;
    - `waitDurationInOpenState: 5s` — через 5 секунд в OPEN пробуем перейти в HALF_OPEN.
- Блок `ratelimiter.instances.backendA`:
    - `limitForPeriod: 5` — не больше 5 запросов за период;
    - `limitRefreshPeriod: 10s` — период длиной 10 секунд;
    - `timeoutDuration: 0` — не ждать освобождения слота, сразу отказ при превышении.

**⚠️ Подводные камни.**
- 🎯 **Не путай Rate Limiter и Bulkhead:** лимит на частоту не спасёт от забивания пула потоков медленными вызовами — для этого нужен Bulkhead (ограничение параллелизма). Часто нужны оба одновременно.
- 🎯 **Строгий порядок выполнения декораторов:** Bulkhead → TimeLimiter → RateLimiter → CircuitBreaker → Retry. Если перепутать, можно получить, например, что Retry будет «долбить» сервис, который Circuit Breaker как раз пытается изолировать — и сделаешь только хуже.
- 🎯 Обязательно выводи метрики (например, `resilience4j_circuitbreaker_state`) в Prometheus/Grafana и пиши осмысленные fallback (вернуть кеш или дефолт), а не просто лог. По умолчанию `waitDurationInOpenState` = 60 секунд — настраивай под свой SLA.

**📌 Запомнить.** Circuit Breaker: CLOSED → OPEN (при превышении порога ошибок/медленных) → HALF_OPEN (пробные вызовы) → снова CLOSED/OPEN. Rate Limiter ограничивает частоту, Bulkhead — параллелизм (это разное!). Порядок декораторов: Bulkhead → TimeLimiter → RateLimiter → CircuitBreaker → Retry. Цель — деградация, а не падение.

## 1. Реактивное программирование: Project Reactor, Reactive Streams, тестирование

**Простыми словами.** Представь официанта в кафе. Обычный («блокирующий») официант, приняв заказ у столика, идёт на кухню и **стоит там**, ждёт, пока повар приготовит, и только потом обслуживает следующий столик. Реактивный официант так не делает: он отдал заказ повару и сразу пошёл к другим столикам, а когда блюдо готово — его «дёрнут», и он его отнесёт. Главный девиз реактивности: **«не стой и не жди — иди делай другое»**. Один официант так обслуживает целый зал.

**Разбираемся.**

**Reactive Streams** — это спецификация (договорённость, набор правил) для асинхронной обработки потоков данных с **backpressure**. **Backpressure (обратное давление)** — это механизм, при котором получатель сам говорит, сколько данных он готов принять, чтобы его не «завалили». Спецификация состоит из 4 интерфейсов (по сути — ролей в этом договоре):
- **`Publisher`** — издатель, тот, кто поставляет данные.
- **`Subscriber`** — подписчик, тот, кто данные потребляет.
- **`Subscription`** — «договор подписки» между ними, через него подписчик регулирует поток.
- **`Processor`** — гибрид, одновременно и издатель, и подписчик (звено посередине).

Ключевые свойства этого контракта (правила игры):
- **Ленивость** — пока никто не вызвал `subscribe()` (не «подписался»), **ничего не происходит**. Ты можешь описать весь конвейер обработки, но он не запустится, пока на него не подпишутся. Это как рецепт: написать его — не значит начать готовить.
- **Pull-push гибрид** — смешанная модель «тяни-толкай». `onNext` — это **push**: издатель «толкает» очередной элемент подписчику. `request(n)` — это **pull**: подписчик «тянет», то есть запрашивает себе n элементов. Главное правило: пока подписчик не вызвал `request(n)`, издатель **не имеет права** слать `onNext`. Вот это и есть backpressure в действии — получатель управляет краном.
- **Терминальность** — после сигнала `onError` (ошибка) или `onComplete` (завершение) **сигналов больше не будет**. Поток закончился — точка.
- **Non-blocking (неблокирующий)** — методы не должны «вставать и ждать», блокируя поток.

Зачем всё это: backpressure спасает от **OOM (OutOfMemoryError — нехватка памяти)**. Если быстрый издатель завалит медленного потребителя данными, они будут копиться в памяти, пока та не кончится. С backpressure скорость всего конвейера определяется его **слабым звеном**: медленный потребитель сам запрашивает ровно ту порцию, которую потянет.

**Project Reactor** — это эталонная реализация Reactive Streams для **JVM (Java Virtual Machine — виртуальная машина Java)** и ядро реактивного Spring. Она даёт два главных типа-контейнера для данных:
- **`Mono<T>`** — поток на **0 или 1** элемент. Подходит для: поиска по id, сохранения, одиночного HTTP-запроса.
- **`Flux<T>`** — поток на **0..N** элементов (сколько угодно). Подходит для: потоков данных, **SSE (Server-Sent Events — события, которые сервер шлёт клиенту)**, лент событий.
- **`Mono<Void>`** — поток, который не несёт данных вообще, только **сигнал завершения**. Подходит для операций вроде DELETE, где важен сам факт «готово».

Ещё одно важное различие — «температура» потоков:
- **Cold (холодный)** — данные генерируются **заново для каждого подписчика**. Подписался — получил свою свежую копию с начала. Примеры: `Mono.just`, `fromCallable`. Аналогия: заказал фильм по запросу — он начинается для тебя с начала.
- **Hot (горячий)** — это **общий поток** для всех; кто подключился — тот и слышит то, что идёт сейчас. Примеры: `Flux.interval`, `Sinks`, Kafka. Аналогия: прямой телеэфир — включил телевизор посреди передачи, начало уже пропустил.

И ещё тонкая, но частая на экзамене деталь:
- `Mono.fromCallable(...)` оборачивает блокирующий код **лениво** — то есть код выполнится только в момент подписки.
- `Mono.just(value)` вычисляет значение **немедленно**, прямо при создании (ещё до всякой подписки).

**Тестирование реактивного кода:**
- **`StepVerifier`** — инструмент, который детерминированно (предсказуемо, шаг за шагом) проверяет последовательность сигналов: какие элементы пришли, какие ошибки, было ли завершение и как сработал backpressure. Он сам играет роль подписчика. Важно: внутри тестов **не используют `.block()`** — блокировка ломает весь реактивный контракт и прячет реальные проблемы.
- **`BlockHound`** — «ищейка блокировок». Через **байткод-инструментацию** (то есть подмешивая проверки прямо в скомпилированный код во время работы) он ловит **скрытые блокировки** (`Thread.sleep`, обращения через JDBC, `synchronized`) в реактивных пулах потоков и бросает `BlockingOperationError`. Это как датчик дыма для случайно «заснувших» потоков.

**Разбор кода.**
```java
Flux<Integer> flux = Flux.range(1, 5)
    .map(i -> i * 2)
    .filter(i -> i > 4);           // ничего не выполняется...
flux.subscribe(System.out::println); // ...пока не подписались

@Test
void backpressure() {
    StepVerifier.create(Flux.just("A", "B", "C"), 1) // буфер подписчика 1
        .thenRequest(2)
        .expectNext("A", "B")
        .thenRequest(1)
        .expectNext("C")
        .verifyComplete();
}

static { BlockHound.install(); } // в @BeforeAll/тесте — ловит скрытые блокировки
```
**Что здесь происходит:**
- `Flux.range(1, 5)` создаёт поток чисел от 1 до 5. Дальше `.map(i -> i * 2)` удваивает каждое, а `.filter(i -> i > 4)` пропускает только числа больше 4. Но всё это пока лишь **описание конвейера** — комментарий честно говорит: «ничего не выполняется».
- `flux.subscribe(...)` — вот эта строчка запускает всё. Только после подписки числа реально потекут и распечатаются. Это и есть ленивость наглядно.
- В тесте `StepVerifier.create(Flux.just("A", "B", "C"), 1)` — второй аргумент `1` задаёт начальный буфер подписчика: «я готов принять пока всего 1 элемент».
- `.thenRequest(2)` — подписчик запрашивает ещё 2 элемента; `.expectNext("A", "B")` — проверяем, что пришли именно «A» и «B». Затем `.thenRequest(1)` запрашивает ещё один, и `.expectNext("C")` ждёт «C». Так пошагово моделируется backpressure.
- `.verifyComplete()` — проверяет, что поток корректно завершился сигналом `onComplete`.
- `static { BlockHound.install(); }` — статический блок инициализации; ставит «ищейку», которая будет ловить скрытые блокировки во время тестов.

**⚠️ Подводные камни.**
- **Ничего не сработает без подписки.** Самая частая ошибка новичка: построил красивую цепочку `Mono`/`Flux` и удивляется, почему «код не выполняется». Потому что на него **никто не подписался** — это ленивость. Помни: без `subscribe()` (или без того, кто подпишется за тебя, например WebFlux) конвейер мёртв.
- **Не блокируй в реактивном коде.** Внутри тестов и реактивных цепочек избегай `.block()`, `Thread.sleep`, JDBC, `synchronized`. Блокировка ломает контракт Reactive Streams и **прячет** проблемы, вместо того чтобы их выявить. Именно поэтому существует `BlockHound` — он подсвечивает такие «засады» сразу.

**📌 Запомнить.** Реактивность = «не жди, иди делай другое». `Mono` — 0..1 элемент, `Flux` — 0..N. Ничего не выполняется до `subscribe()` (ленивость). Backpressure (`request(n)`) защищает от перегрузки и OOM: темп задаёт слабое звено. Тестируем через `StepVerifier`, блокировки ловим `BlockHound`.

## 2. Spring WebFlux, Netty и EventLoop, тестирование

**Простыми словами.** Обычный веб-сервер (Spring MVC) — это как контора, где **на каждого посетителя выделяют отдельного сотрудника** (поток), и тот сидит с ним всё время, даже когда просто ждёт справку из архива. Сотрудники быстро кончаются. WebFlux устроен иначе: горстка очень шустрых сотрудников бегает между тысячами посетителей и **никогда не простаивает** в ожидании — отправив запрос в архив, сотрудник сразу бежит к следующему, а когда справка готова, его «окликнут». Эту горстку шустрых сотрудников зовут **Event Loop**.

**Разбираемся.**

**WebFlux** — это реактивный неблокирующий веб-стек Spring, альтернатива классическому **Spring MVC** (стеку «поток на запрос»). Он работает поверх сервера **Netty** с моделью **Event Loop (цикл событий)**: небольшое число **worker-потоков** (рабочих потоков) обрабатывает тысячи соединений. Здесь **нет** модели **thread-per-request** (один поток на каждый запрос) — в этом вся соль.

Как проходит запрос внутри:
- **Boss-поток** («распорядитель») принимает входящие **TCP-соединения**.
- **Worker-поток** («работник») читает байты через **`epoll`** (механизм ОС Linux, который эффективно сообщает, на каком соединении появились данные — без бесполезного опроса всех подряд).
- Затем worker декодирует HTTP и передаёт запрос в **`DispatcherHandler`** (реактивный диспетчер), а тот — в контроллер.
- Память под данные — это **`ByteBuf`** с **пулами** (переиспользуемые буферы байтов). Благодаря переиспользованию реже создаётся мусор, а значит — **меньше пауз на сборку мусора (GC)**.

🎯 **Критическое правило (запомни накрепко):** Event Loop — это «эстафетная палочка», и держать её надо **минимальное время**. **Блокировать Event Loop нельзя.** Почему так страшно: представь, что один шустрый сотрудник на 100 мс «завис» в ожидании. Если на нём висит 1000 соединений, то это **100 секунд суммарной задержки** для всех — он же один на всех, и пока он стоит, стоят все. Поэтому:
- Блокирующий код (например, **legacy JDBC** — старый блокирующий драйвер базы) **выносят** в отдельный пул потоков через `subscribeOn(Schedulers.boundedElastic())` — это «комната ожидания» из расширяемого числа потоков специально под блокирующие задачи, чтобы они не отравляли Event Loop.
- А вот реактивный ввод-вывод (**`WebClient`** для HTTP, **`R2DBC`** для базы) ничего не блокирует: он регистрирует **колбэк** (функцию «перезвони мне, когда будет готово») и **сразу освобождает поток**.

Ещё пара важных деталей про WebFlux:
- WebFlux **сам подписывается** на возвращённый из контроллера `Mono`/`Flux` (помнишь — без подписки ничего не работает; тут подписку делает за тебя фреймворк) и сериализует результат в ответ.
- Пустоту (когда ничего не нашлось) обрабатывают операторами `defaultIfEmpty` (подставить значение по умолчанию) или `switchIfEmpty` (переключиться на другой поток).
- Тестируют WebFlux через **`WebTestClient`** (специальный тестовый HTTP-клиент) вместе с аннотацией `@WebFluxTest` (поднимает только веб-слой, без всей базы).

**Разбор кода.**
```java
@RestController
@RequestMapping("/api/users")
public class UserController {
    private final UserRepository repo;
    UserController(UserRepository repo) { this.repo = repo; }

    @GetMapping("/{id}")
    public Mono<ResponseEntity<User>> get(@PathVariable Long id) {
        return repo.findById(id)
            .map(ResponseEntity::ok)
            .defaultIfEmpty(ResponseEntity.notFound().build()); // обработка пустоты -> 404
    }

    @GetMapping
    public Flux<User> all() { return repo.findAll(); } // стрим без блокировки
}

@WebFluxTest(UserController.class)
class WebFluxTest {
    @Autowired WebTestClient client;
    @MockitoBean UserRepository repo;
    @Test void ok() {
        when(repo.findById(1L)).thenReturn(Mono.just(new User(1L)));
        client.get().uri("/api/users/1").exchange()
            .expectStatus().isOk()
            .expectBody(User.class);
    }
}
```
**Что здесь происходит:**
- `@RestController` — помечает класс как реактивный REST-контроллер (возвращает данные, а не страницы). `@RequestMapping("/api/users")` задаёт общий префикс адресов.
- Конструктор `UserController(UserRepository repo)` получает репозиторий через **внедрение зависимостей** — Spring сам передаёт нужный объект.
- `@GetMapping("/{id}")` обрабатывает GET-запрос с id в адресе; `@PathVariable Long id` вытаскивает этот id. Метод возвращает `Mono<ResponseEntity<User>>` — то есть **обещание** одного ответа. Цепочка: нашли пользователя (`findById`), обернули в ответ 200 OK (`.map(ResponseEntity::ok)`), а если ничего не нашли — `.defaultIfEmpty(...)` отдаёт 404. Никакой блокировки: всё это лишь описание, выполнит его WebFlux при подписке.
- Второй `@GetMapping` без пути возвращает `Flux<User>` — потенциально много пользователей **стримом**, опять же без блокировки потока.
- В тесте `@WebFluxTest(UserController.class)` поднимает только этот контроллер. `@Autowired WebTestClient client` — тестовый клиент, `@MockitoBean UserRepository repo` — **мок** (подставная заглушка) репозитория.
- `when(repo.findById(1L)).thenReturn(Mono.just(new User(1L)))` — учим заглушку: «на id=1 верни вот такого пользователя». Дальше клиент шлёт GET на `/api/users/1`, и мы проверяем, что статус `isOk()` (200) и тело — объект `User`.

**⚠️ Подводные камни.**
- 🎯 **Никогда не блокируй Event Loop.** Это самое важное правило WebFlux. Worker-потоков мало (примерно по числу ядер), и каждый обслуживает тысячи соединений. Один заблокированный поток замораживает всех «своих» клиентов. Поэтому любой блокирующий вызов (старый JDBC, тяжёлый `synchronized`, `Thread.sleep`) обязательно уноси в `subscribeOn(Schedulers.boundedElastic())`, а лучше — используй реактивные аналоги (`WebClient`, `R2DBC`), которые освобождают поток через колбэк.
- **Не забывай про пустоту.** Реактивный поток может прийти пустым. Если не обработать это через `defaultIfEmpty`/`switchIfEmpty`, клиент получит не тот ответ, что ты ждёшь (например, пустое тело вместо честного 404).

**📌 Запомнить.** WebFlux = неблокирующий веб на Netty + Event Loop: мало потоков на тысячи соединений (нет thread-per-request). Event Loop блокировать нельзя — блокирующее уноси в `boundedElastic()`. Подписку на твой `Mono`/`Flux` делает сам WebFlux. Тесты — `WebTestClient` + `@WebFluxTest`.

## 3. R2DBC: блокировка потоков, проблемы JDBC в реактивном стеке

**Простыми словами.** Представь, что весь твой реактивный конвейер — это команда курьеров, которые **никогда не стоят на месте**. И вдруг один из них упирается в старую дверь, которую можно открыть только постояв и подождав (это и есть блокирующий **JDBC**). Один застрявший курьер тормозит всю эстафету. R2DBC — это «правильная дверь», через которую можно пройти не останавливаясь: оставил запрос — побежал дальше, базе ответит позже.

**Разбираемся.**

**JDBC (Java Database Connectivity)** — классический, **блокирующий** способ работать с базой из Java. Поток отправляет запрос в БД и **стоит, ждёт ответа**, всё это время занимая поток впустую. В обычном MVC это терпимо, а вот в реактивном стеке это катастрофа: такой вызов **блокирует Event Loop** и ломает всю модель (см. тему 2).

Важная мысль для экзамена: **полумеры не делают JDBC реактивным**. Можно вынести его в `boundedElastic`, обернуть в `@Async`, даже запустить на **Virtual Threads (виртуальных потоках — лёгких потоках из новой Java)** — но всё это лишь «дешевле блокировать», а не настоящая реактивность. Причины, почему это **архитектурный mismatch (несовпадение по самой сути)**, а не вопрос настроек:
- транзакции и сессии Hibernate **привязаны к потоку** (thread-bound) — а реактивность как раз не привязана к потоку;
- ввод-вывод остаётся блокирующим;
- на уровне драйвера **нет backpressure** (база не умеет «придерживать» поток);
- слабая поддержка **отмены** запросов.

**R2DBC (Reactive Relational Database Connectivity)** — это неблокирующий драйвер для реляционных (SQL) баз, который корректно дружит с Reactor: поддерживает backpressure и отмену. Чем R2DBC отличается от привычной **JPA/Hibernate** — список различий:
- **нет lazy loading** (ленивой подгрузки связанных данных);
- **нет каскадов** и автоматических связей между сущностями;
- **нет dirty checking** (автоотслеживания изменений, когда Hibernate сам видит, что объект поменялся, и сохраняет);
- **нет генерации схемы** — таблицы придётся создавать самому, обычно через **Flyway** (инструмент миграций БД);
- **нет L2-кеша** (кеша второго уровня);
- используются **собственные аннотации** (`@Table`, `@Column`, `@Id` из пакета `spring-data-relational`), а не JPA-шные;
- **новизна сущности определяется по id**: если `@Id == null` → это новая запись → **INSERT**; если `!= null` → **UPDATE**;
- пул соединений — это **`r2dbc-pool`**, а **не HikariCP** (который для JDBC).

🎯 **Когда выбирать, а когда нет:**
- **Стоит** брать R2DBC для **высококонкурентных I/O-нагрузок** (много одновременных запросов, упирающихся в ожидание) и когда у тебя **сквозной (end-to-end) реактивный стек**.
- **Не стоит** — для сложной ORM-графики (много связей, каскадов), для legacy-миграций со старого кода и для **CPU-bound** задач (где тормозит вычисление, а не ожидание). Там проще и понятнее обычные JDBC/JPA — в том числе с Virtual Threads.

**Разбор кода.**
```java
@Table("orders")
public class Order {
    @Id private Long id;                       // null -> INSERT, иначе UPDATE
    @Column("customer_name") private String customerName;
    @Column("amount") private BigDecimal amount;
}

public interface OrderRepository extends ReactiveCrudRepository<Order, Long> {
    Flux<Order> findByCustomerName(String name);   // возвращает Flux/Mono, не List
}

@Service
public class OrderService {
    private final OrderRepository repo;
    OrderService(OrderRepository repo) { this.repo = repo; }
    public Mono<Order> create(Order o) { return repo.save(o); }     // неблокирующий I/O
    public Flux<Order> byCustomer(String n) { return repo.findByCustomerName(n); }
}
```
**Что здесь происходит:**
- `@Table("orders")` привязывает класс `Order` к таблице `orders` — это **R2DBC-аннотация** из `spring-data-relational`, не путать с JPA `@Entity`.
- `@Id private Long id` — поле-идентификатор. Комментарий напоминает ключевое правило: если id `null`, при сохранении будет INSERT, иначе UPDATE.
- `@Column("customer_name")` и `@Column("amount")` связывают поля класса с колонками таблицы.
- `OrderRepository extends ReactiveCrudRepository<Order, Long>` — **реактивный** репозиторий. Метод `findByCustomerName` Spring создаст сам по имени, и он вернёт **`Flux<Order>`** (поток заказов), а **не `List`** — это принципиально: данные текут, а не отдаются разом.
- В `OrderService`: `create` вызывает `repo.save(o)` и возвращает `Mono<Order>` — неблокирующее сохранение; `byCustomer` возвращает `Flux<Order>` — неблокирующий поиск. Поток нигде не «встаёт и ждёт».

**⚠️ Подводные камни.**
- 🎯 **R2DBC — не «JDBC побыстрее», а другая модель.** Не тащи его в проект только ради моды. Для сложной ORM с кучей связей, каскадов и ленивой подгрузки, для миграции старого кода и для задач, упирающихся в процессор (CPU-bound), он только усложнит жизнь — там уместнее JDBC/JPA. R2DBC оправдан там, где много **одновременных** запросов, упирающихся в **ожидание** ввода-вывода.
- **Блокирующий JDBC в реактивном стеке нельзя «починить» настройками.** `boundedElastic`, `@Async`, Virtual Threads — это обходные пути «дешевле блокировать», а не реактивность. Корень проблемы архитектурный: thread-bound транзакции, блокирующий I/O, отсутствие backpressure и отмены на уровне драйвера.
- **Многое из JPA просто не работает.** Привыкнув к Hibernate, легко ждать lazy loading, каскадов, dirty checking и автосоздания таблиц — всего этого в R2DBC **нет**. Схему готовь сам (Flyway), связи разруливай руками.

**📌 Запомнить.** JDBC блокирует поток и в реактивном стеке убивает Event Loop. R2DBC — неблокирующий драйвер SQL с backpressure и отменой, но без lazy loading, каскадов, dirty checking, генерации схемы и L2-кеша; пул — `r2dbc-pool`, новизна по `@Id == null`. Бери R2DBC для высококонкурентного I/O и сквозной реактивности; для сложной ORM и CPU-bound — JDBC/JPA.

## 4. Реактивные транзакции, полный путь реактивного запроса

**Простыми словами.** **Транзакция** — это «всё или ничего»: либо все операции прошли, либо откатываемся, будто ничего не было (как перевод денег: либо и списали, и зачислили, либо ни то ни другое). В обычном мире Java транзакцию «приклеивают» к потоку. Но в реактивности поток постоянно меняется (помнишь — курьеры не стоят на месте), поэтому транзакцию приклеивают не к потоку, а **к самой цепочке обработки**, которая едет вместе с данными.

**Разбираемся.**

В реактивном стеке транзакция **не привязана к потоку**. В обычном Spring контекст транзакции лежит в **`ThreadLocal`** (хранилище «на каждый поток своё»). Но в реактивности один запрос обслуживается разными потоками, поэтому `ThreadLocal` не годится. Вместо него используется **`Reactor Context`** — специальное хранилище, которое **передаётся вдоль всей цепочки** `Mono`/`Flux`, путешествуя вместе с данными.

Как управлять транзакциями реактивно:
- через **`ReactiveTransactionManager`** (реактивный менеджер транзакций) и декларативную аннотацию **`@Transactional`** на методах, которые возвращают `Mono`/`Flux`;
- либо императивно (вручную) через **`TransactionalOperator`** — объект, которым ты сам оборачиваешь цепочку в транзакцию.

Можно настраивать: **изоляцию** (насколько транзакции «видят» друг друга), **propagation** (как вести себя, если транзакция уже идёт), **timeout** (таймаут) и **readOnly** (только чтение).

**Полный путь реактивного запроса** (от байтов до ответа) — собери всё из прошлых тем воедино:
1. **Netty Boss-поток** принимает TCP-соединение и регистрирует канал в **Worker EventLoop**.
2. **Worker** читает байты через `epoll`, декодирует HTTP (причём тело читается **лениво** — только когда понадобится).
3. **`ReactorHttpHandlerAdapter`** → **`DispatcherHandler`** → контроллер **строит** цепочку `Mono`/`Flux`. Важно: на этом шаге **ничего ещё не выполняется** — это всё та же ленивость.
4. WebFlux **подписывается** на цепочку → R2DBC/WebClient выполняет неблокирующий запрос; когда данные готовы, **колбэк** продолжает цепочку дальше. Всё время ожидания **поток не занят** — он работает на других клиентов.
5. Результат сериализуется (например, через **Jackson** — библиотеку JSON), Netty ставит байты в очередь записи, ОС асинхронно их отправляет, а соединение остаётся открытым (**keep-alive** — переиспользование соединения).

Главный итог: **ни один шаг не блокирует EventLoop**.

**Разбор кода.**
```java
@Service
public class TransferService {
    private final AccountRepository repo;
    private final TransactionalOperator tx;        // поверх ReactiveTransactionManager
    TransferService(AccountRepository repo, TransactionalOperator tx) {
        this.repo = repo; this.tx = tx;
    }

    public Mono<Void> transfer(Long from, Long to, BigDecimal amount) {
        Mono<Void> ops = repo.debit(from, amount)
            .then(repo.credit(to, amount))
            .then();
        return tx.transactional(ops);              // вся цепочка атомарна реактивно
    }
}

// настройка изоляции/propagation
@Bean
TransactionalOperator txOperator(ReactiveTransactionManager tm) {
    var def = new DefaultTransactionDefinition();
    def.setIsolationLevel(TransactionDefinition.ISOLATION_READ_COMMITTED);
    def.setPropagationBehavior(TransactionDefinition.PROPAGATION_REQUIRED);
    def.setTimeout(5);
    return TransactionalOperator.create(tm, def);
}
```
**Что здесь происходит:**
- `@Service` помечает класс как сервис (бизнес-логику). Через конструктор внедряются репозиторий счетов и `TransactionalOperator` — он работает поверх `ReactiveTransactionManager`.
- В `transfer` строится цепочка: `repo.debit(from, amount)` (списать) → `.then(repo.credit(to, amount))` (затем зачислить) → `.then()` (завершить как `Mono<Void>`). Оператор `.then(...)` означает «после успеха предыдущего шага делай следующий».
- `tx.transactional(ops)` оборачивает **всю цепочку в одну реактивную транзакцию** — теперь это атомарно: либо списали И зачислили, либо откат. Это та самая логика «всё или ничего» для перевода денег.
- В `@Bean txOperator` настраивается сам оператор: `setIsolationLevel(... READ_COMMITTED)` — уровень изоляции (видны только зафиксированные чужие изменения); `setPropagationBehavior(... PROPAGATION_REQUIRED)` — «нужна транзакция: есть — присоединиться, нет — создать»; `setTimeout(5)` — таймаут 5 секунд. `TransactionalOperator.create(tm, def)` собирает оператор с этими настройками.

**⚠️ Подводные камни.**
- **`ThreadLocal` в реактивности не работает.** Любой код, который рассчитывает хранить контекст «в потоке» (старые транзакции, MDC для логов, контекст безопасности), в реактивном стеке сломается — поток-то меняется по дороге. Контекст нужно вести через **`Reactor Context`**, который едет вместе с цепочкой.
- **Транзакция охватывает только то, что внутри обёрнутой цепочки.** Если забыть обернуть цепочку в `tx.transactional(...)` (или повесить `@Transactional` не на тот метод), операции пройдут **без атомарности** — и при сбое посреди перевода деньги «зависнут» списанными, но не зачисленными.

**📌 Запомнить.** Реактивная транзакция привязана не к потоку, а к цепочке — контекст живёт в `Reactor Context`, не в `ThreadLocal`. Управление: `@Transactional` на методах с `Mono`/`Flux` либо `TransactionalOperator`. Полный путь запроса WebFlux+Netty ленив и неблокирующ от приёма TCP до записи ответа — EventLoop не блокируется нигде.

## 5. Apache Kafka: topic, partition, offset, consumer group, replication, ISR

**Простыми словами.** Представь огромный **журнал событий** или ленту: каждое новое событие просто дописывается в конец, и его уже не стереть и не поменять. Все желающие могут читать эту ленту с любого места и сколько угодно раз перечитывать. Вот это и есть Kafka — гигантский, надёжный, распределённый «журнал», куда сервисы складывают события, а другие сервисы их вычитывают в своём темпе.

**Разбираемся.**

**Kafka** — это распределённый **лог сообщений** (платформа event streaming, потоковой передачи событий). Разберём по понятиям:
- **Topic (топик)** — логическая категория событий (например, `user-events`). Физически топик разбит на несколько **partition**.
- **Partition (партиция)** — это **упорядоченный неизменяемый лог** (та самая лента, куда только дописывают в конец). Партиции дают **параллелизм** (несколько лент читают параллельно) и **порядок** внутри ленты. Важно: сообщения с одинаковым **key (ключом)** гарантированно попадают в **одну** партицию (партиция выбирается по hash ключа; если ключ `null` — round-robin/sticky, то есть по очереди/«липко»). Это нужно, чтобы события одного объекта (например, одного пользователя) сохраняли строгий порядок.
- **Record (запись)** — единица данных. Содержит: key, value, timestamp, headers и offset.
- **Offset (оффсет)** — 64-битный, **монотонно растущий** номер записи внутри партиции (начинается с 0). Это как номер строки в журнале. Consumer хранит номер последней обработанной записи во внутреннем топике **`__consumer_offsets`**, чтобы после перезапуска **продолжить с того же места**, а не с начала.
- **Consumer group (группа потребителей)** — набор consumer'ов, которые **вместе** читают один топик, поделив работу. Ключевое правило: каждая партиция назначается **ровно одному** consumer внутри группы (**single membership**). Отсюда следствия: если consumer'ов больше, чем партиций — лишние **простаивают** (работы для них нет); если меньше — один consumer читает несколько партиций. Назначением и его пересмотром (**rebalance**) управляет **Group coordinator** (координатор группы); стратегии назначения — **Range, Round-robin, Sticky**. При этом **разные группы читают независимо**: одну и ту же партицию спокойно читают consumer'ы из разных групп (каждая группа ведёт свой offset).
- **Replication / ISR (репликация и синхронные реплики)** — каждая партиция **копируется** на несколько брокеров: 1 **leader** (главная копия) + **followers** (последователи). **ISR (In-Sync Replicas — синхронные реплики)** — это те реплики, которые **успевают** за лидером. Настройка `acks=all` заставляет продюсера ждать подтверждения **от всех ISR** — тогда запись **не потеряется**, даже если лидер упадёт (его заменит одна из синхронных реплик).

🎯 **Семантики доставки** (важно для экзамена) — какие гарантии того, сколько раз сообщение обработается:
- **at-most-once (не более одного раза)** — commit делается **до** обработки. Возможна **потеря** (упали после commit, но до обработки).
- **at-least-once (хотя бы один раз)** — commit **после** обработки. Возможны **дубли** (обработали, но не успели закоммитить — прочитаем снова). Лекарство — **идемпотентность** (повторная обработка не вредит).
- **exactly-once (ровно один раз)** — достигается через **транзакции Kafka**.

🎯 **Отличие Kafka от RabbitMQ:**
- **Kafka** — это **лог** с долгим хранением и возможностью **переигрывать** (перечитывать) сообщения. Модель **pull** (потребитель сам тянет), высокий **throughput** (пропускная способность). Сложной маршрутизации нет — фильтрация ложится на сторону consumer.
- **RabbitMQ** — это классический брокер с **exchange и routing** (умной маршрутизацией). Модель **push** (брокер сам толкает сообщения потребителю).

**Разбор кода.**
```java
@KafkaListener(topics = "users", groupId = "user-processor")
public void consume(ConsumerRecord<String, String> r) {
    System.out.printf("p=%d off=%d key=%s val=%s%n",
        r.partition(), r.offset(), r.key(), r.value());
}
```
**Что здесь происходит:**
- `@KafkaListener(topics = "users", groupId = "user-processor")` — аннотация, которая делает метод **слушателем** топика `users` в составе группы `user-processor`. Spring сам подпишет метод и будет звать его на каждое новое сообщение.
- Параметр `ConsumerRecord<String, String> r` — это одна пришедшая запись (ключ-строка, значение-строка).
- Внутри метод печатает основные поля записи: `r.partition()` (из какой партиции), `r.offset()` (номер записи), `r.key()` (ключ) и `r.value()` (значение). По сути это наглядная демонстрация всех понятий из темы: партиция, offset, ключ, значение.

**⚠️ Подводные камни.**
- 🎯 **Выбери семантику доставки осознанно.** at-most-once может **терять** сообщения (commit до обработки), at-least-once может **дублировать** (commit после) — и тогда твой обработчик обязан быть **идемпотентным**, иначе повторная доставка наделает бед (например, дважды спишет деньги). exactly-once возможен, но только через транзакции Kafka и не бесплатно.
- 🎯 **Не путай Kafka с RabbitMQ.** Kafka — это лог с хранением и переигрыванием, pull, высокий throughput, **без** сложной маршрутизации (фильтруешь на стороне consumer). RabbitMQ — брокер с exchange/routing и моделью push. Перепутать их роли на экзамене — типичная ошибка.
- **Партиций должно быть достаточно.** Параллелизм чтения в группе **ограничен числом партиций**: consumer'ов больше партиций — лишние простаивают зря. А ещё ключ сообщения определяет порядок: один ключ → одна партиция → гарантированный порядок событий этого объекта.

**📌 Запомнить.** Kafka = распределённый неизменяемый лог событий. Topic делится на partition (порядок + параллелизм), запись адресуется offset. Один key → одна партиция → порядок. В группе партиция = ровно один consumer; разные группы читают независимо. Надёжность: leader + ISR + `acks=all`. Семантики: at-most/at-least/exactly-once.

## 6. Kafka консьюмер: архитектура, конфигурации, batch, throughput

**Простыми словами.** **Консьюмер (consumer)** — это «читатель» ленты Kafka. Он в цикле спрашивает: «есть что-нибудь новенькое?» — и забирает пачку записей. Несколько читателей в одной группе делят между собой партиции, как официанты делят столики: каждый столик закреплён ровно за одним. Если кто-то пришёл или ушёл — столики **перераспределяют** (это rebalance).

**Разбираемся.**

Consumer подписывается на топик и в цикле вызывает **`poll()`** — забирает очередную пачку записей. Распределением партиций между consumer'ами заведует **Group coordinator** (координатор группы). Если состав группы меняется (кто-то добавился или отвалился), происходит **rebalance** — партиции **перераспределяются**. Стратегии назначения:
- **Range** — каждому достаются подряд идущие партиции;
- **Round-robin** — партиции раздаются по кругу;
- **Sticky** — «липкая», старается **минимизировать перемещения** партиций при rebalance.

**Управление offset** (где запомнить, докуда дочитали):
- **auto-commit** (`enable.auto.commit=true`) — offset коммитится **периодически** автоматически. Просто, но рискованно: можно потерять или продублировать сообщения.
- **manual** (ручной) — ты сам коммитишь `commitSync`/`commitAsync` или вызываешь `Acknowledgment.acknowledge()` **после** обработки. Надёжнее.

**Batch consumer (пакетный потребитель)** (`listener.type=batch`) — обрабатывает **пачку записей за раз**. Это даёт выше **throughput (пропускную способность)**, но 🎯 **несовместим с non-blocking retry** (неблокирующими повторами).

**Как поднять пропускную способность (throughput):**
- больше партиций → больше consumer'ов в группе (предел параллелизма = число партиций);
- параметр **`concurrency`** в `ConcurrentKafkaListenerContainerFactory` — несколько потоков-консьюмеров внутри приложения;
- batch-обработка;
- тюнинг `max.poll.records` (сколько записей забирать за `poll`), `fetch.min.bytes` (минимальный размер выборки) и `fetch.max.wait.ms` (сколько ждать накопления).

Ошибки обработки → отправляются в **retryable-топик** (топик повторов) → если не вышло → в **DLT (Dead Letter Topic — топик «мёртвых писем»)**, который разбирают методом с `@DltHandler`.

**Разбор кода.**
```java
@KafkaListener(topics = "events", groupId = "g1", containerFactory = "batchFactory")
public void consumeBatch(List<ConsumerRecord<String, String>> records, Acknowledgment ack) {
    records.forEach(r -> process(r.value()));
    ack.acknowledge();   // ручной commit offset ПОСЛЕ обработки всей пачки (at-least-once)
}

@DltHandler   // обработка сообщений из Dead Letter Topic (в том же классе)
public void dlt(ConsumerRecord<String, String> r,
                @Header(KafkaHeaders.EXCEPTION_MESSAGE) String err) {
    log.error("DLT: topic={} err={} payload={}", r.topic(), err, r.value());
}
```
**Что здесь происходит:**
- `@KafkaListener(..., containerFactory = "batchFactory")` — слушатель в **пакетном** режиме: указанная фабрика настроена на batch. Поэтому метод принимает не одну запись, а `List<ConsumerRecord<...>>` — целую пачку.
- `records.forEach(r -> process(r.value()))` — обрабатываем каждую запись пачки. Затем `ack.acknowledge()` — **ручной commit offset, но только ПОСЛЕ обработки всей пачки**. Это даёт семантику **at-least-once**: если упадём до `acknowledge()`, пачку перечитаем заново (возможны дубли — будь идемпотентным).
- `@DltHandler` — помечает метод как обработчик **Dead Letter Topic** (в том же классе). Сюда попадают сообщения, которые не удалось обработать даже с повторами.
- `@Header(KafkaHeaders.EXCEPTION_MESSAGE) String err` — вытаскивает из заголовков **текст ошибки**, из-за которой сообщение «умерло». Дальше мы логируем топик, ошибку и payload, чтобы разобраться.

```yaml
spring:
  kafka:
    consumer:
      group-id: g1
      enable-auto-commit: false
      max-poll-records: 500
    listener:
      type: batch
      ack-mode: manual
      concurrency: 3        # 3 параллельных консьюмера
```
**Что здесь происходит:**
- `group-id: g1` — группа потребителей. `enable-auto-commit: false` — **выключаем** автокоммит (будем коммитить вручную, надёжно). `max-poll-records: 500` — за один `poll()` забираем до 500 записей.
- `listener.type: batch` — пакетный режим (метод получает список). `ack-mode: manual` — ручное подтверждение offset (под наш `ack.acknowledge()`). `concurrency: 3` — три параллельных консьюмера в приложении (как комментарий и говорит) для большей пропускной способности.

**⚠️ Подводные камни.**
- 🎯 **Batch несовместим с non-blocking retry.** Если ты включил пакетную обработку (`listener.type=batch`), не рассчитывай на неблокирующие повторы — они вместе не работают. Это прямое ограничение, которое легко забыть и потом удивляться, почему ретраи не срабатывают.
- **Auto-commit рискует потерями и дублями.** При `enable.auto.commit=true` offset коммитится по таймеру, не глядя на то, успел ты обработать запись или нет. Упал в неудачный момент — либо потеряешь (закоммитил до обработки), либо продублируешь. Для надёжности — ручной commit **после** обработки.
- **Лишние consumer'ы простаивают.** Параллелизм в группе ограничен числом партиций. Накрутишь `concurrency` или поднимешь больше consumer'ов, чем партиций — «лишние» будут стоять без дела. Хочешь больше параллелизма — добавляй партиции.

**📌 Запомнить.** Consumer в цикле делает `poll()` и получает пачку; распределение партиций ведёт coordinator, при смене состава — rebalance (Range/Round-robin/Sticky). Offset: auto-commit (рискованно) или manual после обработки (надёжно, at-least-once). Throughput растёт от партиций, `concurrency`, batch и тюнинга `poll`. Непереваренное → retryable-топик → DLT (`@DltHandler`). Batch ≠ non-blocking retry.

## 7. Kafka продьюсер: архитектура, KafkaTemplate, Routing/Reply, throughput

**Простыми словами.** **Продьюсер (producer)** — это «писатель», который складывает события в ленту Kafka. Чтобы не бегать с каждым письмом по отдельности, он копит письма в пачку и отправляет их разом — так быстрее (как почтальон, который ждёт, пока наберётся полная сумка, и несёт всё за один заход). Отправляет он, не дожидаясь на месте: «бросил и пошёл», а о результате узнаёт позже по колбэку.

**Разбираемся.**

Producer формирует **`ProducerRecord(topic, key, value)`**, сериализует key/value (превращает в байты), а **партиционер** выбирает партицию:
- по умолчанию — по **hash ключа** (одинаковый ключ → одна партиция → порядок);
- если ключ `null` — **sticky/round-robin** («липко»/по кругу);
- можно написать **кастомный `Partitioner`**.

Сообщения **батчатся** (копятся в буфер) по настройкам `batch.size` (размер пачки) и `linger.ms` (сколько подождать ради наполнения пачки) и шлются **пачками** — отсюда и высокий throughput. Метод `send()` **асинхронный**: он возвращает `Future` (обещание результата) или принимает колбэк. Продюсер **потокобезопасен** — его делают **один на всё приложение**, и его обязательно нужно **закрывать**.

**Надёжность доставки:**
- `acks=all` + **idempotent producer (идемпотентный продюсер)** через `enable.idempotence=true` — защита от дублей при повторных отправках (ретраях).
- 🎯 **Требования к idempotent-продюсеру:** `max.in.flight.requests.per.connection ≤ 5`, `retries > 0`, `acks=all`.
- Он **включён по умолчанию**, если параметры этому не противоречат. Если же явно включить идемпотентность с несовместимыми настройками — будет `ConfigException` (ошибка конфигурации).
- Без надёжных настроек семантика — **«unknown once»** (доставок 0..N, как повезёт).

**В Spring есть три инструмента отправки:**
- **`KafkaTemplate`** — основной, обычный способ отправки.
- **`RoutingKafkaTemplate`** — выбирает нужный `ProducerFactory` **по имени топика** (хранит `Map<String, ProducerFactory>`). Нужен, когда на разных топиках разные сериализаторы (например, JSON на одном, Avro на другом).
- **`ReplyKafkaTemplate`** — синхронный **request-reply** (запрос-ответ) поверх Kafka: метод `sendAndReceive` отправляет и **ждёт ответ** в `REPLY_TOPIC`. Применяют редко: вносит задержки и противоречит самой асинхронной природе Kafka.

**Throughput (пропускная способность) продюсера** растёт от:
- батчинга (`linger.ms` / `batch.size`);
- сжатия (`compression.type=lz4`);
- большего числа партиций;
- асинхронной отправки;
- retry с backoff (повторы с нарастающей паузой).

**Разбор кода.**
```java
@Service
@RequiredArgsConstructor
public class UserEventProducer {
    private final KafkaTemplate<String, String> kafkaTemplate;

    public void publish(String userId, String json) {
        kafkaTemplate.send("users", userId, json)   // key=userId -> одна партиция, порядок
            .whenComplete((res, ex) -> {
                if (ex == null) System.out.println("offset=" + res.getRecordMetadata().offset());
                else log.error("send failed", ex);
            });
    }
}
```
**Что здесь происходит:**
- `@Service` помечает класс сервисом, `@RequiredArgsConstructor` (из Lombok) автоматически создаёт конструктор для `final`-полей — сюда Spring внедрит `KafkaTemplate`.
- `kafkaTemplate.send("users", userId, json)` — отправка в топик `users` с ключом `userId`. Комментарий подчёркивает суть: **key=userId → одна партиция → порядок** событий этого пользователя.
- `.whenComplete((res, ex) -> {...})` — это **колбэк** на результат асинхронной отправки (помнишь — «бросил и пошёл»). Если ошибки нет (`ex == null`) — печатаем полученный offset; если есть — логируем сбой. Сам `send` не блокирует поток, мы лишь подписываемся на исход.

```yaml
spring:
  kafka:
    producer:
      acks: all
      properties:
        enable.idempotence: true
        max.in.flight.requests.per.connection: 5
        linger.ms: 10
        compression.type: lz4
```
**Что здесь происходит:**
- `acks: all` — ждать подтверждения от **всех ISR** (надёжность, запись не потеряется).
- `enable.idempotence: true` — включаем **идемпотентного** продюсера (защита от дублей при ретраях).
- `max.in.flight.requests.per.connection: 5` — не больше 5 «в полёте» (требование идемпотентности `≤ 5`).
- `linger.ms: 10` — подождать до 10 мс, чтобы набрать пачку побольше (батчинг → throughput).
- `compression.type: lz4` — сжатие данных алгоритмом lz4 (меньше байт по сети → выше throughput).

**⚠️ Подводные камни.**
- 🎯 **У идемпотентного продюсера жёсткие требования.** Чтобы `enable.idempotence=true` работал, нужны `max.in.flight.requests.per.connection ≤ 5`, `retries > 0` и `acks=all`. Поставишь несовместимые настройки и явно включишь идемпотентность — получишь `ConfigException` на старте. А если вообще не настроить надёжность — семантика будет «unknown once» (доставок от 0 до N), то есть никаких гарантий.
- **Продюсер — один на приложение, и его надо закрывать.** Он потокобезопасен, поэтому плодить много экземпляров не нужно (это дорого). Но забудешь закрыть — рискуешь потерять данные из буфера, которые не успели отправиться.
- **ReplyKafkaTemplate — почти всегда не то, что тебе нужно.** Синхронный request-reply поверх Kafka вносит задержки и идёт вразрез с асинхронной идеологией. Применяй его только осознанно и редко.

**📌 Запомнить.** Producer батчит сообщения (`batch.size`/`linger.ms`) и шлёт асинхронно (`send()` → `Future`/колбэк); партицию выбирает партиционер (по hash key, null → sticky/round-robin). Надёжность: `acks=all` + идемпотентность (`≤5` in-flight, `retries>0`, `acks=all`). Отправка в Spring: `KafkaTemplate` (обычно), `RoutingKafkaTemplate` (разные топики/сериализаторы), `ReplyKafkaTemplate` (синхронный request-reply, редко). Throughput: батчинг, сжатие, партиции, async, retry.

## 8. 4 золотых сигнала мониторинга, типы метрик Prometheus, квантили/персентили

**Простыми словами.** Метрики — это **приборная панель машины**: спидометр, тахометр, датчик топлива, температура двигателя. Тебе не нужно разбирать мотор, чтобы понять, что что-то не так — достаточно взгляда на приборы. **4 золотых сигнала** — это «обязательный минимум приборов» для любого сервиса, без которого ехать вслепую.

**Разбираемся.**

**Observability (наблюдаемость)** — это способность отвечать на вопросы о **внутреннем** состоянии системы, глядя лишь на её **выходные** данные. Стоит на трёх столпах:
- **метрики** — агрегированные числа, дёшево (спидометр);
- **логи** — детальные записи событий, дорого (бортовой журнал);
- **трассировки (traces)** — путь одного запроса между сервисами (маршрут поездки от и до).

Метрики дают самую быструю обратную связь о здоровье системы.

**4 золотых сигнала (Google SRE)** — минимум для мониторинга любого сервиса:
1. **Latency (задержка)** — время ответа. Смотреть **отдельно** для успешных и неуспешных запросов; ориентироваться на **перцентили** p50/p95/p99 (о них ниже).
2. **Traffic (трафик)** — нагрузка. Например, **RPS** (запросов в секунду) = `rate(_count[1m])`.
3. **Errors (ошибки)** — доля или количество неуспешных (коды 5xx, бизнес-исключения). **error rate** = ошибки / трафик.
4. **Saturation (насыщение)** — насколько забиты ресурсы: CPU, память, GC, длина очереди.

**Типы метрик Prometheus:**

| Тип | Что это | Как смотреть (PromQL) |
|---|---|---|
| **Counter** | только растёт (запросы, ошибки) | `rate(...)` — интенсивность; абсолютное значение не смотрят |
| **Gauge** | растёт и падает (активные соединения, память, размер очереди) | мгновенное значение, без `rate()` |
| **Histogram** | наблюдения раскладываются по бакетам (`le="0.1"`...) + `_sum`/`_count`; квантиль считается **на стороне Prometheus** | `histogram_quantile(0.95, ...)` — **агрегируется** между инстансами |
| **Summary** | квантили считаются **в самом приложении** | приблизительны, **не агрегируются** между инстансами |

Поясним каждый по-простому:
- **Counter (счётчик)** — как **одометр пробега**: только вперёд, никогда назад. Смотрят не само число, а его **скорость роста** через `rate(...)`.
- **Gauge (измеритель)** — как **стрелка спидометра** или указатель топлива: туда-сюда. Смотрят **мгновенное значение**, `rate()` тут не нужен.
- **Histogram (гистограмма)** — раскладывает наблюдения по **бакетам** (корзинам) с границей `le` (less-or-equal — «меньше либо равно»), плюс ведёт `_sum` и `_count`. Главное: **квантиль вычисляется уже на стороне Prometheus** из бакетов, поэтому его можно честно **агрегировать** между несколькими инстансами сервиса.
- **Summary (сводка)** — квантили считаются прямо **в приложении** заранее. Они приблизительны и их **нельзя складывать** между инстансами (нельзя «усреднить персентили»).

**Квантиль / персентиль** — это значение, ниже которого попадает заданная **доля** наблюдений. Например, **p95 = 95% запросов быстрее этого значения** (и только 5% медленнее). Это гораздо информативнее среднего, потому что **среднее скрывает «хвост»** медленных запросов: пара очень долгих ответов в среднем «растворится», а в p99 — будет видна. Для latency предпочитают **Histogram** — он даёт максимальную гибкость (квантили считаются и агрегируются на стороне Prometheus).

**Разбор кода.**
```java
@Service
public class UserMetrics {
    private final Counter logins;
    private final Timer requestTimer;          // под капотом Histogram
    private final AtomicInteger active = new AtomicInteger();

    public UserMetrics(MeterRegistry registry) {
        logins = Counter.builder("user.login.attempts").tag("result", "success").register(registry);
        requestTimer = Timer.builder("user.request").publishPercentileHistogram().register(registry);
        Gauge.builder("user.sessions.active", active, AtomicInteger::get).register(registry);
    }
    public void onLogin() { logins.increment(); }
    public void login()  { active.incrementAndGet(); }
}
// PromQL: rate(user_login_attempts_total[5m]); histogram_quantile(0.95, sum(rate(user_request_seconds_bucket[5m])) by (le))
```
**Что здесь происходит:**
- `Counter logins` — счётчик попыток логина; растёт только вверх. `Timer requestTimer` — таймер длительности запроса (комментарий уточняет: **под капотом это Histogram**). `AtomicInteger active` — потокобезопасный счётчик активных сессий (его покажет Gauge).
- В конструкторе получаем `MeterRegistry registry` — реестр метрик. `Counter.builder("user.login.attempts").tag("result", "success")` создаёт счётчик с **лейблом** `result=success`; `register(registry)` регистрирует его.
- `Timer.builder("user.request").publishPercentileHistogram()` создаёт таймер и просит публиковать **гистограмму перцентилей** — чтобы потом считать p95/p99 на стороне Prometheus.
- `Gauge.builder("user.sessions.active", active, AtomicInteger::get)` — измеритель, который при опросе читает текущее значение `active` (растёт и падает).
- `logins.increment()` увеличивает счётчик логинов, `active.incrementAndGet()` — число активных сессий.
- Комментарий с PromQL показывает запросы: `rate(user_login_attempts_total[5m])` — интенсивность логинов; `histogram_quantile(0.95, sum(rate(user_request_seconds_bucket[5m])) by (le))` — p95 длительности запроса из бакетов гистограммы.

**⚠️ Подводные камни.**
- 🎯 **Кардинальность лейблов — главный риск мониторинга.** **Кардинальность** — это число различных значений лейбла. Лейбл вроде `user_id` (или конкретный URI с id внутри) порождает **отдельный временной ряд на каждое значение** — это взрывает их число и **перегружает Prometheus**. Лекарство: используй **шаблоны** (`uri="/users/{id}"`), а не конкретные значения; бери лейблы **низкой** кардинальности (`method`, `status`, `region`), у которых вариантов мало.
- **Среднее обманывает — смотри перцентили.** Если глядеть только на среднее время ответа, «хвост» из редких, но очень медленных запросов будет незаметен. Именно p95/p99 показывают реальную боль пользователей.
- **Не путай типы метрик.** К Counter применяют `rate(...)` (абсолют бессмыслен), к Gauge — нет (смотрят мгновенное значение). Квантили Summary **нельзя** агрегировать между инстансами, а Histogram — можно. Для latency бери Histogram.

**📌 Запомнить.** 4 золотых сигнала: Latency, Traffic, Errors, Saturation. Типы метрик: Counter (только растёт, `rate`), Gauge (туда-сюда, мгновенно), Histogram (бакеты, квантиль на стороне Prometheus, агрегируется), Summary (квантили в приложении, не агрегируются). p95 = 95% запросов быстрее; среднее скрывает хвост. Для latency — Histogram. Главный враг — высокая кардинальность лейблов.

## 9. Spring Actuator, Micrometer, Prometheus/Grafana, оверхед

**Простыми словами.** Если метрики — это приборы, то нужна и сама **приборная панель** с проводкой. **Actuator** — это набор готовых «датчиков» в приложении (здоровье, метрики, список бинов). **Micrometer** — единый разъём, к которому эти датчики подключаются, не завязываясь на конкретную систему. **Prometheus** — диспетчерская, которая сама регулярно опрашивает датчики и хранит историю. **Grafana** — красивые экраны с графиками поверх этой истории.

**Разбираемся.**

**Spring Boot Actuator** — это набор **production-ready** (готовых к боевой эксплуатации) эндпоинтов для мониторинга и управления:
- `/actuator/health` — статус «жив/болен»;
- `/metrics` — метрики;
- `/prometheus` — метрики в формате, понятном Prometheus;
- `/beans` — граф бинов с их зависимостями и scope;
- `/env`, `/mappings`, `/info` — окружение, маршруты, инфо о приложении.

Включают их через настройку `management.endpoints.web.exposure.include`. По умолчанию **наружу торчит мало** — из соображений безопасности (не светить лишнего).

**Micrometer** — это **фасад** для метрик: единый API поверх разных систем (как **SLF4J** для логов — один интерфейс, разные реализации). Один набор понятий — `MeterRegistry`, `Counter`, `Gauge`, `Timer`, `DistributionSummary` — а **бэкендов много**: Prometheus, Graphite, Datadog и т. д. Spring **автоматически инструментирует** (сам обвешивает метриками): HTTP-запросы, JVM (память, GC, потоки), пулы соединений (Hikari), Kafka, Resilience4j.

**Prometheus** работает по **pull-модели** (модели опроса): он **сам** периодически опрашивает `/actuator/prometheus` — не нужны агенты внутри приложения. Что важно про Prometheus:
- хранит данные во временной БД (time-series);
- язык запросов — **PromQL**;
- **Alertmanager** — для алертов (оповещений);
- **экспортёры** (например, `node_exporter` для метрик операционной системы).

**Grafana** — это **дашборды** (панели с графиками) поверх Prometheus. Типовые запросы:
- RPS = `rate(_count[1m])`;
- среднее = `rate(_sum)/rate(_count)`;
- p95 = `histogram_quantile(0.95, ...)`;
- **heatmap** (тепловая карта) распределения;
- error rate = `rate(_count{status=~"5.."}[1m])`.

**Разбор кода.**
```yaml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
  endpoint:
    health:
      show-details: always
  metrics:
    tags:
      application: user-service   # общий лейбл на все метрики
```
**Что здесь происходит:**
- `management.endpoints.web.exposure.include: health, info, metrics, prometheus` — **явно открываем наружу** только эти эндпоинты (остальные по умолчанию скрыты ради безопасности).
- `management.endpoint.health.show-details: always` — показывать **подробности** в `/health` (по компонентам), а не только общий статус.
- `management.metrics.tags.application: user-service` — добавить **общий лейбл** `application=user-service` ко всем метрикам, чтобы отличать сервисы в Prometheus/Grafana.

```xml
<dependency>
  <groupId>io.micrometer</groupId>
  <artifactId>micrometer-registry-prometheus</artifactId>
</dependency>
```
**Что здесь происходит:**
- Это зависимость **`micrometer-registry-prometheus`** — бэкенд Micrometer под Prometheus. Подключив её, ты получаешь эндпоинт `/actuator/prometheus` в нужном формате, который Prometheus сможет опрашивать.

**⚠️ Подводные камни.**
- 🎯 **Оверхед метрик — это почти всегда кардинальность, а не сами счётчики.** Сами метрики дёшевы (это атомарные счётчики, копейки по производительности). Но **высокая кардинальность лейблов** бьёт по двум фронтам сразу: перегружает приложение (память на хранение множества рядов) **и** перегружает Prometheus. Поэтому следи за лейблами (см. тему 8). На дашборде **золотые сигналы выводят первыми**; алерты вешают на **p95 latency, error rate и насыщение**.
- **По умолчанию эндпоинты Actuator скрыты намеренно.** Это безопасность: `/env`, `/beans`, `/mappings` могут раскрыть внутренности. Открывай через `exposure.include` только то, что реально нужно, и защищай чувствительные эндпоинты.

**📌 Запомнить.** Actuator даёт готовые эндпоинты (`/health`, `/metrics`, `/prometheus`, `/beans`...), по умолчанию закрытые. Micrometer = фасад метрик (как SLF4J для логов): один API, много бэкендов; Spring инструментирует HTTP/JVM/пулы/Kafka сам. Prometheus — pull-модель + PromQL + Alertmanager; Grafana — дашборды (RPS, avg, p95, heatmap, error rate). Оверхед метрик — это кардинальность лейблов, а не счётчики.
